## 3. 환경 분리 원칙
### 3.1 개발 환경
- 기본 작업은 로컬 작업공간과 로컬 서버에서 진행합니다.
- 로컬 개발 검증은 운영 URL 대신 `localhost` 또는 `127.0.0.1` 기반으로 수행합니다.
- 기능 개발, UI 수정, 회귀 점검은 먼저 여기서 끝냅니다.

### 3.2 스테이징(검증) 환경
- 원격 검증이 필요하면 운영 서버와 분리된 스테이징 환경에서 먼저 확인합니다.
- 스테이징은 `main`이 아닌 별도 브랜치/별도 배포 경로/별도 서버를 사용합니다.
- 스테이징이 아직 준비되지 않았으면, 운영 서버를 테스트 서버처럼 사용하지 말고 먼저 분리 환경을 만들거나 사용자에게 그 필요를 알립니다.

### 3.3 운영 환경
- 운영 환경은 최종 승인된 코드만 반영합니다.
- 운영 반영은 “스테이징 검증 완료 -> 사용자 승인 -> 운영 반영” 순서를 반드시 지킵니다.

## 4. 브랜치 및 반영 규칙
- `main`은 운영 반영 전용 브랜치로 취급합니다.
- 개발 작업은 기본적으로 로컬 작업 또는 별도 작업 브랜치에서 진행합니다.
- 스테이징 검증 전에는 `main`에 직접 push하지 않습니다.
- 사용자가 `승인`, `운영 반영`, `main push`를 명시하기 전에는 운영 반영을 금지합니다.

## 5. Firebase 및 설정값 안전 규칙
- 이 프로젝트는 정적 파일만 바뀌는 것이 아니라 Firebase RTDB 기본 설정도 함께 바뀔 수 있습니다.
- 스테이징 관리자 페이지는 기본적으로 운영 Firebase를 직접 쓰면 안 됩니다.
- 스테이징에서 설정 저장 테스트가 필요하면 아래 중 하나를 먼저 만족해야 합니다.
  - 별도 Firebase 프로젝트/DB 사용
  - 운영 DB를 건드리지 않는 읽기 전용 검증 방식 사용
  - 사용자가 운영 DB 대상 검증을 명시적으로 승인
- 운영 기본값 변경이 포함되면, 반영 전후에 실제 RTDB 값을 직접 조회해 확인합니다.

## 6. 기본값/설정 변경 시 필수 5중 체크
`75분 -> 79분` 문제처럼 기본값은 여러 층에서 어긋날 수 있으므로, 아래 5개를 항상 같이 점검합니다.

1. 정적 HTML 기본값
- `index.html`, `admin.html`의 기본 표시값/입력값/버전 표기 확인

2. JS 런타임 기본값
- `main.js` 내부 상수, 초기 상태, 기본 fallback 값 확인

3. 관리자 저장/로드 로직
- 관리자 페이지의 normalize/sanitize/save/load 경로에 강제 보정이 없는지 확인

4. 원격 Firebase 실제값

5. 브라우저 캐시/저장소
- `localStorage`, `sessionStorage`, 오래된 기본값 migration, JS/CSS 캐시 버스팅 필요 여부 확인

## 7. 기본값/설정 관련 회귀 방지 규칙
- 사용자가 직접 저장한 값과, 예전 기본값 찌꺼기를 구분할 수 있어야 합니다.
- 로컬 저장값이 기본값을 덮는 구조라면 “사용자 저장인지, 과거 기본값 잔재인지”를 분리하는 표식을 둡니다.
- 기본값이 바뀌면, 아래 회귀 케이스를 반드시 재현 테스트합니다.
  - 새 브라우저
  - 예전 `localStorage`가 남아 있는 브라우저
  - 관리자 저장 직후 새로고침
  - 운영 URL과 개발자 우회 URL 모두
