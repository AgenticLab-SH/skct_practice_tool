## 2. ★ 최우선 목표: 자동 구독 승인 라이브 성공

> 사용자 요구: "자동 구독 승인을 반드시 성공해야 한다."

### 현재까지 된 것 (검증 완료)
- `scripts/donation_auto_issuer/` 에 **검증 + 자동발급 파이프라인 완성**. 관리자 페이지와 **바이트 단위로 동일한** 라이선스 레코드를 생성(같은 `subscription-crypto.js` 재사용).
- 테스트 15개 통과(`cd scripts/donation_auto_issuer && npm test`): 상호운용(브라우저 복호화+ECDSA검증+위조거부), 종단(실암호화 신청→후원→발급), 웹훅 스모크(HMAC/멱등).
- 입력은 정규화 후원 이벤트 `{ id, donorName, amount, message }`. `message`에 신청ID(`REQ-XXXX`) 포함 시 매칭, 금액 ≥ 요금제 가격이면 발급.
- 수신 경로 3가지: `--once`(단건), `--file`(배치), `--webhook`(HMAC-SHA256 로컬 수신기).

### ★ 아직 안 된 것 (다음 에이전트가 반드시 완성)
**투네이션 실시간 수신 연결.** 투네이션은 공식 웹훅 API가 **없고**, 알림창(AlertBox) 위젯 키 기반 **Socket.IO** 연결로만 후원을 받습니다.

- 위젯 URL: `https://toon.at/widget/alertbox/<KEY>` — 이 `<KEY>`가 핵심. **사용자에게 이 KEY를 받아야 함.**
- 커뮤니티 레퍼런스(검증된 방식):
  - `https://github.com/outstanding1301/donation-alert-api` (Java, socket.io + jsoup로 위젯페이지 파싱 후 연결. 후원 객체 = `{id, nickname, comment, amount}`)
  - `https://github.com/nomomo/Twip-Toonation-Afreehp-Parser-Example` (Node 파서 예제)
- 매핑: `nickname→donorName`, `comment→message`, `amount→amount(원)`, `id→id`.
- 현재 `scripts/donation_auto_issuer/toonation-bridge.example.js` 는 **raw WebSocket 골격(템플릿)**. → **socket.io-client 기반으로 재작성** 필요. 정확한 핸드셰이크(위젯 페이지 fetch → 연결 파라미터 추출 → socket 이벤트 구독)는 위 레퍼런스로 확인.

**할 일 체크리스트:**
1. 사용자에게서 투네이션 알림창 KEY 확보.
2. `socket.io-client` 로 `toonation-bridge.js` 구현(현재 `.example.js` 대체). 실제 메시지 1건으로 포맷 확인 후 `parseToonationMessage()` 확정.
3. 브리지 → 로컬 `run.js --webhook` 로 HMAC 서명 POST(이미 동작).
4. **라이브 종단 테스트:** 테스트 신청(REQ 발급) → 소액 실제 후원(메시지에 REQ 포함) → 자동 발급 확인 → 해당 loginId/password로 고급 로그인 성공 확인.
5. 상시 구동 환경 마련(→ 4번 AWS 항목). 로컬 PC가 꺼지면 자동발급도 멈추므로, 24/7 구동이 "반드시 성공"의 핵심.
6. 운영자 설정 파일 작성: `config.example.json` → `private/donation-auto-issuer.config.json` (Firebase apiKey/databaseURL, 운영자 email/pw, 개인키 PEM 경로 2개, webhookSecret).

**개인키 위치(운영자 로컬):** `private/keys/skct-manual-subscription-private-key.pem`(RSA, 신청 복호화), `private/keys/skct-manual-license-signing-private-key.pem`(ECDSA, 라이선스 서명). 없으면 `admin.html`에서 생성/내보내기. 절대 서버/깃에 올리지 말 것.

## 3. 못다한 기능 / 개선 백로그

- **고급 신청→후원→발급 UX 매끄럽게:** 신청 완료 화면에서 REQ-ID를 후원 메시지에 자동 복사/안내 강화(현재 신청ID 매칭이 자동발급의 전제).
- **자동발급 실패/예외 알림:** `no-request`/`insufficient` 건을 운영자에게 알림(이메일/텔레그램). 현재는 로그/state 파일만 남김.
- **만료 자동 처리:** 만료된 `advancedAccountLicenses` 정리/비활성 배치(현재 만료일은 기록되나 자동 정리 로직 점검 필요).
- **`subscription-crypto.js`** 는 이번에 UMD(브라우저+Node 공용)로 변경됨 — 프론트 동작 회귀 점검(고급 로그인/신청 흐름) 한 번 더 권장.
- **public-clean 브랜치/배포 경로 정리:** 과거 `public-clean` worktree가 깨진 옛 경로를 가리켜 prune함. 현재 배포는 `main`→Actions 단일 경로. `artifacts/releases/public-clean/` 미러가 repo에 남아있어 Pages 아티팩트를 키움 — 정리 검토.
- **문서 최신화:** `docs/` 의 절대경로는 새 위치로 일괄 수정 완료. 신규 구조 반영해 `docs/README.md` 인덱스 점검.

## 4. 서버 부하/오류 정리 + AWS 무료 서버 준비

> 사용자 요구: "서버 부하 주는 불필요한 것/오류를 잡고, AWS 무료 서버 사용 준비."

### 점검 대상 (부하/비용/오류)
- **RTDB 읽기/쓰기 빈도:** `active_visitors`, `total_visits`, `daily_visits`, `stats/live_peak_daily` 등 실시간 카운터가 방문자마다 쓰기 발생 → Firebase 무료 한도(동시접속/다운로드) 소진 위험. 폴링 주기/쓰기 디바운스/세션 정리 점검.
- **Functions 호출량 & 콜드스타트:** `skctSecureApi` 레이트리밋은 in-memory(인스턴스별)라 다중 인스턴스에서 부정확. 남용 라우트 점검.
- **배포 아티팩트 크기:** Pages 배포가 `path: .` 로 repo 전체 업로드 → `functions/node_modules`(~59MB), `artifacts/`, 이미지 다수 포함. 배포에서 불필요 항목 제외(예: 별도 publish 디렉터리 또는 `.gitattributes`/워크플로 필터) 검토 → 배포 속도/용량 개선.
- **콘솔 오류:** 라이브에서 브라우저 콘솔/네트워크 오류(404 자산, CORS, mixed content) 수집·수정.

### AWS 무료 서버 준비 (자동발급 24/7 구동용)
- **용도:** 투네이션 socket.io 리스너 + `run.js --webhook` 자동발급 서비스를 **상시 구동**. (로컬 PC 대신.)
- **후보:**
  - **EC2 t4g.small/t3.micro 프리티어**(12개월) 또는 **Lightsail 최소 인스턴스**(저가 고정요금). 상시 1개 프로세스라 사양 낮아도 충분.
  - Node 20+ 설치, `pm2` 또는 systemd로 데몬화, 자동 재시작/재연결.
- **보안:** 개인키(RSA/ECDSA), Firebase 운영자 계정, webhookSecret을 서버의 보호된 경로/환경변수(또는 AWS SSM Parameter Store/Secrets Manager)로. **깃/이미지에 넣지 말 것.**
- **주의(보안 모델 변화):** 자동발급을 클라우드로 옮기면 개인키가 운영자 PC 밖(서버)에 존재하게 됨. 현재 설계 철학("개인키는 로컬")과 상충하므로, 사용자에게 트레이드오프를 명확히 알리고 승인받을 것. 대안: 서버는 socket 수신/중계만 하고 발급(서명)은 여전히 로컬에서 — 단 그러면 로컬 PC가 켜져 있어야 함. **"반드시 24/7 자동" 요구 = 서버에 키 필요**. 이 결정은 사용자 확인 필수.
- 비용 점검: 프리티어 만료/트래픽 초과 시 과금. 모니터링/알람 설정.

## 5. 도메인 + 광고 수익

> 사용자 요구: "무료/싼 도메인 구매 + 광고 수익."

- **커스텀 도메인 연결:** GitHub Pages는 커스텀 도메인 지원. 구매한 도메인의 DNS(A/CNAME) → Pages, repo에 `CNAME` 파일 + HTTPS 강제. 현재 `agenticlab-sh.github.io/skct_practice_tool/` 의 절대 URL/`canonical`/`sitemap.xml`/`robots.txt`를 새 도메인으로 일괄 갱신 필요(다수 파일에 하드코딩됨).
- **도메인 선택:** 저가 TLD(.com 추천, 신뢰도; .io/.app 등은 비쌈). 무료(.eu.org 등)는 신뢰도/광고 승인 측면에서 비권장. SEO/신뢰 위해 짧고 명확한 .com 권장.
- **광고(애드센스):**
  - 현재 정책: pricing 페이지에 "풀이 중 광고 없음" 명시. 집중 화면(OMR/타이머/풀이)은 광고 배제 유지 — 사용자 경험/정책 일관성.
  - 광고는 가이드/FAQ/대기/결과 화면 등 비집중 영역에만. AdSense 승인 요건(자체 도메인, 충분한 콘텐츠, 개인정보처리방침 — `privacy/` 이미 있음) 점검.
  - 애드센스 정책상 시험/인적성 콘텐츠 적합성, 트래픽 확인.
- **순서 제안:** 도메인 구매 → Pages 연결 + URL 일괄 갱신 → 콘텐츠/정책 페이지 보강 → AdSense 신청 → 비집중 영역에 광고 슬롯.

## 7. 주의: git 히스토리 분기 이력
- 과거 보안 목적(Firebase 키 제거)으로 히스토리가 재작성된 적이 있어, 한때 로컬 main과 origin/main이 **같은 내용·다른 SHA로 분기**했었음. 2026-06-28에 origin/main 위에 변경분만 cherry-pick하여 정리·동기화 완료(현재 `main == origin/main`).
- 앞으로 `git push` 전 항상 `git fetch` 후 `git status`로 분기 여부 확인. 함부로 force push 금지.
