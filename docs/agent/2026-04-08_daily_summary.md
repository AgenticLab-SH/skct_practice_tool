1. 오늘은 고급 안내와 고급모드 진입 흐름을 전반적으로 손봐서 로그인, 기능 설명, 이용권 신청, 신청 조회가 한 흐름으로 보이도록 정리했다.
2. 고급 안내의 오래된 `신청번호 + 조회 비밀번호` 표현을 없애고, 신청할 때 직접 정한 `ID + 비밀번호`를 신청 저장, 상태 조회, 승인 후 로그인까지 같은 값으로 쓰도록 문구와 로직을 맞췄다.
3. 수동 구독 신청이 막히던 원인은 운영 Firebase의 `manualSubscriptionConfig.adminPublicKeyPem`이 비어 있었기 때문이었고, 실제 공개키를 넣어 사용자 신청 저장이 다시 동작하게 복구했다.
4. 운영자용 신청 암호화 체계도 보강해서 개발자 페이지에서 공개키가 없으면 저장 시 자동으로 새 키를 만들고, 개인키는 별도 다운로드 버튼으로 안전하게 백업할 수 있게 했다.
5. 신청 조회는 RTDB 쿼리 인덱스 상태에 따라 실패할 수 있어서, 쿼리가 막히면 전체 신청 목록을 읽어 클라이언트에서 ID 기준으로 다시 찾는 fallback을 추가했다.
6. 실제 운영 DB에는 임시 검증 신청을 저장한 뒤 다시 읽고, 사용자 비밀번호와 관리자 개인키로 각각 복호화가 되는지 확인하고, 마지막에 테스트 데이터를 삭제하는 end-to-end 검증까지 마쳤다.
7. `?advanced=1` 직접 링크만으로 고급모드가 열려 보일 수 있는 여지도 찾아서, 인증 상태가 없으면 주소에서 `advanced` 파라미터를 바로 제거하고 일반모드로 정리되게 차단했다.
8. 고급모드에서는 이미 잠금이 풀린 상태가 더 자연스럽게 보이도록 사이드바의 고급 버튼에서 자물쇠를 없애고 `고급 모드` 라벨 중심으로 보이게 조정했다.
9. 작업 중에는 Windows의 `.codex` 환경 잔재 때문에 `python`과 `pip`가 앱 선택 창을 띄우는 문제도 정리해서, 깨진 링크와 bare 스크립트를 비활성화하고 실제 Python과 Codex 래퍼가 정상 경로를 타도록 복구했다.
