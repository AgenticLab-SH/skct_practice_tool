# SKCT Tool 학습 메모
작성일시: 2026-04-10 10:38:13 +09:00

이 문서는 같은 실수를 반복하지 않기 위한 짧은 재발 방지 메모입니다.

## 1. 공개 화면의 숨은 운영 통로는 기능이 아니라 위험이다

- 도움말 제목 더블클릭처럼 일반 사용자가 예상하지 못하는 숨은 관리자 진입은 운영 편의처럼 보여도, 결국 공개 화면에 백도어를 남기는 셈입니다.
- 관리자 화면은 명시적인 별도 경로와 인증으로만 접근하게 두는 편이 안전합니다.

## 2. 문서 링크가 끊기면 다음 작업자가 잘못된 기준으로 판단한다

- 이번에는 `docs/README.md`가 없는 TODO 인덱스를 가리키고, 기능 카탈로그는 없는 `30_MASTER_DOC.md`를 가리키는 상태였습니다.
- 이런 종류의 drift는 코드 버그보다 눈에 덜 띄지만, 다음 작업자가 오래된 기준을 따라가게 만들어 작업 품질을 떨어뜨립니다.
- 새 기준 문서를 만들거나 폴더 구조를 바꾸면, 링크를 거는 문서도 같은 턴에 같이 맞춰야 합니다.

## 3. 책임 경계가 애매한 부가 기능은 메인 앱 안에 직접 넣지 않는다

- 확장 ZIP처럼 핵심 연습 도구와 성격이 다른 부가 자료는 메인 화면 안에서 바로 설치/다운로드하게 두면, 사용자가 그 기능까지 메인 서비스가 직접 책임지는 것으로 받아들이기 쉽습니다.
- 이번에는 메인 앱에서 직접 ZIP을 노출하던 구조를 없애고, `extension-info.html` 같은 별도 안내 페이지로 분리했습니다.
- 이런 종류의 기능은 `핵심 기능`이 아니라 `별도 테스트 자료`라는 메시지가 화면 구조만으로도 드러나게 만들어야 합니다.

## 4. 운영 문구의 실제 기준은 HTML이 아니라 siteTextConfig일 수 있다

- 이번에는 `index.html`의 숨김 제목과 설명을 바꿔도, 로딩 후에는 다시 예전 문구로 돌아가는 현상이 있었습니다.
- 원인은 Firebase `config/siteTextConfig`와 `site-text-config.js` fallback이 HTML을 다시 덮어쓰는 구조였기 때문입니다.
- 메타/가이드/더보기 같은 문구를 바꿀 때는 `index.html`만 수정하지 말고, `site-text-config.js` 기본값과 legacy migration까지 같이 봐야 같은 문제가 반복되지 않습니다.

## 5. 정적 앱을 서버 분리로 옮길 때는 프런트 fallback부터 정리해야 한다

- GitHub Pages 정적 앱에서 갑자기 direct RTDB를 전부 끊어버리면, 서버가 준비되기 전까지 신청/조회/고급 로그인 흐름이 한꺼번에 죽습니다.
- 이번에는 `manualSubscriptionConfig.secureApiBaseUrl`이 있을 때만 서버 경유를 우선 쓰고, 없으면 기존 direct 경로를 유지하는 식으로 전환 발판을 먼저 만들었습니다.
- 이런 식으로 `새 경로 우선 + 구 경로 fallback`을 먼저 넣어 두면, 운영 반영은 `서버 배포 -> 설정 저장 -> rules 잠금` 순서로 끊김 없이 옮길 수 있습니다.

## 6. 배포 전에는 읽기 전용 API 조회로 외부 blocker를 먼저 잡는다

- 코드와 로컬 emulator가 멀쩡해도, 실제 Firebase 프로젝트 쪽 API가 꺼져 있으면 첫 배포에서 바로 막힙니다.
- 이번에는 `npx firebase-tools functions:list --project skct-tool` 같은 읽기 전용 조회만으로도 `Cloud Functions API 비활성`을 실제 배포 전에 확인할 수 있었습니다.
- 다음부터는 배포 직전 점검에 `로그인 확인 -> 프로젝트 조회 -> Functions 목록 조회`를 넣어 두면, 권한 문제인지 API 비활성인지 더 빨리 분리할 수 있습니다.

## 7. 공개 배포에서 숨길 경로는 빠뜨리지 말고 명시적으로 막는다

- 이번에는 `public-clean` 브랜치에서 `admin.html`을 빼면 끝이라고 봤지만, 라이브 URL에서는 기존 관리자 경로가 그대로 응답하는 상황을 실제로 확인했습니다.
- 이런 종류의 경계는 `파일이 없으면 괜찮다`에 기대지 말고, 공개 배포물 안에 아예 차단용 `admin.html`을 명시적으로 넣는 편이 안전합니다.
- 앞으로 공개에서 제외할 경로는 `삭제`보다 `안전한 차단 페이지로 대체`를 우선 검토합니다.

## 8. 새 레이아웃과 운영 siteTextConfig가 섞이면 화면은 반만 바뀐다

- 이번에는 고급 안내 모달 레이아웃을 새로 바꿨는데도, 실제 화면에서는 Firebase `siteTextConfig`의 예전 문구 일부가 다시 올라와 설명이 반쯤만 정리된 상태가 남았습니다.
- 이런 경우 HTML과 CSS만 고치면 끝이 아니라, `site-text-config.js` 기본값과 `LEGACY_SITE_TEXT_DEFAULTS`에 현재 운영 문구까지 함께 넣어야 새 구조와 새 문구가 같이 살아납니다.
- 사용자 체감 화면을 손댈 때는 `정적 코드 수정 -> 현재 운영 문구 실제값 확인 -> legacy migration 보강` 순서까지 묶어서 보는 편이 안전합니다.

## 9. 팝업 문제는 데스크톱 화면이 아니라 실제 팝업 폭으로 검증해야 한다

- 이번에는 일반 데스크톱 창에서는 멀쩡해 보여도, 실제 `window.open`으로 열린 팝업 안쪽 폭에서는 모달과 OMR이 다르게 보였습니다.
- 이런 문제는 `브라우저 전체 폭`만 보고 판단하면 놓치기 쉽고, 실제 팝업 안쪽 폭이나 `body.popup-mode` 상태에서 한 번 더 봐야 잡힙니다.
- 다음부터는 팝업 관련 수정 뒤에 `실제 popup open 결과 폭 확인 -> OMR 열기 -> 안내 모달 열기` 순서로 확인하는 편이 안전합니다.

## 10. 안내 모달은 내용이 아니라 역할부터 분리해야 겹침이 줄어든다

- 이번에는 같은 내용을 조금씩 줄여 적는 것만으로는 일반 도움말, 고급 신청 안내, 고급 활용 안내의 겹침이 잘 사라지지 않았습니다.
- 해결은 문장을 예쁘게 다듬는 것보다 `이 창이 무엇을 설명하는지`를 먼저 갈라 놓는 쪽이었습니다.
- 앞으로는 `일반 흐름`, `신청/진입`, `들어온 뒤 사용법`처럼 모달별 책임을 먼저 정한 뒤 문구를 써야 설명이 짧아져도 덜 헷갈립니다.

## 11. 좁은 창에서는 설명보다 조작이 먼저 보여야 한다

- 이번에는 고급 OMR와 여러 버튼 내부 모달에서 설명 문단이 기본 화면을 밀어내 버튼과 입력칸을 가리는 문제가 반복됐습니다.
- 좁은 화면에서는 `행동에 바로 필요한 버튼/입력칸`이 먼저 보여야 하고, 보충 설명은 헤더의 `?` 같은 빠른 도움말 뒤로 보내는 편이 안전합니다.
- 다음부터는 팝업, 작은 모달, 버튼 내부 화면을 만들 때 `기본 노출은 한 줄 요약`, `나머지는 ? 도움말` 원칙으로 먼저 설계하는 편이 좋습니다.

## 12. 운영 도구는 절차 설명보다 실행 스크립트가 먼저다

- 관리자 페이지처럼 공개 URL로 열 수 없고, 로컬 서버와 보조 브리지가 필요한 도구는 사용법을 문서로만 설명하면 매번 번거롭고 실수도 늘어납니다.
- 이번에는 `scripts/open_local_admin.cmd`와 `scripts/open_local_admin.ps1`로 로컬 서버 실행, 브라우저 열기, 키 브리지 시작을 한 번에 묶었습니다.
- 다음부터는 운영자 전용 흐름이 2단계를 넘기면, 설명 문서와 함께 `원클릭 실행 스크립트`를 같이 두는 편이 낫습니다.

## 13. VS Code를 쓰는 작업은 Task까지 묶어야 실제로 편하다

- 스크립트가 있어도 매번 탐색기에서 더블클릭하거나 경로를 치는 건 다시 귀찮아집니다.
- 이번에는 `.vscode/tasks.json`에 `로컬 관리자 페이지 열기/종료` 작업을 추가해 `Tasks: Run Task`만으로 실행되게 묶었습니다.
- 반복 실행이 필요한 운영 보조 도구는 스크립트만 만들지 말고, 자주 쓰는 편집기 진입점까지 같이 두는 편이 좋습니다.

## 14. VS Code PowerShell shell task는 .cmd 실행에서 `-Command` 오류를 만들 수 있다

- 이번에는 `.cmd` 파일을 shell task로 바로 물리면 VS Code가 PowerShell `-Command` 경로로 감싸면서 `-Command: The term '-Command' is not recognized` 오류가 났습니다.
- Windows에서 `.cmd`를 안정적으로 띄우려면 shell task보다 `type: process`와 `cmd.exe /d /c` 조합이 훨씬 안전합니다.
- 다음부터는 VS Code에서 배치 파일을 실행할 때 `shell`보다 `process`를 먼저 검토하는 편이 낫습니다.

## 15. `window.open(..., 'noopener')`는 성공해도 반환값이 `null`일 수 있다

- 이번에는 자료 보관함을 새 탭으로 열 때 `window.open(url, '_blank', 'noopener')` 뒤에 `if (!popup) location.assign(url)` fallback을 붙여 두었습니다.
- 문제는 일부 브라우저에서 `noopener`를 쓰면 새 탭을 열어도 반환값이 `null`일 수 있어, 결과적으로 `새 탭도 열리고 현재 페이지도 같이 이동`하는 이중 이동이 생길 수 있다는 점입니다.
- 다음부터는 `noopener`가 필요한 새 탭 열기에서 반환값을 실패 여부로 바로 해석하지 말고, `window.open(url, '_blank')` 후 `opener = null` 처리나 앵커 기반 열기처럼 반환값 의미가 더 분명한 방식을 쓰는 편이 안전합니다.

## 16. Firebase Auth가 살아 있어도 RTDB 쓰기 세션은 따로 꼬일 수 있다

- 이번에는 관리자 페이지에서 이미 로그인된 상태인데도 공지 저장이 `permission-denied`로 막히는 상황이 나왔습니다.
- 원인은 `auth.currentUser`와 `getIdToken()`은 살아 있어도, Realtime Database 연결이 예전 인증 상태를 물고 있으면 첫 쓰기에서 바로 권한 거절이 날 수 있다는 점입니다.
- 다음부터는 관리자 저장처럼 중요한 쓰기 경로에서 `토큰 확인 -> 필요 시 RTDB 재연결 -> 1회 자동 재시도`를 공통 함수로 묶어 두는 편이 안전합니다.

## 17. RTDB wildcard 쓰기 규칙은 명시적 자식 노드의 실패를 자동으로 덮지 못한다

- 이번에는 `config/$key/.write = auth != null`가 있으니 관리자 `config/*` 저장도 될 거라고 보기 쉬웠지만, 실제로는 `config/notice_sidebar`, `config/notice_help`처럼 명시적으로 선언된 자식 경로가 있어 첫 저장이 계속 `permission-denied`로 막혔습니다.
- 해결은 `config` 루트에 아예 `.write = auth != null`를 두는 쪽이었습니다.
- 다음부터 RTDB rules에서 루트 아래 명시적 자식과 wildcard를 섞어 쓸 때는, `읽기/쓰기 권한이 실제로 어느 레벨에서 평가되는지`를 REST 호출이나 실제 관리자 버튼으로 확인해야 합니다.

## 18. 관리자 액션 상태줄을 쓸 때는 조용한 `return`보다 예외 전파가 낫다

- 이번에는 신청 승인/반려/차단 같은 함수가 필수값이 없으면 내부에서 메시지만 그리고 `return`해 버리는 분기가 있었습니다.
- 이런 구조에서는 바깥 공통 래퍼가 실패를 모르고 `완료` 상태를 찍을 수 있습니다.
- 다음부터 관리자 공통 상태줄이나 공통 토스트 래퍼를 쓸 때는, `검증 실패 -> Error throw -> 바깥 catch에서 상태/토스트/로그 처리` 구조로 맞추는 편이 더 안전합니다.
