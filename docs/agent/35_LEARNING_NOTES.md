# SKCT Tool 개발 및 문제 해결 기록 (Learning Notes)
작성일시: 2026-04-05 11:49:30 KST

본 문서는 SKCT 연습 도구의 고도화 작업간 발생했던 핵심 기술적 문제들과, 이를 어떤 방식으로 해결했는지 초보자 및 향후 유지보수 담당자가 쉽게 이해할 수 있도록 기록한 문서입니다.

## 1. 실시간 접속자 무한 증식(좀비 노드) 버그와 해결법
### 🚨 문제 상황
최초에는 방문자가 들어오면 Firebase RTDB에 `push()`로 접속 상태(`true`)를 기록하고, 나갈 때 `onDisconnect()`를 이용해 상태를 삭제하는 가장 기본적인 방식을 사용했습니다. 하지만 방문자가 1시간 동안 F5(새로고침)를 10번 누르면, 이전 접속 기록이 미처 지워지기 전에 새로운 기록이 쌓여 **1명의 유저가 10명으로 뻥튀기(좀비 노드)**되는 문제가 발생했습니다.

### 💡 1차 해결 (Session ID 도입)
브라우저 탭마다 발급되는 `sessionStorage`를 이용해 고유한 ID(`sess_xxxx`)를 부여했습니다. 유저가 새로고침을 하더라도 새 데이터를 쌓는 대신 본인의 ID 자리에 덮어쓰기(`set`)를 하게 만들어 증식 버그를 1차적으로 막았습니다.

### 🚀 최종 해결 (Heartbeat TTL 시스템 도입)
Session ID 도입에도 불구하고 스마트폰에서 브라우저를 훅 꺼버리거나, 인터넷이 끊긴 채로 잠들면 `onDisconnect` 신호가 증발하여 서버에 가짜 데이터가 영원히 남는 문제가 여전히 존재했습니다.
이를 완전히 끊어내기 위해 대형 게임 서버 등에서 사용하는 **'하트비트 생존 추적(Heartbeat TTL)'** 기법을 적용했습니다.

*   **동작 방식**: 클라이언트는 30초마다 `Date.now()`(현재 시각)를 서버에 갱신합니다(심장 박동).
*   **검증 방식**: 화면에 "현재 접속자"를 렌더링할 때, 서버 데이터를 가져와서 **최근 90초 이내에 시각이 갱신된 사람만 카운트**합니다.
*   **결과**: 강제 종료, 캐시 지연, 와이파이 끊김 등 그 어떤 변수에도 90초 뒤면 통계에서 무조건 깨끗하게 사라지는 완벽한 자동 정화 시스템이 완성되었습니다.

## 2. 일일 방문자 중복 집계 차단 및 KST 타임존 문제
### 🚨 문제 상황
처음엔 `new Date().toISOString()` 함수를 사용해 날짜를 계산했습니다. 이 함수는 영국 시간(UTC)을 기준이므로, 한국 시간(KST)으로 오전 9시가 되어야 비로소 다음 날로 날짜가 바뀌는 치명적인 집계 오류가 있었습니다.

### 💡 해결 (Local Time 기반 원자적 연산)
*   **타임존 해결**: `d.getFullYear()`, `d.getMonth()`, `d.getDate()` 등 브라우저의 로컬 기기 시간을 그대로 뽑아내어 조합하는 방식으로 전환하여 한국 시간 자정에 정확히 날짜가 넘어가도록 수정했습니다.
*   **중복 차단**: 브라우저의 `localStorage` 공간에 방문 날짜를 도장 찍어두고, 같은 날 재방문 시에는 카운트를 올리지 않는 `if (lastVisitDate === today) return;` 방어벽을 쳤습니다.
*   **안전한 카운트업**: 여러 명이 동시에 방문하여 카운트가 씹히는 충돌을 막기 위해 Firebase의 `runTransaction()` 메서드를 사용했습니다. 이 함수는 DB 값을 100% 안전하고 원자적으로 1씩 올려주는 역할을 합니다.

## 3. 후원자 롤링 애니메이션 UI 잘림 현상 방어
### 🚨 문제 상황
투네이션 후원자 내역을 팝업 모달 하단에서 위로 스르륵 올라오는 애니메이션 티커로 구현했습니다. 그런데 후원 메시지가 너무 길거나 창이 좁아지면, 텍스트가 아래로 뚝 떨어지면서 화면 밖으로 잘리거나 날짜가 가려지는 에러가 발생했습니다.

### 💡 해결 (CSS Flexbox & Line-Clamp)
*   **레이아웃 4배 확장**: 높이를 20px에서 80px로 키워 충분한 여유 공간을 확보했습니다.
*   **구조 분리**: 상단 영역(이름+금액, 빈 공간, 날짜)과 하단 영역(메시지 본문)을 분리 배치하여 서로 침범하지 않게 했습니다.
*   **말줄임표(...) 최후 방어선**: 하단 메시지에 CSS `-webkit-line-clamp: 3` 속성을 적용했습니다. 이 마법의 속성은 글이 아무리 길어도 최대 3줄까지만 표시하고, 그 이후는 자동으로 `...`으로 잘라주는 브라우저 공식 지원 기능입니다. 이로써 어떤 환경에서도 디자인이 깨지지 않게 되었습니다.

## 4. 자동 팝업 진입 회귀와 타이머 총시간 불일치
### 🚨 문제 상황
업데이트 과정에서 메인 페이지가 열리자마자 `window.open()`으로 팝업을 강제로 띄우고, 원본 화면을 곧바로 덮어쓰는 로직이 다시 들어왔습니다. 이 때문에 사용자는 도구 첫 화면 자체를 제대로 보지 못하고, 브라우저가 팝업을 막으면 버튼을 누를 기회도 없이 "팝업 모드 실행 중..." 화면에 갇히게 되었습니다.

타이머도 별도 문제가 있었습니다. 기본값이 `전체 75분 / 과목당 15분 / 쉬는 시간 1분`인데, 실제 필요 시간은 `15 x 5 + 1 x 4 = 79분`입니다. 즉 기본값만 써도 마지막 과목 전에 전체 타이머가 먼저 끝나 버리는 모순이 있었습니다.

### 💡 해결
*   **팝업은 수동 버튼만 허용**: 페이지 진입 즉시 자동 팝업을 여는 로직을 제거하고, 사용자가 `창 제한 해제` 버튼을 눌렀을 때만 팝업 모드가 열리도록 되돌렸습니다.
*   **총시간 최소값 보정**: `전체 제한 시간`이 `과목 시간 합 + 쉬는 시간 합`보다 작으면 자동으로 올려서, 불가능한 타이머 설정이 저장되지 않게 만들었습니다.
*   **관리자 기본값도 동일 규칙 적용**: 사용자 화면만 고치면 다시 원격 기본값에서 깨질 수 있으므로, 관리자 페이지 저장 시에도 같은 보정 로직을 넣었습니다.

## 5. 점검 중 개발자 전용 우회 링크
### 🚨 문제 상황
운영 중에는 일반 사용자를 `update.html`로 보내 막아두고 싶지만, 개발자는 배포된 실제 웹에서 바로 수정 결과를 확인해야 할 때가 있습니다. 기존 구조는 점검이 켜지면 모두 같은 점검 페이지로 보내기만 해서, 운영 중 검증이 매우 불편했습니다.

## 6. 자동 팝업 대신 "수동 팝업 진입점 2개"가 더 안전한 이유
### 🚨 문제 상황
사용자들은 원래 팝업 모드로 많이 사용했기 때문에, 자동 팝업을 없애면 "예전처럼 좁게 못 쓴다"는 불편을 느낄 수 있습니다. 그렇다고 페이지가 열리자마자 자동으로 `window.open()`을 실행하면, 브라우저 팝업 차단 정책 때문에 다시 팝업이 막히고 원본 화면까지 덮어쓰는 회귀가 발생할 수 있습니다.

### 💡 해결
*   **자동 실행은 금지**: 팝업은 반드시 사용자의 클릭 이벤트에서만 열리게 둡니다. 이렇게 해야 브라우저가 정상적인 사용자 동작으로 판단해 차단 확률이 크게 줄어듭니다.
*   **진입점은 2개 제공**: 사용성이 떨어지지 않도록 상단 버튼과 좌측 탭 목록에 모두 `화면 더 줄이기` 버튼을 둡니다.
*   **실제 파일은 그대로 유지**: 팝업이라고 해서 다른 HTML을 쓰지 않고, 같은 `index.html`을 새 창으로 여는 방식으로 맞춥니다. 그러면 일반 페이지와 팝업 페이지의 기능 차이는 "창 크기를 더 좁게 조절할 수 있느냐"뿐입니다.
*   **팝업 내부에서는 버튼 숨김**: 이미 팝업 안에 들어간 뒤에는 다시 팝업을 띄울 이유가 없으므로, 상단 버튼과 좌측 탭 버튼을 숨겨 UI를 단순하게 유지합니다.
*   **경고창은 실패할 때만**: 열기 전마다 설명 alert를 띄우면 흐름이 끊기므로, 실제로 팝업이 차단된 경우에만 해제 안내를 보여주는 편이 더 자연스럽습니다.

### 💡 해결
*   **쿼리 파라미터 우회**: `index.html?dev=1`처럼 접속한 현재 URL에서만 점검 우회를 허용하도록 했습니다.
*   **일반 주소는 항상 차단 유지**: 개발자 링크로 한 번 들어갔다고 해서 이후 일반 주소까지 계속 열리면 안 되므로, `sessionStorage` 저장 방식은 제거했습니다.
*   **점검 페이지 직접 진입도 지원**: `update.html?...&dev=1`로 접속하면 즉시 `index.html?dev=1`로 넘겨 실제 화면을 확인할 수 있게 했습니다.

## 6. 관리자 미리보기에서 로컬 설정이 원격 기본값을 덮는 문제
### 🚨 문제 상황
관리자 페이지에서 타이머 기본값을 바꾸고 저장해도, 곧바로 `사이트 보기`로 들어갔을 때 변경이 안 보이는 문제가 있었습니다. 원인은 미리보기 브라우저가 예전에 저장해 둔 `localStorage` 값을 먼저 읽고 있었기 때문입니다.

### 💡 해결
*   **관리자 미리보기 모드 추가**: `preview=1` 쿼리로 들어온 현재 URL에서만 미리보기 동작을 활성화합니다.
*   **미리보기에서는 로컬값 무시**: 이 모드에서는 타이머, 문항 가이드, 레이아웃 비율의 로컬 저장값을 읽지 않고, 관리자 페이지에서 저장한 Firebase 기본값을 항상 우선 반영하도록 했습니다.
*   **관리자 링크도 같이 수정**: 관리자 상단 `사이트 보기` 링크가 자동으로 `index.html?dev=1&preview=1`를 실제 사용자와 같은 팝업 창 형태로 열리게 바꿨습니다.

## 7. 설정 반영 지연의 실제 원인: 10분 config 캐시
### 🚨 문제 상황
관리자에서 `전체 시간`을 바꿔 저장했는데도 바로 안 보이는 경우가 있었던 이유는, 사용자 페이지가 `config` 전체를 `sessionStorage`에 10분 동안 캐시하고 있었기 때문입니다. 그래서 Firebase에는 새 값이 저장돼도, 같은 탭에서는 옛 설정을 계속 읽을 수 있었습니다.

### 💡 해결
*   **config 캐시 제거**: 페이지 진입 시마다 Firebase `config`를 다시 읽도록 바꿨습니다.
*   **점검 차단도 즉시 반영**: 이 변경으로 `maintenance.enabled`를 켜면 다음 새로고침부터 바로 차단되고, 관리자에서 시간을 바꾸면 다음 새로고침부터 바로 새 값이 보입니다.

## 8. “저장은 되는데 왜 75분이 안 보일까?”의 진짜 원인
### 🚨 문제 상황
겉으로 보면 관리자 페이지에서 `전체 시간`을 `75`로 바꿔 저장해도 다시 `79`처럼 보였기 때문에, 처음엔 Firebase 저장 실패처럼 보였습니다. 하지만 실제 원인은 저장이 아니라 **입력값을 저장 직전에 다시 보정하는 프론트엔드 함수**였습니다.

### 💡 해결
*   **최소 총시간 강제 보정 제거**: `normalizeTimerDefaults()`와 사용자 페이지의 `clampTotalMins()` 계열 로직이 `75`를 자동으로 `79` 이상으로 올리고 있었기 때문에, 이 강제를 제거했습니다.
*   **관리자 저장과 사용자 적용 규칙 일치**: 이제 관리자 화면과 사용자 화면 모두 `75`를 그대로 유효한 기본값으로 받아들입니다.
*   **원격 기본값도 직접 갱신 확인**: 로컬 수정만으로 끝내지 않고, 실제 관리자 저장 동선으로 다시 `75`를 써서 프리뷰 화면에 `75:00`이 반영되는 것까지 확인했습니다.

## 9. 정답 입력 모드에서는 타이머가 멈춰야 하는 이유
### 🚨 문제 상황
기존에는 `정답 입력 모드`가 OMR의 색상과 버튼 동작만 바꾸고, 타이머는 계속 돌아갔습니다. 그래서 채점용 정답을 적는 동안에도 전체 시간과 문항 가이드 시간이 계속 줄어드는 어색한 동작이 생겼습니다.

### 💡 해결
*   **공통 정지 헬퍼 도입**: `stopTimer()` 하나로 인터벌 해제, 버튼 문구 복원, 가이드 타이머 숨김까지 함께 처리하도록 묶었습니다.
*   **모든 진입 경로에 동일 적용**: 사용자가 직접 `정답 입력 모드` 버튼을 눌렀을 때뿐 아니라, `채점 및 통계 확인` 버튼이 정답 입력 모드로 자동 전환시키는 경우에도 같은 정지 로직을 태웠습니다.
*   **가이드 타이머도 자동 정지**: 가이드 표시는 `timerIsRunning` 조건을 보기 때문에, 전체 타이머를 멈추면 문항 가이드도 같이 멈추고 숨겨집니다.

## 10. 상세 통계는 “과목별 카드”로 묶는 편이 읽기 쉽다
### 🚨 문제 상황
기존 상세 통계는 과목별 오답/미응답과 과목별 소요 시간이 서로 떨어져 출력되었습니다. 그래서 어떤 시간이 어느 과목의 어떤 오답과 연결되는지 한눈에 파악하기 어려웠습니다.

### 💡 해결
*   **과목별 접기 구조 적용**: HTML `details/summary` 요소를 사용해 과목마다 접고 펼칠 수 있게 만들었습니다.
*   **기본은 접힘 상태**: 처음에는 전체 목록만 가볍게 보고, 필요할 때만 과목 하나를 열어보도록 설계했습니다.
*   **오답과 시간 기록을 한 묶음으로 배치**: 같은 과목 카드 안에서 `오답/미응답`과 `소요 시간`을 같이 보여주도록 바꿔서 탐색 비용을 줄였습니다.

## 11. “원격은 75인데 화면은 왜 79인가?”를 구분하는 방법
### 🚨 문제 상황
이번 케이스에서는 Firebase 원격 설정값 `config/timerDefaults`가 이미 `75`로 정상 저장돼 있었는데도, 실제 웹에서는 `79`가 계속 보였습니다. 즉 저장이 안 된 것이 아니라 **브라우저가 예전에 저장한 로컬 기본값이 계속 이기고 있던 상황**이었습니다.

### 💡 해결
*   **원격값과 로컬값을 먼저 분리해서 확인**: Firebase REST로 `config/timerDefaults.json`을 직접 확인해 원격값이 실제로 무엇인지 먼저 검증했습니다.
*   **오래된 기본값 패턴만 자동 폐기**: `localStorage.skct_timer_cfg`가 예전 기본값 `79/15/1`이고, 사용자가 직접 저장했다는 표식(`source: 'user'`)이 없으면 오래된 찌꺼기로 간주하고 삭제합니다.
*   **이후 사용자 저장은 명시적으로 표기**: 새로 저장할 때는 `source: 'user'`를 함께 넣어, 진짜 사용자 커스텀 값은 앞으로 자동 삭제되지 않게 했습니다.
*   **캐시 버스팅 추가**: `main.js` 로드 경로에 버전 쿼리를 붙여 브라우저가 오래된 JS를 계속 쓰는 문제도 줄였습니다.
