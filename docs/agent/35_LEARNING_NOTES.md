# SKCT Tool 개발 및 문제 해결 기록 (Learning Notes)
작성일시: 2026-04-08 23:34:48 KST

본 문서는 SKCT 연습 도구의 고도화 작업간 발생했던 핵심 기술적 문제들과, 이를 어떤 방식으로 해결했는지 초보자 및 향후 유지보수 담당자가 쉽게 이해할 수 있도록 기록한 문서입니다.

## 0-6. 관리자 페이지가 길어질수록 "기능 추가"보다 "탭 분리"가 먼저다
### 🚨 문제 상황
운영 기능이 많아지면 한 화면에 전부 쌓는 방식은 금방 한계가 옵니다. 특히 구독, 기본값, 문구, 공지, 커뮤니티를 한 줄로 늘어놓으면 스크롤이 길어지고, "지금 내가 어느 작업 묶음 안에 있는지"가 흐려집니다.

### 💡 해결
이럴 때는 기존 기능을 지우지 말고 **작업 묶음 단위로 탭 패널에 나누는 방식**이 가장 안전합니다.

*   **기능은 그대로 두고 위치만 재배치**: 기존 카드와 DOM을 그대로 탭 패널 안으로 옮기면 이벤트와 저장 로직을 다시 연결할 필요가 적습니다.
*   **자주 쓰는 흐름을 앞 탭에 배치**: 이 프로젝트는 `구독 -> 기본 설정 -> 문구/공지 -> 커뮤니티` 순서가 자연스러워서 그 순서대로 탭을 두는 편이 이해가 쉽습니다.
*   **빠른 이동도 탭을 알아야 함**: 버튼이 단순 스크롤만 하면 숨겨진 탭 안의 카드로는 못 갑니다. 대상 카드가 어느 탭에 있는지 먼저 찾고, 탭을 연 다음 스크롤해야 동작이 자연스럽습니다.

즉, 긴 관리자 페이지를 정리할 때는 새 UI를 통째로 다시 만드는 것보다 **기존 동작을 보존한 채 작업 묶음별 탭으로 감싸는 방식**이 훨씬 안전합니다.

## 0-5. "중복 신청"은 서버가 아니라도 프론트에서 먼저 많이 막을 수 있다
### 🚨 문제 상황
정적 사이트에서 Firebase RTDB로 바로 신청을 저장하면, 사용자가 버튼을 두 번 누르거나 저장 결과를 못 보고 다시 누르는 순간 같은 내용이 여러 건 쌓일 수 있습니다. 이번 운영 데이터도 실제로 두 사람의 신청이 짧은 간격으로 여러 번 들어와 있었습니다.

### 💡 해결
서버 함수가 없어도 아래 2가지는 프론트에서 바로 막을 수 있습니다.

*   **제출 중 버튼 잠금**: 저장 요청이 끝날 때까지 버튼을 비활성화하면, 가장 흔한 더블클릭 중복을 막을 수 있습니다.
*   **최근 동일 신청 fingerprint 저장**: 이메일, 희망 ID, 시작일, 닉네임, 후원명, 플랜 같은 핵심 필드를 묶어 최근 신청 fingerprint를 남기면, 같은 브라우저에서 곧바로 다시 보내는 실수를 줄일 수 있습니다.
*   **이미 저장된 신청번호를 안내**: 막기만 하면 사용자는 불안해합니다. "이미 저장된 신청번호가 있으니 먼저 조회해 보라"는 안내가 같이 있어야 혼란이 줄어듭니다.

즉, 중복 방지는 거대한 백엔드 로직이 먼저가 아니라 **클릭 잠금 + 최근 동일 요청 감지** 같은 작은 프론트 안전장치부터 챙기는 것이 실전에서는 훨씬 효과적입니다.

## 0-3. 관리자 페이지는 "기능 추가"보다 "운영 순서 정리"가 먼저다
### 🚨 문제 상황
관리자 페이지에 기능이 많아지면, 실제로는 정상 동작하는 버튼도 사용자가 찾지 못하거나 잘못된 순서로 누르게 됩니다. 특히 이 프로젝트처럼 `고급 구독`, `수동 신청`, `문구 수정`, `공지`, `커뮤니티 관리`가 한 화면에 같이 있으면 더 그렇습니다.

### 💡 해결
기존 기능을 새로 만드는 것보다 먼저, **운영자가 실제로 쓰는 순서대로 화면을 다시 묶는 것**이 중요합니다.

*   **빠른 이동 추가**: 자주 쓰는 기능으로 한 번에 점프할 수 있어야 긴 관리자 화면에서 길을 잃지 않습니다.
*   **같은 목적끼리 묶기**: `고급 구독`과 `수동 신청`은 모두 "구독 운영"이므로 한 카드로, `후원 문구`, `사이트 텍스트`, `후원 CSV`는 "문구/콘텐츠 운영"으로 묶는 편이 자연스럽습니다.
*   **저장 버튼 이름을 사실대로 쓰기**: 일부만 저장하는데 `전체 저장`이라고 쓰면 오해가 생깁니다. 저장 범위를 실제 동작과 맞춰야 합니다.
*   **기존 DOM을 옮기는 방식이 안전함**: HTML을 통째로 다시 쓰기보다, 기존 요소를 그대로 다른 카드로 옮기면 ID와 이벤트를 유지할 수 있어 기능 소실 위험이 작습니다.

즉, 관리자 화면 정리는 디자인 문제가 아니라 **운영 사고를 줄이는 안전장치**로 보는 편이 맞습니다.

## 0-4. 웹페이지가 로컬 파일을 자동으로 읽으려면 "브라우저 권한"이 먼저다
### 🚨 문제 상황
정적 관리자 페이지에서 "내 PC 특정 경로에 있는 개인키 파일을 자동으로 읽어오고 싶다"는 요구가 생길 수 있습니다. 하지만 일반 웹페이지는 보안상 사용자의 PC 임의 경로를 직접 읽을 수 없습니다.

### 💡 해결
이럴 때는 **브라우저가 허용한 폴더 핸들(File System Access API)** 을 이용하는 방식이 가장 현실적입니다.

*   **한 번만 폴더 연결**: 사용자가 `키 폴더 연결` 버튼으로 폴더를 한 번 고르면, 브라우저가 그 폴더에 대한 읽기 권한을 기억할 수 있습니다.
*   **다음부터 자동 읽기**: 이후에는 관리자 페이지가 저장된 폴더 핸들에서 표준 파일명(`manual_subscription_request_private.pem`, `manual_subscription_license_private.pem`)을 자동으로 읽을 수 있습니다.
*   **임의 경로 직접 읽기는 불가**: 경로 문자열만 아는 상태에서 웹페이지가 바로 `C:\...` 파일을 읽는 것은 브라우저 보안상 막혀 있습니다.
*   **로컬 브리지는 예비책**: 필요하면 `127.0.0.1` 브리지 서버를 둘 수 있지만, 운영 `https` 페이지에서는 사설 네트워크 접근 제한 때문에 추가 제약이 생길 수 있습니다.

즉, "경로 자동 읽기" 요구는 결국 **브라우저 권한을 어떻게 안전하게 확보하느냐**의 문제로 봐야 합니다.

## 0. 지시 용어를 먼저 고정하면 왜 좋은가
### 🚨 문제 상황
작업이 길어질수록 사용자가 `normal`, `고급버전`, `테스트사이트`, `실서버` 같은 말을 섞어 쓰고, 에이전트도 그때그때 문맥으로 해석하게 됩니다. 그러면 같은 말을 두고도 "로컬 반영인지", "스테이징 배포인지", "운영 적용인지"가 어긋날 수 있습니다.

### 💡 해결
`docs/SKCT_TOOL_지시_용어집.md`처럼 **용어집 문서를 따로 만들어 고정 표현을 먼저 합의**하면 이후 작업이 훨씬 빨라집니다.

*   **환경 이름 고정**: `로컬 / 스테이징 / 운영`
*   **화면 종류 고정**: `일반모드 / 고급모드`
*   **작업 동사 고정**: `반영`, `이식`, `검증`, `기본값 저장`

이렇게 해두면 "스테이징 반영 후 운영 이식" 같은 말을 모두가 같은 뜻으로 이해하게 되어, 배포 사고와 소통 비용을 동시에 줄일 수 있습니다.

## 0-1. OMR 번호 정렬은 숫자 폭보다 "열 폭 고정"이 더 중요하다
### 🚨 문제 상황
OMR에서 `1.`부터 `9.`까지는 얼핏 정렬돼 보이는데, `10.`부터는 숫자 자릿수가 늘어나면서 선택지 버튼 시작 위치가 밀려 보일 수 있습니다. 여기에 버튼 간격까지 인라인 스타일과 CSS가 서로 다르게 잡혀 있으면, 화면 폭이 좁을수록 행마다 위치가 어긋나 보입니다.

### 💡 해결
번호 텍스트 길이에 맡기지 말고, **문항 번호 칸 자체의 폭을 고정**해야 합니다.

*   **번호 열 고정**: `.q-num`에 고정 폭과 `flex-basis`를 주면 `1.`이든 `10.`이든 항상 같은 위치에서 끝납니다.
*   **숫자 폭 보정**: `font-variant-numeric: tabular-nums;`를 쓰면 숫자마다 폭이 들쭉날쭉한 현상을 더 줄일 수 있습니다.
*   **간격 기준 하나로 통일**: 렌더링 HTML에 박아둔 `gap:4px` 같은 인라인 스타일을 제거하고 CSS 공통값만 따르게 해야 나중에 OMR 폭을 조정해도 한 번에 제어됩니다.

즉, OMR 정렬 문제는 "폰트를 바꾸는 문제"가 아니라 **번호 열을 하나의 고정 컬럼처럼 다루는 문제**로 보는 것이 맞습니다.

## 0-2. "기능이 안 된다"와 "기능이 보이지 않는다"는 다르다
### 🚨 문제 상황
사용자 제보를 보면 "건너뛴 문제는 오답풀이 때 정답 입력이 안 된다"처럼 들릴 수 있습니다. 그런데 코드를 직접 보면, 실제로는 이미 입력이 가능한데 화면에서 그 사실이 잘 안 보여서 막힌 것처럼 느껴지는 경우가 있습니다.

### 💡 해결
이럴 때는 무작정 로직부터 뜯지 말고 **먼저 사실 여부를 확인**해야 합니다.

*   **로직 확인**: 이 프로젝트에서는 `정답 입력 모드`에서 모든 문항의 선택 버튼이 `correctAnswers`로 들어가므로, 건너뛴 문제도 이미 입력 가능했습니다.
*   **진짜 문제는 UX**: 사용자는 "가능한지"를 즉시 알아야 하므로, `미응답도 클릭해 정답 입력 가능` 같은 짧은 안내를 보여주는 편이 더 효과적입니다.
*   **불필요한 로직 변경 방지**: 원래 되던 기능을 다시 손대면 오히려 회귀가 날 수 있으므로, 사실이 아닌 제보는 안내 개선으로 푸는 것이 안전합니다.

즉, 사용자 제보를 받았을 때는 **버그인지, 안내 부족인지**를 먼저 가르는 습관이 중요합니다.

## 1. 실시간 접속자 무한 증식(좀비 노드) 버그와 해결법
### 🚨 문제 상황
최초에는 방문자가 들어오면 Firebase RTDB에 `push()`로 접속 상태(`true`)를 기록하고, 나갈 때 `onDisconnect()`를 이용해 상태를 삭제하는 가장 기본적인 방식을 사용했습니다. 하지만 방문자가 1시간 동안 F5(새로고침)를 10번 누르면, 이전 접속 기록이 미처 지워지기 전에 새로운 기록이 쌓여 **1명의 유저가 10명으로 뻥튀기(좀비 노드)**되는 문제가 발생했습니다.

### 💡 1차 해결 (Session ID 도입)
브라우저 탭마다 발급되는 `sessionStorage`를 이용해 고유한 ID(`sess_xxxx`)를 부여했습니다. 유저가 새로고침을 하더라도 새 데이터를 쌓는 대신 본인의 ID 자리에 덮어쓰기(`set`)를 하게 만들어 증식 버그를 1차적으로 막았습니다.

### 🚀 최종 해결 (Heartbeat TTL 시스템 도입)
Session ID 도입에도 불구하고 스마트폰에서 브라우저를 훅 꺼버리거나, 인터넷이 끊긴 채로 잠들면 `onDisconnect` 신호가 증발하여 서버에 가짜 데이터가 영원히 남는 문제가 여전히 존재했습니다.
이를 완전히 끊어내기 위해 대형 게임 서버 등에서 사용하는 **'하트비트 생존 추적(Heartbeat TTL)'** 기법을 적용했습니다.

*   **동작 방식**: 클라이언트는 30초마다 `Date.now()`(현재 시각)를 서버에 갱신합니다(심장 박동).
*   **검증 방식**: 화면에 "현재 접속자"를 렌더링할 때, 서버 데이터를 가져와서 **최근 90초 이내에 시각이 갱신된 사람만 카운트**합니다.
*   **결과**: 강제 종료, 캐시 지연, 와이파이 끊김 등 그 어떤 변수에도 90초 뒤면 통계에서 무조건 깨끗하게 사라지는 완벽한 자동 정화 시스템이 완성되었습니다.

## 2. 일일 방문자 중복 집계 차단 및 KST 타임존 문제
### 🚨 문제 상황
처음엔 `new Date().toISOString()` 함수를 사용해 날짜를 계산했습니다. 이 함수는 영국 시간(UTC)을 기준이므로, 한국 시간(KST)으로 오전 9시가 되어야 비로소 다음 날로 날짜가 바뀌는 치명적인 집계 오류가 있었습니다.

### 💡 해결 (Local Time 기반 원자적 연산)
*   **타임존 해결**: `d.getFullYear()`, `d.getMonth()`, `d.getDate()` 등 브라우저의 로컬 기기 시간을 그대로 뽑아내어 조합하는 방식으로 전환하여 한국 시간 자정에 정확히 날짜가 넘어가도록 수정했습니다.
*   **중복 차단**: 브라우저의 `localStorage` 공간에 방문 날짜를 도장 찍어두고, 같은 날 재방문 시에는 카운트를 올리지 않는 `if (lastVisitDate === today) return;` 방어벽을 쳤습니다.
*   **안전한 카운트업**: 여러 명이 동시에 방문하여 카운트가 씹히는 충돌을 막기 위해 Firebase의 `runTransaction()` 메서드를 사용했습니다. 이 함수는 DB 값을 100% 안전하고 원자적으로 1씩 올려주는 역할을 합니다.

## 3. 후원자 롤링 애니메이션 UI 잘림 현상 방어
### 🚨 문제 상황
투네이션 후원자 내역을 팝업 모달 하단에서 위로 스르륵 올라오는 애니메이션 티커로 구현했습니다. 그런데 후원 메시지가 너무 길거나 창이 좁아지면, 텍스트가 아래로 뚝 떨어지면서 화면 밖으로 잘리거나 날짜가 가려지는 에러가 발생했습니다.

---

### 💡 해결 (CSS Flexbox & Line-Clamp)
*   **레이아웃 4배 확장**: 높이를 20px에서 80px로 키워 충분한 여유 공간을 확보했습니다.
*   **구조 분리**: 상단 영역(이름+금액, 빈 공간, 날짜)과 하단 영역(메시지 본문)을 분리 배치하여 서로 침범하지 않게 했습니다.
*   **말줄임표(...) 최후 방어선**: 하단 메시지에 CSS `-webkit-line-clamp: 3` 속성을 적용했습니다. 이 마법의 속성은 글이 아무리 길어도 최대 3줄까지만 표시하고, 그 이후는 자동으로 `...`으로 잘라주는 브라우저 공식 지원 기능입니다. 이로써 어떤 환경에서도 디자인이 깨지지 않게 되었습니다.

## 3-1. 방문자 수만 보고 가격을 정하면 왜 위험한가
### 🚨 문제 상황
서비스에 사람이 많이 들어오면 바로 "이 정도면 얼마를 받아도 되겠다"라고 생각하기 쉽습니다. 하지만 이 프로젝트의 Firebase 수치는 현재 `로그인 회원 수`가 아니라, `브라우저 기준 방문 수`와 `실시간 heartbeat 접속 수`에 가깝습니다. 즉 숫자가 커 보여도 실제 결제 가능한 사람 수와는 다를 수 있습니다.

### 💡 해결
가격은 방문자 수 하나로 정하지 말고, **전환율 기준으로 다시 해석**해야 합니다.

*   **누적 방문자 수는 모수일 뿐**: `total_visits`가 커도 같은 사람이 여러 날 다시 들어온 값이 섞일 수 있습니다.
*   **일 방문 수가 더 현실적 기준**: 현재처럼 시험 대비형 서비스는 `어제 방문자`, `최근 7일 평균 방문자`가 실제 결제 가능 모수에 더 가깝습니다.
*   **실시간 접속자는 혼잡도 지표**: `active_visitors`는 지금 서비스가 어느 정도 동시에 쓰이는지 보여주지만, 이 숫자를 곧바로 유료 전환 후보 수로 읽으면 과장됩니다.
*   **처음엔 기간제 이용권이 더 자연스럽다**: 시험 직전 단기 수요가 강한 서비스는 월 정기구독보다 `7일권`, `14일권`, `30일권`처럼 필요한 기간만 사는 구조가 결제 장벽이 더 낮습니다.

즉, 방문 지표는 "사람이 오고 있는가"를 보여주고, 가격은 "그중 몇 %가 돈을 낼 만큼 가치를 느끼는가"를 기준으로 잡아야 합니다.

## 3-2. 정적 사이트에서 "비밀글"을 만들 때 왜 UI 숨김만으로는 안 되는가
### 🚨 문제 상황
게시판 글을 화면에서만 숨기고 "비밀번호를 맞히면 보이게" 만들면 겉보기에는 비밀글처럼 보일 수 있습니다. 하지만 Firebase에 본문이 평문으로 저장되어 있으면, 개발자도구나 직접 API 호출만으로 내용을 읽을 수 있습니다. 즉, 이 방식은 `비밀글처럼 보이는 공개글`일 뿐입니다.

### 💡 해결
정적 사이트에서 사용자 본문을 진짜로 숨기려면 **저장 전에 브라우저에서 암호화**해야 합니다.

*   **사용자용 복호화 경로**: 사용자가 직접 정한 조회 비밀번호로만 자신의 신청 내용을 다시 열람할 수 있어야 합니다.
*   **관리자용 복호화 경로**: 운영자는 별도의 개인키로 전체 신청을 열람하고 승인해야 합니다.
*   **공개 목록 대신 조회형이 더 안전함**: 모든 비밀글 목록을 공개해 두면 열거와 탐색이 쉬워지므로, `신청번호 + 조회 비밀번호` 방식이 더 단단합니다.
*   **민감정보는 승인 후 발급이 낫다**: 사용자가 신청 단계에서 최종 서비스 비밀번호까지 남기게 하는 것보다, 운영자가 승인 시 계정을 발급하는 편이 저장 리스크와 운영 실수를 줄입니다.

즉, "숨김 UI"와 "암호화 저장"은 전혀 다릅니다. 후자만이 실제 보호에 가깝습니다.

## 3-3. 백업 복원 기능은 "복원"보다 "복원 직전 안전백업"이 더 중요하다
### 🚨 문제 상황
관리자 페이지에 백업 업로드와 복원 버튼만 만들면 편해 보입니다. 하지만 잘못된 파일을 올리거나, 예전 백업을 실수로 복원하면 현재 운영 데이터가 한 번에 덮어써질 수 있습니다. 결제나 구독이 들어간 뒤라면 이 실수는 치명적입니다.

### 💡 해결
복원 기능은 아래 2개가 같이 있어야 합니다.

*   **현재 상태 자동 백업**: 복원 버튼을 누르는 순간, 지금 DB 전체를 먼저 로컬 JSON으로 저장해야 합니다.
*   **전체 백업만 복원 허용**: 일부 섹션만 들어 있는 파일까지 복원 허용하면, 특정 경로만 남고 나머지가 비어버리는 사고가 날 수 있습니다.
*   **미리보기 후 복원**: 업로드 직후 파일 안에 무엇이 들어 있는지 요약을 보여줘야 사람이 마지막으로 한 번 더 확인할 수 있습니다.

즉, 복원 기능의 본질은 "파일 업로드"가 아니라 **실수해도 바로 되돌릴 수 있는 안전장치**입니다.

## 3-4. 이용자 안내는 시스템 설명보다 "지금 할 행동"이 먼저 보여야 한다
### 🚨 문제 상황
개발자는 자신이 만든 구조를 설명하고 싶어서 안내문에 내부 처리 방식을 많이 넣게 됩니다. 그래서 "수동 승인", "비밀글처럼 저장", "운영자가 확인 후 처리" 같은 표현이 길게 들어갑니다. 그런데 이용자는 구조 설명보다 **내가 지금 무엇을 하면 되는지**를 더 빨리 알고 싶어 합니다.

### 💡 해결
이용자 안내는 아래 순서로 정리하는 편이 훨씬 이해가 빠릅니다.

*   **행동 순서 먼저**: 로그인, 신청, 조회처럼 지금 해야 할 행동이 먼저 보여야 합니다.
*   **기능은 묶어서 설명**: 세부 기능을 길게 나열하기보다 "실전형 도구", "빠른 복기", "반복 연습"처럼 묶는 편이 더 잘 읽힙니다.
*   **운영자 시점 표현은 최소화**: 내부 승인 절차나 관리자만 아는 용어는 꼭 필요한 범위만 남겨야 합니다.

즉, 좋은 안내문은 구조 설명서가 아니라 **사용자 행동 가이드**에 더 가깝습니다.

## 3-5. 기능 버튼은 "설명 모달"보다 실제 작업 구역에 두는 편이 덜 헷갈린다
### 🚨 문제 상황
고급 기능 모달 안에 `문항별 상세 통계 TXT 다운로드` 같은 실행 버튼이 들어 있으면, 사용자는 설명을 읽다가 실제 작업 위치를 다시 찾아 이동해야 합니다. 특히 이 버튼은 채점과 복기 흐름의 일부인데, OMR 탭 바깥에 있으면 “기능은 여기서 설명하고 실행은 저기서 한다”는 식으로 동선이 갈라집니다.

### 💡 해결
이럴 때는 **설명은 설명대로 짧게 두고, 실행 버튼은 실제 작업이 일어나는 구역으로 옮기는 것**이 맞습니다.

*   **설명 모달 역할 축소**: 모달은 “어디서 무엇을 쓰는지”만 알려주는 가이드로 제한합니다.
*   **실행 위치 일치**: 채점, 상세 통계, TXT 다운로드, 정오표 일괄입력처럼 복기 흐름에 속한 버튼은 OMR 액션 영역에 모아 둡니다.
*   **문맥 유지**: 사용자가 OMR 탭에서 답안 작성 -> 정답 입력 -> 채점 -> 다운로드를 한 자리에서 이어서 끝낼 수 있어야 덜 헷갈립니다.

즉, 기능을 설명하는 화면과 기능을 실제로 누르는 위치가 다르면 사용성은 바로 떨어집니다. **설명은 짧게, 실행은 작업 구역 안에** 두는 원칙이 더 안전합니다.

## 3-6. 운영 기준이 관리자라면, 코드 기본값은 결국 "임시 안전망"이어야 한다
### 🚨 문제 상황
문구를 코드에 넣어 두면 화면은 바로 바뀌어서 편합니다. 하지만 운영자가 나중에 관리자 페이지에서 문구를 고치고 싶을 때, 실제 기준이 코드인지 Firebase 저장값인지 애매해질 수 있습니다. 특히 `config/siteTextConfig`가 비어 있으면 화면은 멀쩡해 보여도, 사실상 운영 기준이 코드 fallback에 묶여 있는 상태가 됩니다.

### 💡 해결
운영 기준을 관리자 페이지로 고정하려면 역할을 분명히 나눠야 합니다.

*   **코드 기본값**: 최초 seed 또는 비상 fallback 용도
*   **관리자 저장값**: 실제 운영에서 보이는 최종 기준값
*   **작업 절차**: 문구를 코드에서 바꿨다면 같은 턴 안에 관리자 저장값(`config/siteTextConfig`)도 같이 올려서 두 기준이 다시 하나로 합쳐지게 해야 합니다.

즉, 관리자 페이지를 최상위로 두겠다는 말은 단순히 “수정 UI가 있다”는 뜻이 아니라, **운영에서 실제 읽는 기준값을 Firebase 쪽으로 확정해 둔다**는 뜻입니다.

## 4. 자동 팝업 진입 회귀와 타이머 총시간 불일치
### 🚨 문제 상황
업데이트 과정에서 메인 페이지가 열리자마자 `window.open()`으로 팝업을 강제로 띄우고, 원본 화면을 곧바로 덮어쓰는 로직이 다시 들어왔습니다. 이 때문에 사용자는 도구 첫 화면 자체를 제대로 보지 못하고, 브라우저가 팝업을 막으면 버튼을 누를 기회도 없이 "팝업 모드 실행 중..." 화면에 갇히게 되었습니다.

타이머도 별도 문제가 있었습니다. 기본값이 `전체 75분 / 과목당 15분 / 쉬는 시간 1분`인데, 실제 필요 시간은 `15 x 5 + 1 x 4 = 79분`입니다. 즉 기본값만 써도 마지막 과목 전에 전체 타이머가 먼저 끝나 버리는 모순이 있었습니다.

### 💡 해결
*   **팝업은 수동 버튼만 허용**: 페이지 진입 즉시 자동 팝업을 여는 로직을 제거하고, 사용자가 `창 제한 해제` 버튼을 눌렀을 때만 팝업 모드가 열리도록 되돌렸습니다.
*   **총시간 최소값 보정**: `전체 제한 시간`이 `과목 시간 합 + 쉬는 시간 합`보다 작으면 자동으로 올려서, 불가능한 타이머 설정이 저장되지 않게 만들었습니다.
*   **관리자 기본값도 동일 규칙 적용**: 사용자 화면만 고치면 다시 원격 기본값에서 깨질 수 있으므로, 관리자 페이지 저장 시에도 같은 보정 로직을 넣었습니다.

### 🚨 문제 상황
운영 중에는 일반 사용자를 `update.html`로 보내 막아두고 싶지만, 개발자는 배포된 실제 웹에서 바로 수정 결과를 확인해야 할 때가 있습니다. 기존 구조는 점검이 켜지면 모두 같은 점검 페이지로 보내기만 해서, 운영 중 검증이 매우 불편했습니다.

## 6. 자동 팝업 대신 "수동 팝업 진입점 2개"가 더 안전한 이유
### 🚨 문제 상황
사용자들은 원래 팝업 모드로 많이 사용했기 때문에, 자동 팝업을 없애면 "예전처럼 좁게 못 쓴다"는 불편을 느낄 수 있습니다. 그렇다고 페이지가 열리자마자 자동으로 `window.open()`을 실행하면, 브라우저 팝업 차단 정책 때문에 다시 팝업이 막히고 원본 화면까지 덮어쓰는 회귀가 발생할 수 있습니다.

### 💡 해결
*   **자동 실행은 금지**: 팝업은 반드시 사용자의 클릭 이벤트에서만 열리게 둡니다. 이렇게 해야 브라우저가 정상적인 사용자 동작으로 판단해 차단 확률이 크게 줄어듭니다.
*   **진입점은 2개 제공**: 사용성이 떨어지지 않도록 상단 버튼과 좌측 탭 목록에 모두 `화면 더 줄이기` 버튼을 둡니다.
*   **실제 파일은 그대로 유지**: 팝업이라고 해서 다른 HTML을 쓰지 않고, 같은 `index.html`을 새 창으로 여는 방식으로 맞춥니다. 그러면 일반 페이지와 팝업 페이지의 기능 차이는 "창 크기를 더 좁게 조절할 수 있느냐"뿐입니다.
*   **팝업 내부에서는 버튼 숨김**: 이미 팝업 안에 들어간 뒤에는 다시 팝업을 띄울 이유가 없으므로, 상단 버튼과 좌측 탭 버튼을 숨겨 UI를 단순하게 유지합니다.
*   **경고창은 실패할 때만**: 열기 전마다 설명 alert를 띄우면 흐름이 끊기므로, 실제로 팝업이 차단된 경우에만 해제 안내를 보여주는 편이 더 자연스럽습니다.

### 💡 해결
*   **쿼리 파라미터 우회**: `index.html?dev=1`처럼 접속한 현재 URL에서만 점검 우회를 허용하도록 했습니다.
*   **일반 주소는 항상 차단 유지**: 개발자 링크로 한 번 들어갔다고 해서 이후 일반 주소까지 계속 열리면 안 되므로, `sessionStorage` 저장 방식은 제거했습니다.
*   **점검 페이지 직접 진입도 지원**: `update.html?...&dev=1`로 접속하면 즉시 `index.html?dev=1`로 넘겨 실제 화면을 확인할 수 있게 했습니다.

## 6. 관리자 미리보기에서 로컬 설정이 원격 기본값을 덮는 문제
### 🚨 문제 상황
관리자 페이지에서 타이머 기본값을 바꾸고 저장해도, 곧바로 `사이트 보기`로 들어갔을 때 변경이 안 보이는 문제가 있었습니다. 원인은 미리보기 브라우저가 예전에 저장해 둔 `localStorage` 값을 먼저 읽고 있었기 때문입니다.

### 💡 해결
*   **관리자 미리보기 모드 추가**: `preview=1` 쿼리로 들어온 현재 URL에서만 미리보기 동작을 활성화합니다.
*   **미리보기에서는 로컬값 무시**: 이 모드에서는 타이머, 문항 가이드, 레이아웃 비율의 로컬 저장값을 읽지 않고, 관리자 페이지에서 저장한 Firebase 기본값을 항상 우선 반영하도록 했습니다.
*   **관리자 링크도 같이 수정**: 관리자 상단 `사이트 보기` 링크가 자동으로 `index.html?dev=1&preview=1`를 실제 사용자와 같은 팝업 창 형태로 열리게 바꿨습니다.

## 9. 정답 입력 모드에서는 타이머가 멈춰야 하는 이유

*   **가이드 타이머도 자동 정지**: 가이드 표시는 `timerIsRunning` 조건을 보기 때문에, 전체 타이머를 멈추면 문항 가이드도 같이 멈추고 숨겨집니다.

## 11. “원격은 75인데 화면은 왜 79인가?”를 구분하는 방법
### 🚨 문제 상황
이번 케이스에서는 Firebase 원격 설정값 `config/timerDefaults`가 이미 `75`로 정상 저장돼 있었는데도, 실제 웹에서는 `79`가 계속 보였습니다. 즉 저장이 안 된 것이 아니라 **브라우저가 예전에 저장한 로컬 기본값이 계속 이기고 있던 상황**이었습니다.

### 💡 해결
*   **원격값과 로컬값을 먼저 분리해서 확인**: Firebase REST로 `config/timerDefaults.json`을 직접 확인해 원격값이 실제로 무엇인지 먼저 검증했습니다.
*   **오래된 기본값 패턴만 자동 폐기**: `localStorage.skct_timer_cfg`가 예전 기본값 `79/15/1`이고, 사용자가 직접 저장했다는 표식(`source: 'user'`)이 없으면 오래된 찌꺼기로 간주하고 삭제합니다.
*   **이후 사용자 저장은 명시적으로 표기**: 새로 저장할 때는 `source: 'user'`를 함께 넣어, 진짜 사용자 커스텀 값은 앞으로 자동 삭제되지 않게 했습니다.
*   **캐시 버스팅 추가**: `main.js` 로드 경로에 버전 쿼리를 붙여 브라우저가 오래된 JS를 계속 쓰는 문제도 줄였습니다.

## 12. 운영 문구도 설정 데이터로 빼야 배포가 덜 흔들린다
### 🚨 문제 상황
후원 안내 같은 운영 문구가 HTML 안에 박혀 있으면, 문장 하나만 바꿔도 다시 배포해야 합니다. 이런 작은 수정은 기능 변경이 아닌데도 캐시 문제나 회귀 위험을 같이 끌고 오기 쉽습니다.

### 💡 해결

*   **관리자 화면에서 직접 수정**: 운영자가 [admin.html](C:/dev/01_career/_assets/tools/skct_tool/admin.html)에서 저장하면, 사용자 페이지는 다음 로드 때 그 값을 읽어 즉시 반영합니다.
*   **여러 위치를 한 번에 맞춤**: 후원 모달과 쉬는 시간 안내처럼 같은 톤을 유지해야 하는 문구는 같은 설정 객체를 공유하게 하면 관리가 훨씬 쉬워집니다.
*   **기본값은 코드에 남긴다**: Firebase 설정이 비어 있어도 깨지지 않도록, 안전한 기본 문구를 코드에 같이 두는 편이 운영 안정성 면에서 가장 낫습니다.

## 13. “실전모드” 규칙은 타이머 상태와 분리해서 생각해야 한다
### 🚨 문제 상황
처음 구현에서는 실전모드라도 `timerIsRunning`이 `false`면 모든 문항을 다시 클릭할 수 있게 열려 있었습니다. 그래서 일시정지 상태나 시작 전 상태에서 이전 문항을 다시 바꿀 수 있었고, “현재 포커스 문항만 입력 가능”이라는 실전 규칙이 깨졌습니다.

### 💡 해결
*   **실전모드와 연습모드를 먼저 분리**: 문항 활성화 조건을 “타이머가 도느냐”가 아니라 “연습모드냐 아니냐” 기준으로 다시 정리했습니다.
*   **현재 문항만 활성**: 실전모드에서는 현재 `currentGlobalIndex`에 해당하는 문항만 클릭 가능하게 하고, 답을 입력하면 다음 문항으로 이동하면서 이전 문항은 즉시 다시 잠기게 했습니다.
*   **과목 잠금은 일시정지와 무관하게 유지**: 시간이 끝난 과목은 실전모드에서는 타이머를 멈춰도 계속 잠겨 있어야 하므로, 잠금 조건에서도 `timerIsRunning` 의존을 제거했습니다.
*   **스테이징에서 먼저 확인**: 이런 규칙성 변경은 사용자 체감이 크기 때문에, 운영본에 바로 넣지 않고 스테이징 사본에서 `실전모드 제한 -> 연습모드 자유입력` 흐름을 먼저 브라우저로 검증한 뒤 운영본에 옮기는 편이 안전합니다.

## 14. “기본값”은 로컬 저장과 서버 저장을 섞으면 쉽게 꼬인다
### 🚨 문제 상황
팝업 창의 크기와 위치를 예전 방식처럼 픽셀(`350x800`)과 `localStorage`로만 저장하면, 개발자가 서버에서 기본값을 바꿔도 이미 방문한 브라우저는 예전 로컬값을 계속 우선해서 써버립니다. 그러면 “서버 기본값을 바꿨는데 왜 사용자 화면이 안 바뀌지?”라는 문제가 반복됩니다.

### 💡 해결
*   **역할을 분리한다**: 서버는 “전체 공통 기본값”, 브라우저 로컬 저장은 “개인 임시 조절값”으로 역할을 나눠야 합니다.
*   **팝업 기본 크기/위치는 비율로 저장한다**: `widthRatio`, `heightRatio`, `leftRatio`, `topRatio`처럼 모니터 크기 대비 비율로 저장하면 14인치 노트북이든 다른 화면이든 같은 느낌으로 다시 계산할 수 있습니다.
*   **서버 저장은 관리자만 한다**: 사용자 페이지가 Firebase 설정을 직접 쓰기 시작하면 보안이 약해집니다. 그래서 팝업 페이지는 현재 값을 관리자 페이지로 `postMessage`로만 보내고, 실제 `set(ref(db, ...))`는 관리자 화면에서만 실행하는 구조가 안전합니다.
*   **드래그 결과도 결국 숫자로 환산해야 한다**: 분리선을 끌어놓은 최종 높이/폭을 현재 전체 높이/폭으로 나눠 `ratio` 값으로 바꾸면, 다음 화면에서도 거의 같은 비율로 다시 복원할 수 있습니다.
*   **팝업 전용 기본값은 팝업에서만 우선 적용한다**: 일반 전체 화면 레이아웃과 팝업 레이아웃은 목적이 다를 수 있으므로, 어떤 기본값이 어느 화면에 적용되는지 경계를 명확히 두는 편이 유지보수에 훨씬 유리합니다.

## 15. “실제 계산기처럼 보이게” 하려면 표시부 구조부터 바꿔야 한다
### 🚨 문제 상황
단순 입력창 하나에 현재 숫자만 보여주는 계산기는 사칙연산은 되더라도 실제 시험 화면 느낌이 잘 나지 않습니다. 특히 `=`를 누른 뒤 결과를 보고 다시 `+`, `-` 같은 연산을 이어가는 흐름이나, 바로 직전 계산들이 위로 쌓여 있는 느낌이 사라집니다.

### 💡 해결
*   **입력창 대신 히스토리 스택을 쓴다**: 현재 값 하나만 보여주는 `input` 대신, 이전 계산 결과 3개와 현재 계산 1개를 쌓는 표시부로 바꾸면 실제 계산기 느낌이 훨씬 강해집니다.
*   **계산 완료 시에만 이력을 확정한다**: `2 + 3 = 5`처럼 `=`가 눌린 시점의 계산만 히스토리로 밀어 넣고, 연산 기호만 누른 중간 계산은 현재 줄에서만 보여주면 읽기가 훨씬 자연스럽습니다.
*   **연산 결과를 다음 계산의 시작값으로 남긴다**: `=` 직후 결과를 지우지 않고 현재 값으로 유지하면, 바로 `+ 4 =`처럼 일반 계산기처럼 이어서 계산할 수 있습니다.
*   **디자인과 기능은 같이 맞춰야 한다**: 버튼 배열만 실제 계산기처럼 바꾸고 표시부가 예전 입력창 형태로 남아 있으면 어색합니다. 이런 경우는 버튼, 표시부, 정렬 방식까지 한 번에 묶어서 바꾸는 편이 훨씬 안정적입니다.

## 16. “실제 시험처럼 보이는 장식 버튼”도 별도 레이어로 두는 편이 낫다
### 🚨 문제 상황
우측 상단 채팅 버튼 자리나 우측 하단 물음표 버튼 자리는 기능보다 위치와 존재감이 중요한 요소입니다. 이런 버튼을 실제 도구 레이아웃 안에 억지로 끼워 넣으면 메모장이나 계산기 영역이 밀리면서 본래 기능 UI가 오히려 불안정해질 수 있습니다.

### 💡 해결
*   **별도 예약 공간을 먼저 뺀다**: 우측 버튼 열처럼 시험 화면에 항상 존재하는 장식 영역은 메모장/계산기 본문과 같은 비율 싸움에 넣지 말고, 먼저 고정 패딩처럼 예약해 두는 편이 더 안정적입니다.
*   **버튼 크기도 같은 값으로 묶는다**: 우측 열 폭을 바꾸면 채팅 버튼과 물음표 버튼 크기도 같은 기준으로 같이 바뀌게 해야 실제 화면 느낌이 유지됩니다.
*   **장식 패딩과 본문 비율을 분리한다**: 하단 여백이나 우측 버튼 열처럼 “절대적인 여백” 성격의 값은 먼저 제외하고, 남은 영역만 타이머/메모/계산기 비율로 나누는 구조가 직관적이고 덜 꼬입니다.

## 17. 팝업 실전 비율 조정은 “화면 기준 절대값”과 “남은 영역 비율”을 분리해야 덜 흔들린다
### 🚨 문제 상황
하단 여백을 단순히 현재 본문 높이 기준으로 계산하면, 창 높이를 줄이거나 다른 영역 비율을 조금만 건드려도 여백 크기가 같이 흔들립니다. 그러면 “모니터 대비 20% 여백”처럼 맞추고 싶어도 실제로는 계속 다른 값처럼 보이게 됩니다.

### 💡 해결
*   **여백은 화면 기준으로 계산한다**: 하단 여백처럼 시험 환경 흉내를 내기 위한 값은 `mainContent` 높이 대신 `screen.availHeight` 같은 화면 기준으로 계산해야, 팝업 내부 비율을 바꿔도 같은 체감 크기를 유지할 수 있습니다.
*   **남은 높이만 비율로 나눈다**: 하단 여백 픽셀을 먼저 확보한 뒤, 남은 공간만 타이머/메모장/계산기 비율이 나눠 갖게 하면 사용자가 기대하는 동작과 실제 결과가 맞아떨어집니다.
*   **최소 높이 제한은 시험 재현 때 방해가 될 수 있다**: 실전 화면과 최대한 맞추는 단계에서는 “보기 좋게” 남겨둔 최소 높이 제한이 오히려 장애물이 됩니다. 이런 경우는 드래그 제한과 CSS `min-height`를 같이 확인해야 합니다.

## 18. “비율 슬라이더가 안 줄어드는 것처럼 보일 때”는 실제 픽셀 하한을 같이 의심해야 한다
### 🚨 문제 상황
화면에서는 우측 버튼 열 비율을 계속 낮췄는데도 채팅 버튼과 물음표 버튼 크기가 거의 그대로 보일 수 있습니다. 이런 경우는 슬라이더 값이 안 바뀌는 게 아니라, 내부에서 실제 버튼 픽셀 크기에 최소값이 걸려 있을 가능성이 큽니다.

### 💡 해결
*   **비율 하한과 픽셀 하한을 같이 본다**: `sideButtonColumnRatio` 같은 비율 clamp만 낮추면 끝나는 게 아니라, `buttonPx = clampNumber(..., 44, 78)`처럼 실제 크기 하한도 같이 풀어야 체감 변화가 생깁니다.
*   **아이콘도 같이 줄어야 한다**: 버튼 박스만 작아지고 아이콘 글자 크기가 그대로면 또 넘칩니다. 그래서 버튼, 아이콘, 폰트, 간격을 같은 반응형 기준으로 묶는 편이 안전합니다.
*   **축소 UI는 `clamp()`가 단순하고 튼튼하다**: 계산기처럼 비율 조정으로 쉽게 작아지는 UI는 `font-size`, `gap`, `border-radius`를 `clamp()`로 묶어두면 큰 화면과 작은 화면을 한 번에 안정적으로 대응할 수 있습니다.

## 19. 운영 반영은 “정적 파일 배포”와 “Firebase 기본값 쓰기”를 함께 끝내야 진짜 완료다
### 🚨 문제 상황
운영 페이지는 `index.html`, `main.js`, `main.css` 같은 정적 파일만 바꿔서는 끝나지 않습니다. 화면 기본값이 Firebase `config/*`에 남아 있으면, 브라우저는 새 코드 위에 옛 기본값을 다시 덮어씌울 수 있습니다.

### 💡 해결
*   **코드 fallback과 서버 기본값을 같이 맞춘다**: 기본 수치를 바꿀 때는 JS 상수와 Firebase `config/popupLayout`, `config/layoutRatios`, `config/toolUiConfig`를 같은 값으로 맞춰야 합니다.
*   **GET으로 실제값을 다시 읽어 확인한다**: `PUT` 후 바로 `config.json`이나 개별 `.json` 경로를 다시 읽어보면, “코드는 바뀌었는데 서버 기본값은 예전 상태” 같은 문제를 빨리 잡을 수 있습니다.
*   **운영 기준 커밋도 문서에 남긴다**: 다음 작업자가 “지금 실제 서버 기준이 어느 커밋인지”를 헷갈리지 않게 운영 기준 커밋과 핵심 기본값을 문서에 같이 적어 두는 편이 안전합니다.

### 💡 해결
*   **실기능 패널과 분리한다**: 채팅 버튼 자리, 물음표 버튼 자리는 `position:absolute` 레이어로 올리는 편이 본문 레이아웃을 덜 흔듭니다.
*   **기능 없는 버튼임을 명확히 한다**: 클릭 시 안내 알림만 띄우면, “실제 시험 위치 구현용”이라는 목적을 해치지 않으면서도 오해를 줄일 수 있습니다.
*   **하단 여백과 같이 설계한다**: 하단 장식 버튼이 계산기 위를 덮지 않도록, 버튼 위치와 `bottom padding` 기본값을 같이 조절하게 만드는 편이 훨씬 실용적입니다.

## 17. “테스트 저장”과 “실서버 반영”은 아예 다른 버튼이어야 덜 헷갈린다
### 🚨 문제 상황
테스트 사이트에서 화면을 맞추는 과정과 실제 서버에 기본값을 반영하는 과정을 같은 저장 버튼으로 묶어버리면, 개발자는 “지금 저장이 테스트용인지 운영 반영인지”를 순간적으로 헷갈리기 쉽습니다. 특히 Firebase 경로가 `staging_hidden_v1/config`와 `config`로 나뉘어 있을 때, UI가 한 번만 잘못 눌려도 의도치 않게 운영값이 바뀔 수 있습니다.

### 💡 해결
*   **3단계로 끊는다**: `1. 테스트 기본값 편집 -> 2. 테스트 화면 열기 -> 3. 실제 서버 반영`처럼 버튼 단계 자체를 나누면 실수가 줄어듭니다.
*   **테스트 화면은 테스트 경로만 읽게 한다**: 테스트 사이트가 운영 `config`를 계속 읽고 있으면, 아무리 버튼을 나눠도 실제로는 테스트가 분리되지 않습니다. 그래서 테스트 페이지는 `staging_hidden_v1/config`만 읽도록 확실히 고정하는 편이 안전합니다.
*   **반영 버튼은 복사 동작으로 생각한다**: 마지막 단계는 “새 값을 직접 계산해서 저장”이 아니라, 이미 검증한 staging 값을 운영 `config/*`로 그대로 복사하는 버튼으로 설계해야 흐름이 단순해집니다.
*   **개발자 페이지에는 두 상태를 같이 보여준다**: `테스트 기본값`과 `실제 서버 기본값` 요약을 나란히 보여주면, 아직 운영 반영 전인지 한눈에 확인할 수 있습니다.

## 18. 편집 패널은 “항상 보이게”보다 “기본 접힘”이 실사용에 낫다
### 🚨 문제 상황
팝업 편집 패널을 화면 위에 항상 펼친 채로 올려두면, 정작 사용자가 확인해야 할 메모장, 계산기, 하단 버튼 위치를 가려버릴 수 있습니다. 기능은 있어도 실측 조정에는 오히려 방해가 됩니다.

### 💡 해결
*   **기본 상태를 접힘으로 둔다**: 편집 팝업이 열리면 우선 `접힘 상태`로 시작하게 하면, 사용자는 먼저 실제 화면을 보고 필요할 때만 패널을 펼칠 수 있습니다.
*   **헤더만 남긴다**: 패널 전체를 숨기지 말고 제목과 `펼치기` 버튼만 남기면 “편집 기능이 존재한다”는 사실은 유지하면서 화면 가림을 최소화할 수 있습니다.
*   **조정과 확인을 번갈아 하기 쉬워진다**: 펼치기 -> 값 조정 -> 저장 -> 다시 접기 흐름이 가능해야, 실제 시험 화면 위치를 맞출 때 훨씬 덜 답답합니다.

## 19. 드래그 계산식과 실제 flex 구조가 다르면 분리선이 튄다
### 🚨 문제 상황
가로 분리선을 드래그할 때 코드에서는 `타이머 / 메모 / 계산기` 3개 영역 높이를 기준으로 비율을 다시 계산했는데, 실제 CSS 레이아웃은 그 3개가 한 줄로 같은 비율 체계를 쓰지 않고 있었습니다. 이런 경우 사용자는 분리선을 살짝만 움직였는데도 아래쪽이 과장되게 늘어나거나, 드래그 감각이 이상하게 느껴질 수 있습니다.

### 💡 해결
*   **계산식과 레이아웃 구조를 맞춘다**: 코드가 `timer : utils : calc` 3분할을 전제로 한다면, CSS도 실제로 그 합 비율이 같은 flex 체계 안에서 동작하도록 맞춰야 합니다.
*   **중간 컨테이너도 비율 합으로 준다**: `tools-layout`처럼 내부에 다시 `utils / calc`를 품고 있는 컨테이너는 고정 `flex: 1`로 두면 안 되고, `flex: calc(utils + calc)`처럼 바깥 단계와 의미가 이어지게 두는 편이 안전합니다.
*   **작은 드래그로 검증한다**: 10px, 20px 같은 작은 이동에서도 높이가 자연스럽게 바뀌는지 확인해야 진짜로 정상화된 것입니다.

## 20. 계산기에서 “현재 값”만 보여주면 사용자는 식이 끊긴다고 느낀다
### 🚨 문제 상황
계산기가 `12 + 34`를 입력하는 중인데 화면에는 `34`만 보이고, 연산자는 따로 작은 표시로만 남아 있으면 사용자는 “내가 방금 어떤 식을 만들고 있었지?”를 다시 머릿속으로 조합해야 합니다. 특히 시험 도구처럼 빠르게 계산하는 화면에서는 이 불편이 더 크게 느껴집니다.

### 💡 해결
*   **현재 줄에 식 전체를 보여준다**: `storedValue`, `operator`, `current`를 따로 들고 있다가, `=` 전에는 현재 줄에 `12 + 34`처럼 합쳐서 그려주는 편이 훨씬 자연스럽습니다.
*   **표시 제한과 입력 제한을 분리한다**: 화면에 맞춘다고 표시 문자열을 일찍 잘라버리면 사용자는 식이 사라졌다고 느낍니다. 입력 길이 제한은 따로 두고, 표시 쪽은 줄바꿈이나 글씨 크기 축소로 버티는 편이 낫습니다.
*   **문제 전환 초기화와 같은 규칙은 유지한다**: 화면 표현을 바꿔도 “문제가 넘어가면 계산기는 초기화된다” 같은 기존 시험 규칙은 그대로 유지해야 회귀가 적습니다.

## 21. 실제 계산기처럼 보이려면 “아래 줄은 값, 위 줄은 과정” 구조가 더 맞다
### 🚨 문제 상황
수식 전체를 마지막 줄에 계속 붙여서 보여주면, 사용자가 기대하는 “마지막 줄은 현재 값” 감각과 어긋날 수 있습니다. 실제 계산기나 참고 사진처럼 보이려면, 마지막 줄은 현재 입력값 또는 결과값이어야 하고 계산 과정은 위쪽 줄에 쌓이는 편이 더 자연스럽습니다.

### 💡 해결
*   **진행 중인 식은 pending 줄로 분리한다**: `12 + 34`를 입력 중이면 위쪽 줄에 이 식을 보여주고, 마지막 줄은 `34`처럼 현재 값만 유지합니다.
*   **연속 계산 때는 직전 식을 바로 이력에 남긴다**: `12 + 34 -`처럼 다음 연산으로 넘어가는 순간, `12 + 34`는 완료된 계산 과정으로 위쪽에 남기고 마지막 줄은 `46`으로 유지하면 실제 계산기 느낌에 가깝습니다.
*   **결과는 마지막 줄에 남겨 다음 연산 시작점으로 쓴다**: `=`를 누른 뒤 마지막 줄의 결과값에서 `+`, `-` 같은 다음 연산을 바로 이어야 “이전 값에 이어서 계산”하는 흐름이 자연스럽습니다.

## 22. 연산자 누를 때마다 계산하면 우선순위를 절대 지킬 수 없다
### 🚨 문제 상황
`2 + 3 * 4`를 입력할 때 `+`나 `*`를 누르는 즉시 계산해버리면, 계산기는 입력 순서대로만 흘러가서 `20` 같은 잘못된 값이 나올 수 있습니다. 일반 계산기처럼 보이려면 식을 끝까지 보관했다가 `=`에서 한 번에 평가해야 합니다.

### 💡 해결
*   **식 토큰을 따로 쌓는다**: 표시값과 별개로 `['2', '+', '3', '*', '4']` 같은 토큰 배열을 유지하면, 연산자 입력이 “계산 실행”이 아니라 “식 누적”이 됩니다.
*   **평가는 `=`에서만 한다**: 사용자가 `=`를 누를 때만 토큰 배열을 평가해야 연산자 입력 중 중간 계산 부작용이 사라집니다.
*   **우선순위는 2단계로 처리한다**: 먼저 `*`, `/`를 줄여서 중간 값 배열을 만들고, 그 다음 `+`, `-`를 처리하면 기본 계산기 수준의 우선순위를 안정적으로 맞출 수 있습니다.

## 23. 장식 버튼이 겹치면 오버레이보다 “예약 열”이 더 안정적이다
### 🚨 문제 상황
채팅 버튼이나 물음표 버튼을 본문 위에 그냥 띄워두면, 메모장이나 계산기 위를 덮거나 화면 크기에 따라 버튼이 애매하게 걸칠 수 있습니다. 특히 “버튼 자체 크기도 같이 바꾸고 싶다”는 요구가 생기면 단순 absolute 배치만으로는 관리가 어려워집니다.

### 💡 해결
*   **우측 전용 열을 먼저 예약한다**: 본문 레이아웃에서 우측 열 폭을 먼저 빼고, 메모/계산기 본문은 남은 폭만 쓰게 하면 겹침이 훨씬 줄어듭니다.
*   **버튼 크기를 열 폭에서 같이 계산한다**: 버튼 폭/높이를 예약 열 폭에서 같이 파생시키면, 사용자가 열 폭을 조절할 때 버튼 크기도 함께 맞춰집니다.
*   **저장값도 별도 항목으로 둔다**: 이런 UI 폭은 `bottomPaddingRatio`와 성격이 다르므로, `sideButtonColumnRatio`처럼 별도 저장값으로 분리해야 개발자 페이지 저장/요약에서 누락되지 않습니다.

## 24. 숨김 고급 기능은 “진입 트리거”와 “실제 기능 화면”을 분리해야 관리가 쉽다
### 🚨 문제 상황
일반 사용자에게는 절대 보이면 안 되는 기능을 같은 설정 모달 안에 그냥 버튼으로 넣어두면, 나중에 문구를 고치거나 디자인을 바꾸는 순간 힌트가 새어 나가기 쉽습니다. 반대로 고급 기능 로직까지 메인 화면 파일 안에 한 덩어리로 넣으면, 인증과 다운로드 같은 예외 흐름을 고칠 때 일반 기능 회귀가 같이 생기기 쉽습니다.

### 💡 해결
*   **진입은 숨기고, 화면은 분리한다**: 메인 화면에서는 `통합 설정` 제목 연타 같은 숨김 트리거만 두고, 실제 고급 기능 UI는 `advanced-tools.html`처럼 별도 팝업 파일로 분리하는 편이 안전합니다.
*   **비밀번호 검증은 브리지로만 통신한다**: 팝업이 필요한 정보만 `window.opener.SKCTAdvancedBridge`로 받아가게 하면, 일반 화면 DOM에 고급 버튼을 추가하지 않아도 됩니다.
*   **버그 수정과 고급 기능은 구분한다**: 문항 시간 초기화나 계산기 기록처럼 “모두에게 필요한 정상 동작”은 일반 로직에 바로 반영하고, 통계 TXT 다운로드 같은 추가 기능만 고급 기능 팝업에 두는 식으로 경계를 나누면 요구사항이 덜 꼬입니다.

## 25. 숨김 고급 기능이 “별도 도구창”으로 남으면 사용자는 원하는 화면을 못 본다
### 🚨 문제 상황
숨김 진입 뒤에 `advanced-tools.html` 같은 별도 창을 띄워 TXT 다운로드나 비밀번호 목록을 보여주면, 개발자는 기능을 빨리 붙일 수 있습니다. 하지만 사용자가 원한 것이 “실제 시험 팝업 화면이 고급 버전으로 열린다”는 흐름이라면 이 구조는 체감이 완전히 다릅니다. 사용자는 결국 도구창만 보고, 실제 사용자 화면에 고급 기능이 어디에 붙는지 확인할 수 없습니다.

### 💡 해결
*   **인증 창은 게이트로만 쓴다**: 비밀번호를 받는 창은 인증 성공 여부만 판단하고, 성공한 뒤에는 자기 UI를 유지하지 않게 하는 편이 낫습니다.
*   **같은 창을 실제 앱으로 전환한다**: 인증 성공 시 `window.name`을 실제 팝업 이름으로 바꾸고 `index.html?advanced=1`처럼 고급 모드 플래그가 있는 실제 화면으로 이동시키면, 사용자가 기대한 “고급 버전 시험 화면”이 그대로 열립니다.
*   **고급 기능 UI는 실제 화면 안에 넣는다**: TXT 다운로드, 비밀번호 목록 같은 기능은 별도 도구창이 아니라 실제 사용자 화면의 설정 모달 안에 두는 편이 더 자연스럽고 검증도 쉽습니다.

## 26. 고급 사용자 기능과 개발자 관리 기능은 같은 화면에 두면 안 된다
### 🚨 문제 상황
고급 이용자용 화면에 비밀번호 목록 편집까지 같이 넣어두면, 권한 경계가 흐려집니다. 사용자는 “고급 기능을 쓰는 사람”인데, 실제로는 “고급 기능 입장 비밀번호를 바꾸는 관리자” 역할까지 보게 됩니다. 이건 기능 노출도 과하고, 운영 실수도 쉽게 만듭니다.

### 💡 해결
*   **사용자 화면엔 사용 기능만 둔다**: 고급 이용자 팝업에는 TXT 다운로드나 고급 동작 안내처럼 “쓰는 기능”만 남깁니다.
*   **관리 기능은 개발자 페이지로 옮긴다**: 비밀번호 목록처럼 운영자가 바꾸는 값은 `admin.html`에서만 보이게 하고, 실제 앱은 그 저장값만 읽게 만드는 편이 안전합니다.
*   **버튼 위치도 역할에 맞춘다**: 설명과 실행은 설정 모달 안에 숨기기보다, 좌측 사이드바처럼 눈에 띄는 별도 버튼으로 분리하는 편이 “고급 사용자가 실제로 쓰는 기능”이라는 성격에 더 맞습니다.

## 27. 모드별 기능 분리는 “숨김”만으로 끝내면 나중에 다시 튀어나온다
### 🚨 문제 상황
일반 모드에서 어떤 기능을 안 보이게 하고 싶다고 해서 HTML이나 CSS만 숨기면, 타이머 갱신 함수나 점수 계산 함수가 나중에 다시 그 요소를 열어버릴 수 있습니다. 예를 들어 문항별 가이드 타이머는 설정 섹션을 숨겨도, 표시 로직이 살아 있으면 타이머 시작 후 다시 나타납니다.

### 💡 해결
*   **표시 조건 자체에 모드 값을 넣는다**: `isAdvancedMode`처럼 런타임 플래그를 실제 표시 로직 조건에 넣으면, 숨김 회귀를 막을 수 있습니다.
*   **레이아웃 예약 공간도 같이 0으로 만든다**: 우측 버튼 열처럼 공간을 잡아먹는 기능은 요소만 숨기지 말고 CSS 변수까지 `0px`로 바꿔야 본문 폭이 정말 돌아옵니다.
*   **원격 설정이 덮어쓰는 지점도 같이 본다**: 제목이나 앱 이름을 Firebase에서 다시 넣는 구조라면, 로컬에서 붙인 `고급버전` 문구가 나중에 사라질 수 있습니다. 이때는 원격 설정 적용 시점에도 같은 조건을 다시 반영해야 합니다.

## 28. 관리자에서 숫자 설정으로 바꾸려면 “하드코드 + 기본값 + 저장값”을 같이 묶어야 한다
### 🚨 문제 상황
후원 목록 전환 속도처럼 처음엔 `setInterval(..., 4000)`으로 빠르게 넣은 값은 나중에 “관리자에서 바꾸고 싶다”는 요구가 나오기 쉽습니다. 이때 HTML 입력칸만 추가하면 끝날 것 같지만, 실제로는 기본값, 관리자 로드/저장, 사용자 런타임 사용처가 모두 맞아야 제대로 동작합니다.

### 💡 해결
*   **기본값 객체에 먼저 넣는다**: `DEFAULT_SUPPORT_CONFIG` 같은 공통 기본값에 `sponsorTickerSeconds`를 추가하면, 저장값이 없을 때도 동작이 안정됩니다.
*   **관리자 로드/저장을 같은 이름으로 묶는다**: `renderSupportForm()`에서 불러오고, 저장 시 `supportConfig`로 다시 쓰면 Firebase와 UI가 같은 필드를 바라보게 됩니다.
*   **하드코드를 없애고 저장값을 사용한다**: 실제 사용자 화면의 `4000` 같은 숫자를 `supportConfig.sponsorTickerSeconds` 기반으로 바꾸고, 기존 interval이 남지 않게 `clearInterval`까지 해줘야 속도 변경이 정확히 반영됩니다.

## 29. “기능이 안 된다” 제보는 먼저 로직 버그인지, 통계 표현 문제인지 분리해서 봐야 한다
### 🚨 문제 상황
사용자가 “건너뛴 문제는 오답풀이 때 답을 넣을 수 없다”고 느껴도, 실제 코드를 보면 입력 자체는 이미 가능할 수 있습니다. 이때 바로 클릭 로직을 뜯어고치면, 멀쩡한 기능을 건드리고 다른 회귀를 만들 수 있습니다.

### 💡 해결
*   **먼저 사실부터 확인한다**: `정말 클릭이 안 되는지`, 아니면 `통계/안내가 부족해서 그렇게 느끼는지`를 먼저 코드와 화면 흐름으로 분리해 확인합니다.
*   **통계 모델을 문항 단위로 분해한다**: `answered / skipped / unanswered / correct / wrong`처럼 문항 상태를 먼저 만들면, 요약 패널·상세 모달·TXT 다운로드가 모두 같은 기준을 재사용할 수 있습니다.
*   **설정값은 표시와 저장을 같이 묶는다**: `건너뜀 오답 처리`처럼 해석 기준이 바뀌는 옵션은 체크박스만 추가하지 말고, 로컬 저장값과 열린 통계 UI 재계산까지 한 번에 연결해야 사용자가 즉시 차이를 볼 수 있습니다.

## 30. 정적 웹에서 “비밀번호 숨기기”는 평문 제거와 서버 분리로 생각해야 한다
### 🚨 문제 상황
정적 HTML/JS만으로 돌아가는 웹은 브라우저가 결국 코드를 모두 내려받아 실행합니다. 그래서 “클라이언트 코드를 뜯어봐도 비밀번호를 절대 알 수 없게” 하거나 “소스를 완전히 못 보게” 만드는 것은 기술적으로 불가능합니다.

### 💡 해결
*   **평문 비밀번호를 내려보내지 않는다**: 최소한 `passwords: ['0208']`처럼 평문 배열을 클라이언트와 공개 설정값에 두는 구조는 없애야 합니다.
*   **해시 + salt로 바꾼다**: 정적 웹에서 현실적으로 할 수 있는 최선은 `salt + hash`만 내려보내고, 사용자가 입력한 값을 브라우저에서 해시 비교하는 방식입니다. 이러면 코드를 봐도 원문 비밀번호가 바로 드러나지는 않습니다.
*   **진짜 보호는 서버로 넘긴다**: 결제, 구독, 비밀번호 발급처럼 중요한 흐름은 결국 서버 또는 서버리스 함수에서 검증해야 합니다. 정적 웹은 접근 UX를 만들고, 실제 권한 판단은 서버가 하게 분리하는 것이 맞습니다.

## 31. “보안 보강”은 UX도 같이 정리해야 실제로 덜 헷갈린다
### 🚨 문제 상황
평문 비밀번호를 없애고 해시 검증으로 바꿔도, 사용자는 여전히 “지금 입력해도 되는지”, “이미 인증된 브라우저인지”, “왜 또 안 열리는지”를 모르면 불편합니다. 관리자도 어떤 구독이 살아 있는지 바로 안 보이면 실수하기 쉽습니다.

### 💡 해결
*   **사용자 화면에는 현재 상태를 먼저 보여준다**: `사용 가능한 고급 이용권 몇 건`, `이미 인증된 브라우저인지`, `잠시 재시도 대기 중인지` 같은 요약을 먼저 보여주면 실패 원인을 쉽게 이해할 수 있습니다.
*   **관리자 화면에는 숫자 요약을 붙인다**: `활성 / 일시중지 / 만료 / 곧 만료` 카드가 있으면 표를 다 읽지 않아도 현재 운영 상태를 빠르게 파악할 수 있습니다.
*   **저장 전에 막아야 할 실수는 미리 막는다**: 새 비밀번호 중복, 외부 ID 중복 같은 건 저장 후에야 알게 하지 말고 저장 단계에서 바로 실패시켜야, 잘못된 구독 데이터가 서버에 쌓이지 않습니다.

## 32. 작업 로그가 많아지면 “현재 기준 문서”를 따로 만들어야 한다
### 🚨 문제 상황
작업 로그가 계속 쌓이면, 나중에는 `무슨 파일을 먼저 봐야 하는지`, `현재 기준이 무엇인지`, `예전 문서와 최신 구조가 어떻게 다른지`를 한 번에 파악하기 어려워집니다.

### 💡 해결
*   **인덱스 문서를 만든다**: `README.md`처럼 문서 자체의 입구를 두면, 처음 보는 사람도 길을 잃지 않습니다.
*   **구조 문서와 변경 이력 문서를 분리한다**: 하나는 `지금 구조`, 다른 하나는 `어떻게 바뀌었는지`를 담당하게 해야 읽는 목적이 분명해집니다.
*   **오래된 문서는 폐기보다 연결한다**: 예전 운영가이드가 완전히 틀린 것이 아니라면, 앞부분에 `최신 기준은 여기` 링크를 달아 보조 문서로 남기는 것이 가장 안전합니다.

## 33. 결제 설계 문서는 “좋은 큰 그림”보다 “현재 운영 구조와의 접점”이 더 중요하다
### 🚨 문제 상황
토스 결제나 구독 설계 문서를 처음 쓰면 `billing-web / billing-api / DB` 같은 큰 구조는 그럴듯하게 잘 나옵니다. 하지만 실제 프로젝트에 붙일 때 가장 중요한 건 “지금 운영 중인 웹이 무엇을 읽고 있는가”, “관리자 화면은 어떤 필드로 움직이는가”, “수동 등록 데이터와 자동 결제 데이터를 어떻게 같은 표에서 다룰 것인가” 같은 접점입니다. 이게 빠지면 문서는 좋아 보여도 구현 직전에 다시 뜯어고쳐야 합니다.

### 💡 해결
*   **현재 스키마를 먼저 기준으로 둔다**: 새 시스템을 설계할 때도, 이미 운영 웹이 읽고 있는 `advancedFeatureConfig.subscriptions` 같은 실제 필드를 먼저 적어두면 문서가 현실과 붙습니다.
*   **원본 DB와 미러링 DB를 구분한다**: 결제 프로젝트의 원본 레코드는 더 풍부하게 가져가되, 기존 웹으로는 최소 필드만 미러링한다는 규칙을 적어두면 확장성과 호환성을 같이 챙길 수 있습니다.
*   **사용자 흐름에 활성화 단계를 넣는다**: 결제 성공 직후 바로 비밀번호를 받게 하기보다 `활성화 코드 -> 비밀번호 설정` 단계를 두면 운영과 고객지원이 훨씬 쉬워집니다.

## 34. 설계 다음에는 바로 “폴더 구조 + API 계약”으로 내려와야 구현이 덜 흔들린다
### 🚨 문제 상황
설계 문서가 좋아도 실제 구현을 시작하려 하면 다시 막히는 경우가 많습니다. 이유는 “어떤 폴더를 먼저 만들지”, “결제 승인 API와 활성화 API를 어떻게 나눌지”, “RTDB 미러링은 어느 계층이 책임질지”가 아직 코드 단위로 안 내려왔기 때문입니다.

### 💡 해결
*   **폴더 구조를 먼저 고정한다**: `app/api/payments`, `app/api/activations`, `lib/db/skct-rtdb-sync`처럼 경로를 먼저 정하면 책임 경계가 빨리 보입니다.
*   **API 이름을 먼저 고정한다**: `prepare`, `confirm`, `verify`, `consume`, `sync-to-skct`처럼 역할이 보이는 이름을 먼저 정하면 프론트/백엔드가 엇갈리지 않습니다.
*   **환경변수와 동기화 지점을 같이 적는다**: 결제 프로젝트는 시크릿 키만 중요한 게 아니라, 어느 RTDB 경로에 어떤 필드를 쓰는지도 같이 적어야 실제 연결이 덜 흔들립니다.

## 35. “카카오톡으로도 할 수 있나?”를 물으면 결제 채널과 운영 채널을 분리해서 생각해야 한다
### 🚨 문제 상황
결제 시스템을 설계하다 보면 “굳이 웹이 아니라 카카오톡 봇으로도 되지 않을까?”라는 생각이 자연스럽게 듭니다. 하지만 여기서 자주 생기는 실수는, 사용자가 자주 쓰는 채널과 권한을 최종 판단하는 시스템을 같은 것으로 착각하는 것입니다.

### 💡 해결
*   **카카오톡은 안내 채널로 본다**: 카카오톡은 결제 링크 전달, 활성화 링크 안내, 문의 응대에는 좋습니다.
*   **권한 판단은 서버가 한다**: 결제 승인, 활성화 코드 소비, 구독 상태 변경은 결국 서버/API가 진실 소스가 되어야 합니다.
*   **가장 안전한 구조는 “카카오톡 보조 + 웹/API 본체”다**: 사용자 입장에서는 카카오톡에서 시작해도 자연스럽고, 운영 입장에서는 추적과 재처리가 훨씬 쉬워집니다.

## 36. 실시간 접속자 기능은 “현재 몇 명인지”와 “개별 세션 목록”을 분리해야 가벼워진다
### 🚨 문제 상황
접속자 수를 보여주고 싶을 때 가장 쉽게 떠오르는 방식은 `active_visitors.json` 전체를 내려받아서 브라우저에서 숫자를 세는 것입니다. 처음엔 잘 되지만, 접속자가 늘면 사용자 화면과 관리자 화면이 모두 큰 JSON을 반복해서 읽게 되어 Firebase 읽기량이 커집니다.

### 💡 해결
*   **원시 데이터는 짧게만 유지한다**: 세션 heartbeat는 `active_visitors/{sessionId}`처럼 최소한으로 저장합니다.
*   **화면은 집계값만 읽는다**: 실제 화면은 `stats/live/current` 같은 숫자만 읽어도 충분합니다.
*   **집계는 별도 작업자가 한다**: 1분마다 현재 접속자 수를 계산하고, 5분 단위 rollup만 남기면 그래프도 만들 수 있고 DB 부담도 크게 줄어듭니다.

## 37. 구독 관리에서 “사용자 식별”과 “로그인 ID”를 섞으면 운영자가 꼭 헷갈린다
### 🚨 문제 상황
구독자 정보를 표 하나에 넣을 때 `홍길동 / 이메일 / 주문번호 / 비밀번호`를 한 칸씩 대충 넣기 시작하면, 나중에는 "이 값이 사람 이름인지", "로그인할 ID인지", "결제 추적용 값인지"가 섞여버립니다. 그러면 운영자가 계정을 추가할 때도 헷갈리고, 사용자에게 무엇을 전달해야 하는지도 매번 다시 설명해야 합니다.

### 💡 해결
*   **사람 확인용 값과 로그인용 값을 분리한다**: `사용자 식별`은 이름/이메일 같은 관리자 메모이고, `ID`는 사용자가 실제 로그인할 때 입력하는 값으로 역할을 분리해야 합니다.
*   **선택지는 자유입력보다 고정값이 안전하다**: `구독 종류`를 `3일 / 7일 / 2주 / 1달 / 1년`처럼 고정하면 입력 실수와 표기 흔들림이 줄어듭니다.
*   **만료 기준은 문장으로도 같이 적어둔다**: `만료일`만 두지 말고 `선택한 날짜 정오 12:00까지 사용`처럼 화면에 적어두면 운영자와 사용자 해석이 어긋나지 않습니다.
*   **비밀번호는 다시 못 보는 구조라는 점을 UI가 알려줘야 한다**: 해시 저장 자체보다 더 중요한 건, 저장 후 원문이 보이지 않는다는 사실을 운영자가 미리 이해하게 만드는 것입니다.

## 38. 정적 웹에서도 “관리자만 다시 볼 수 있는 비밀번호”는 별도 암호화 계층으로 풀어야 한다
### 🚨 문제 상황
정적 웹에서 운영자가 "비밀번호를 다시 확인하고 싶다"고 요구하면 가장 쉬운 방법은 평문 비밀번호를 그대로 RTDB에 넣는 것입니다. 하지만 이 프로젝트는 일반 사용자 브라우저도 같은 원격 설정을 읽는 구조라, 평문을 그대로 넣으면 사실상 누구나 개발자 도구로 볼 수 있게 됩니다.

### 💡 해결
*   **사용자 인증용 값과 관리자 확인용 값을 분리한다**: 로그인 검증은 기존처럼 `passwordSalt + passwordHash`로 하고, 관리자 확인은 별도 암호문(`passwordCipher`)으로 분리해야 합니다.
*   **관리자 로그인 비밀번호를 복호화 키로 쓴다**: 관리자 화면에 로그인할 때 입력한 비밀번호로만 암호문을 다시 풀 수 있게 하면, 공개 설정 안에 평문을 남기지 않고도 관리자 확인 기능을 만들 수 있습니다.
*   **예전 해시-only 계정은 복구 불가로 인정한다**: 해시는 원래 되돌릴 수 없으므로, 기존 계정은 억지로 "찾기"를 구현할 수 없습니다. 대신 한 번 새 비밀번호로 초기화한 뒤부터만 관리자 확인 기능을 제공하는 것이 정직하고 안전합니다.
*   **영구이용권은 날짜 대신 규칙으로 표현한다**: `만료일 비움 = 영구이용권`처럼 저장 규칙을 하나로 고정하면, UI와 인증 로직이 서로 다르게 해석하는 문제를 줄일 수 있습니다.

## 39. 집계 작업자를 아직 못 붙일 때는 “10초마다 전체 JSON fetch”를 먼저 없애는 것이 가장 큰 개선이다
### 🚨 문제 상황
정적 웹만 있는 상태에서 Cloud Function이나 cron을 아직 못 붙였다고 해서, 현재 접속자 수를 보여주는 화면이 `active_visitors.json` 전체를 10초마다 계속 새로 다운로드하면 읽기 비용이 금방 커집니다. 특히 관리자 페이지와 사용자 모달이 동시에 이 방식을 쓰면 효과가 더 나쁩니다.

### 💡 해결
*   **REST polling보다 실시간 구독이 낫다**: 같은 RTDB를 보더라도 `fetch()` 반복보다 `onValue()` 구독이 더 가볍고 자연스럽습니다.
*   **하트비트는 너무 자주 안 보내도 된다**: 현재 접속 확인이 목적이면 30초보다 60초 heartbeat가 더 현실적이고 비용도 줄어듭니다.
*   **완성형은 나중에 집계값 읽기 구조로 간다**: 지금 단계의 목표는 “반복 전체 다운로드 제거”, 다음 단계의 목표는 “raw presence 대신 집계값만 읽기”입니다.

## 40. “비밀글 게시판처럼 신청받기”는 공개 목록보다 안전하지만, 평문 저장을 없애야 의미가 있다
### 🚨 문제 상황
고급 이용권을 수동 승인으로 운영하려면 신청을 받아야 합니다. 가장 쉬운 방법은 공개 게시판이나 평문 폼 저장이지만, 이 프로젝트처럼 정적 웹이 Firebase를 직접 읽는 구조에서는 신청 내용과 비밀번호가 그대로 노출되기 쉽습니다.

### 💡 해결
*   **신청 번호 + 조회 비밀번호 구조로 분리한다**: 사용자는 전체 목록이 아니라 본인 신청만 다시 열어봐야 하므로, 공개 게시판보다 `신청번호 / 조회 비밀번호` 방식이 더 맞습니다.
*   **본문은 브라우저에서 먼저 암호화한다**: 신청 내용은 저장 전에 암호화하고, 사용자 비밀번호와 관리자 공개키를 각각 사용해 복호화 경로를 분리해야 합니다.
*   **운영자는 개인키로 전체를 열람한다**: 관리자 화면은 신청 목록 메타 정보만 먼저 보고, 실제 본문은 현재 브라우저에 있는 개인키로만 열어보게 해야 합니다.
*   **가입 안내는 순서형으로 보여줘야 한다**: 처음 보는 사람은 `로그인 -> 기능 -> 구독 등록 방법` 순서로 이해해야 혼란이 줄고, 후원형 수요 측정 단계에서도 이탈이 적습니다.

## 41. 운영자가 자주 바꾸는 문구는 “입력칸 추가”보다 “텍스트 카탈로그”가 오래 간다
### 🚨 문제 상황
사이트 문구를 바꿔 달라는 요청이 계속 들어오면 가장 쉬운 해결은 관리자 페이지에 입력칸을 하나씩 계속 추가하는 것입니다. 하지만 이 방식은 금방 한계에 부딪힙니다. 어떤 입력칸이 어느 화면을 바꾸는지 헷갈리고, 저장 전 실제 화면이 어떻게 바뀌는지도 바로 보기 어렵기 때문입니다.

### 💡 해결
*   **문구를 키로 먼저 관리한다**: `advancedGuide.title`, `helpModal.referenceBlockHtml`처럼 화면 문구를 키로 고정하면, HTML과 관리자 UI와 저장 경로가 같은 이름을 공유하게 됩니다.
*   **선택자와 같이 묶는다**: “이 키가 어느 DOM을 바꾸는가”를 카탈로그에 함께 적어두면, 나중에 운영자가 새 문구를 추가할 때도 구조가 흔들리지 않습니다.
*   **미리보기는 저장 전 검증 장치다**: iframe 미리보기와 클릭 선택을 붙이면 운영자가 코드를 몰라도 “이 문구가 여기 바뀌는구나”를 바로 확인할 수 있습니다.
*   **상태 메시지도 같이 묶어야 진짜 관리가 된다**: 버튼 라벨만 editable이고 성공/실패 문구가 하드코딩이면 결국 다시 코드를 열어야 하므로, 자주 노출되는 런타임 메시지도 같은 카탈로그로 끌어오는 편이 좋습니다.

즉, 텍스트 운영은 “폼 필드 모음”이 아니라 **문구 자산 관리 시스템**으로 보는 편이 장기적으로 훨씬 덜 꼬입니다.

## 42. “스크롤 목록”은 `overflow:auto`만 넣어서는 잘 안 된다
### 🚨 문제 상황
관리자 화면에서 좌측 목록을 따로 스크롤하고 싶을 때 `overflow-y: auto`만 주면 끝날 것처럼 보입니다. 하지만 실제로는 부모가 계속 커지면 목록도 같이 커져버려서, 스크롤바가 생기지 않거나 체감상 “독립 스크롤이 안 된다”는 문제가 자주 생깁니다.

### 💡 해결
*   **스크롤 컨테이너에는 높이 제한이 먼저 필요하다**: `max-height`, `height`, 또는 남는 공간을 먹는 flex 레이아웃이 없으면 브라우저는 굳이 내부 스크롤을 만들지 않습니다.
*   **flex 자식에는 `min-height: 0`이 중요하다**: 이 값이 없으면 자식이 내용 크기만큼 버티면서 부모 안에서 안 줄어들 수 있습니다.
*   **긴 우측 패널이 있으면 좌측은 sticky 패널로 보는 편이 낫다**: 편집기처럼 “목록 + 작업영역” 구조는 좌측을 `position: sticky`와 `max-height: calc(100vh - 여백)`로 잡아두면 훨씬 안정적입니다.
*   **모바일은 같은 규칙을 그대로 쓰지 않는다**: 좁은 화면에서는 열이 한 줄로 쌓이므로 sticky를 해제하고 일반 세로 흐름으로 돌리는 편이 자연스럽습니다.

즉, 스크롤 UX는 **overflow 속성 하나의 문제가 아니라, 높이 제약과 레이아웃 축소 조건을 같이 맞추는 문제**입니다.

## 43. 정적 사이트에서 “비밀번호를 공개 DB에 두고 브라우저가 검사”하면 보안이 아니라 착시다
### 🚨 문제 상황
정적 HTML/JS 사이트는 서버가 따로 없으니, 종종 Firebase 같은 공개 DB에 `passwordHash`, `adminHash`를 저장해 두고 브라우저가 직접 읽어 비교하는 방식으로 로그인 비슷한 것을 만들곤 합니다. 겉보기에는 “해시니까 안전해 보이는” 구조지만, 실제로는 누구나 같은 해시 데이터를 내려받아 오프라인 대입이나 클라이언트 우회를 시도할 수 있어 보안 경계가 되지 못합니다.

### 💡 해결
*   **관리자 인증은 Firebase Auth처럼 서버가 서명한 로그인 상태를 써야 한다**: 브라우저가 “비밀번호 정답표”를 직접 들고 검사하면 안 되고, 인증 결과만 받아야 합니다.
*   **고급 기능 해금은 ‘공개 검증키 + 비공개 서명키’ 구조가 맞다**: 정적 사이트에서는 서버가 코드를 숨겨주지 못하므로, 최소한 “아무나 임의로 해금 상태를 만들 수는 없게” 서명된 라이선스를 검증하는 구조가 필요합니다.
*   **공개 DB에는 공개돼도 되는 것만 남겨야 한다**: 신청 공개키, 라이선스 검증 공개키처럼 “읽혀도 괜찮은 값”만 두고, 개인키나 계정 해시 같은 값은 절대 두면 안 됩니다.
*   **예전 민감 데이터는 코드만 안 쓰게 두면 끝이 아니다**: `config/adminHash`, `config/advancedFeatureConfig`처럼 과거 민감 경로는 실제 DB에서도 지워야 재노출을 막을 수 있습니다.

즉, 정적 웹에서 중요한 기준은 **“비밀을 브라우저에 숨길 수 있느냐”가 아니라, “비밀 없이도 위조를 막는 구조인가”** 입니다.

## 44. Firebase RTDB 규칙은 “부모 읽기 허용” 하나로 자식 비밀이 함께 열릴 수 있다
### 🚨 문제 상황
Realtime Database에서 `config/.read = true`를 열어 둔 뒤 `config/adminHash/.read = false`처럼 자식만 막아 두면 안전할 것처럼 보일 수 있습니다. 하지만 실제로는 클라이언트가 `config` 부모 전체를 읽는 순간, 그 안의 민감 자식까지 같이 노출될 수 있어 보안 의도가 무너집니다.

### 💡 해결
*   **민감 자식이 있으면 부모 루트를 통째로 공개하지 않는다**: `config/appName`, `config/timerDefaults`처럼 공개가 필요한 키만 각각 `.read = true`로 허용해야 합니다.
*   **프런트 조회 코드도 같은 단위로 잘라 읽어야 한다**: 규칙만 세분화하고 브라우저 코드가 계속 `config` 전체를 읽으면 바로 권한 오류가 납니다. 규칙 변경과 조회 코드 변경은 세트입니다.
*   **관리자 화면은 인증 후 루트를 읽고, 사용자 화면은 공개 키만 읽게 나눈다**: 같은 DB라도 “운영자용 전체 설정”과 “사용자용 공개 설정”을 같은 방식으로 읽으면 보안과 유지보수 둘 다 꼬입니다.

즉, RTDB 보안에서 중요한 기준은 **“무엇을 막을까”보다 먼저 “어떤 부모 경로를 통째로 열고 있나”를 보는 것**입니다.

## 45. 신청 조회용 번호와 실제 로그인 식별자는 처음부터 분리해야 덜 꼬인다
### 🚨 문제 상황
수동 구독 신청 흐름을 만들다 보면 `신청번호`, `신청 이메일`, `로그인 ID`가 한 번에 등장합니다. 이 셋을 한 입력칸에서 모두 받아 버리면 처음에는 편해 보여도, 나중에 “무엇으로 조회하고 무엇으로 로그인하는가”가 뒤섞여 안내 문구와 코드가 서로 어긋나기 쉽습니다.

### 💡 해결
*   **신청 상태 조회는 한 경로로 고정한다**: 이 프로젝트처럼 신청자 본인이 자기 신청만 다시 확인하면 되는 구조라면 `신청 이메일 + 조회 비밀번호`처럼 조회 경로를 하나로 고정하는 편이 덜 헷갈립니다.
*   **로그인은 계정 식별자만 받는다**: 로그인 입력칸은 `로그인 ID 또는 이메일`처럼 실제 계정 식별자만 받게 두고, `신청번호` 같은 내부 추적 키는 아예 받지 않는 편이 안전합니다.
*   **수기 등록 계정은 별도 계정 경로로 유지한다**: 관리자가 직접 넣은 `ID-only` 계정까지 신청 조회 규칙에 억지로 맞추면 오히려 기존 계정을 망가뜨릴 수 있으므로, 이런 계정은 계정 로그인 경로로 그대로 살려 두는 것이 맞습니다.
*   **문구와 로직을 같이 바꿔야 한다**: 코드만 바꾸고 화면 문구를 안 바꾸면 사용자는 계속 `신청번호`를 넣어 보게 됩니다. 입력 placeholder, 설명문, 오류 메시지까지 같은 정책을 가리키게 맞춰야 혼란이 줄어듭니다.

즉, 식별자 설계에서는 **“추적용 번호”와 “사용자 로그인용 계정값”을 섞지 않는 것**이 가장 중요합니다.
