# SKCT Tool 개발 및 문제 해결 기록 (Learning Notes)
작성일시: 2026-04-04 03:45:00 KST

본 문서는 SKCT 연습 도구의 고도화 작업간 발생했던 핵심 기술적 문제들과, 이를 어떤 방식으로 해결했는지 초보자 및 향후 유지보수 담당자가 쉽게 이해할 수 있도록 기록한 문서입니다.

## 1. 실시간 접속자 무한 증식(좀비 노드) 버그와 해결법
### 🚨 문제 상황
최초에는 방문자가 들어오면 Firebase RTDB에 `push()`로 접속 상태(`true`)를 기록하고, 나갈 때 `onDisconnect()`를 이용해 상태를 삭제하는 가장 기본적인 방식을 사용했습니다. 하지만 방문자가 1시간 동안 F5(새로고침)를 10번 누르면, 이전 접속 기록이 미처 지워지기 전에 새로운 기록이 쌓여 **1명의 유저가 10명으로 뻥튀기(좀비 노드)**되는 문제가 발생했습니다.

### 💡 1차 해결 (Session ID 도입)
브라우저 탭마다 발급되는 `sessionStorage`를 이용해 고유한 ID(`sess_xxxx`)를 부여했습니다. 유저가 새로고침을 하더라도 새 데이터를 쌓는 대신 본인의 ID 자리에 덮어쓰기(`set`)를 하게 만들어 증식 버그를 1차적으로 막았습니다.

### 🚀 최종 해결 (Heartbeat TTL 시스템 도입)
Session ID 도입에도 불구하고 스마트폰에서 브라우저를 훅 꺼버리거나, 인터넷이 끊긴 채로 잠들면 `onDisconnect` 신호가 증발하여 서버에 가짜 데이터가 영원히 남는 문제가 여전히 존재했습니다.
이를 완전히 끊어내기 위해 대형 게임 서버 등에서 사용하는 **'하트비트 생존 추적(Heartbeat TTL)'** 기법을 적용했습니다.

*   **동작 방식**: 클라이언트는 30초마다 `Date.now()`(현재 시각)를 서버에 갱신합니다(심장 박동).
*   **검증 방식**: 화면에 "현재 접속자"를 렌더링할 때, 서버 데이터를 가져와서 **최근 90초 이내에 시각이 갱신된 사람만 카운트**합니다.
*   **결과**: 강제 종료, 캐시 지연, 와이파이 끊김 등 그 어떤 변수에도 90초 뒤면 통계에서 무조건 깨끗하게 사라지는 완벽한 자동 정화 시스템이 완성되었습니다.

## 2. 일일 방문자 중복 집계 차단 및 KST 타임존 문제
### 🚨 문제 상황
처음엔 `new Date().toISOString()` 함수를 사용해 날짜를 계산했습니다. 이 함수는 영국 시간(UTC)을 기준이므로, 한국 시간(KST)으로 오전 9시가 되어야 비로소 다음 날로 날짜가 바뀌는 치명적인 집계 오류가 있었습니다.

### 💡 해결 (Local Time 기반 원자적 연산)
*   **타임존 해결**: `d.getFullYear()`, `d.getMonth()`, `d.getDate()` 등 브라우저의 로컬 기기 시간을 그대로 뽑아내어 조합하는 방식으로 전환하여 한국 시간 자정에 정확히 날짜가 넘어가도록 수정했습니다.
*   **중복 차단**: 브라우저의 `localStorage` 공간에 방문 날짜를 도장 찍어두고, 같은 날 재방문 시에는 카운트를 올리지 않는 `if (lastVisitDate === today) return;` 방어벽을 쳤습니다.
*   **안전한 카운트업**: 여러 명이 동시에 방문하여 카운트가 씹히는 충돌을 막기 위해 Firebase의 `runTransaction()` 메서드를 사용했습니다. 이 함수는 DB 값을 100% 안전하고 원자적으로 1씩 올려주는 역할을 합니다.

## 3. 후원자 롤링 애니메이션 UI 잘림 현상 방어
### 🚨 문제 상황
투네이션 후원자 내역을 팝업 모달 하단에서 위로 스르륵 올라오는 애니메이션 티커로 구현했습니다. 그런데 후원 메시지가 너무 길거나 창이 좁아지면, 텍스트가 아래로 뚝 떨어지면서 화면 밖으로 잘리거나 날짜가 가려지는 에러가 발생했습니다.

### 💡 해결 (CSS Flexbox & Line-Clamp)
*   **레이아웃 4배 확장**: 높이를 20px에서 80px로 키워 충분한 여유 공간을 확보했습니다.
*   **구조 분리**: 상단 영역(이름+금액, 빈 공간, 날짜)과 하단 영역(메시지 본문)을 분리 배치하여 서로 침범하지 않게 했습니다.
*   **말줄임표(...) 최후 방어선**: 하단 메시지에 CSS `-webkit-line-clamp: 3` 속성을 적용했습니다. 이 마법의 속성은 글이 아무리 길어도 최대 3줄까지만 표시하고, 그 이후는 자동으로 `...`으로 잘라주는 브라우저 공식 지원 기능입니다. 이로써 어떤 환경에서도 디자인이 깨지지 않게 되었습니다.
