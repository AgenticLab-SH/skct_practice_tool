# 고급 구독 관리자 운영 가이드
작성일시: 2026-04-08 11:36:14 KST

이 문서는 현재 `admin.html`의 고급 구독 관리 기능을 운영자가 헷갈리지 않고 다루기 위한 기준 문서입니다.  
특히 `운영 반영 여부`, `용어 뜻`, `Firebase 저장`의 의미, `계정 추가 절차`를 쉽게 설명합니다.

## 1. 현재 상태

- 구독 관리자 UI와 `ID + 비밀번호` 인증 코드 변경은 **`main` 브랜치에 반영 완료**되었습니다.
- 운영 반영 기준 커밋은 `bcd2f20`입니다.
- GitHub 원격 `origin/main` push까지 완료했습니다.
- 아직 운영 Firebase RTDB에 새 구독 데이터를 저장하지 않았습니다.
- 따라서 현재 상태는 아래처럼 구분해야 합니다.
  - **운영 코드 반영**: 완료
  - **운영 구독 데이터 변경**: 아직 안 함

정리:
- `코드 수정 완료` = 예
- `로컬 화면 검증 완료` = 예
- `main push 완료` = 예
- `운영 Firebase 구독 데이터 저장 완료` = 아니오

## 2. 이 화면에서 쓰는 말의 정확한 뜻

### 상태
- 이 계정을 지금 사용할 수 있는지 나타냅니다.
- `사용중`: 로그인 가능
- `일시중지`: 비밀번호가 맞아도 로그인 불가
- `만료`: 비밀번호가 맞아도 로그인 불가

### 구독 종류
- 이용 기간 종류를 고르는 칸입니다.
- 현재 선택지는 아래 5개입니다.
  - `3일 이용권`
  - `7일 이용권`
  - `2주 이용권`
  - `1달 이용권`
  - `1년 이용권`
  - `영구이용권`

### 사용자 식별
- 운영자가 "이 계정이 누구 것인지" 알아보기 위한 칸입니다.
- 예:
  - `홍길동`
  - `hong@example.com`
  - `홍길동 / hong@example.com`
- 로그인할 때 사용자가 입력하는 값이 아닙니다.

### ID
- 사용자가 실제 로그인할 때 입력하는 계정 이름입니다.
- 계정마다 **중복 없이 하나씩** 있어야 합니다.
- 예:
  - `skct_hong`
  - `sb_user01`

### 비밀번호
- 사용자가 로그인할 때 ID와 함께 입력하는 값입니다.
- 로그인 검증용 해시와, 관리자 로그인 비밀번호로만 다시 풀 수 있는 암호문을 함께 저장합니다.
- 그래서 일반 사용자 쪽 공개 설정에는 평문이 그대로 남지 않습니다.
- 관리자 화면에서는 아래 두 가지가 가능합니다.
  - `확인`: 현재 저장된 비밀번호 다시 보기
  - `초기화`: 새 비밀번호를 생성해 바로 교체 준비
- 단, **예전 해시-only 방식으로 저장된 오래된 계정은 기존 비밀번호를 복구할 수 없습니다.**
- 그런 계정은 한 번 새 비밀번호로 초기화해서 저장한 뒤부터 확인 가능합니다.

### 만료일
- 이 계정이 언제까지 유효한지 정하는 날짜입니다.
- 현재 기준은 **선택한 날짜의 정오 12:00(KST)까지**입니다.
- 예:
  - 만료일이 `2026-04-30`이면 `2026-04-30 12:00:00 KST`까지 로그인 허용
- 만료일을 비워두면 자동으로 `영구이용권`으로 처리됩니다.
- 반대로 `구독 종류`를 `영구이용권`으로 바꾸면 만료일은 자동으로 비워집니다.

### 메모
- 운영 메모입니다.
- 예:
  - `입금 확인 완료`
  - `1회 연장 약속`
  - `문의 후 수동 발급`

### 삭제
- 해당 행을 목록에서 제거합니다.
- 저장 버튼까지 눌러야 실제 저장 구조에도 반영됩니다.

## 3. 예전의 `외부 ID`는 왜 없어졌는가

- 예전 `외부 ID`는 주문번호나 외부 결제 식별값 같은 보조 추적용 개념이었습니다.
- 그런데 현재 운영 흐름에서는 사용자가 이해해야 할 핵심 값이 아닙니다.
- 운영자가 실제로 필요한 것은 아래 두 가지입니다.
  - 누구 계정인가 (`사용자 식별`)
  - 사용자가 무엇을 입력해 로그인하는가 (`ID + 비밀번호`)
- 그래서 관리자 화면에서는 `외부 ID`를 제거하고 `ID` 중심으로 단순화했습니다.

## 4. `Firebase 저장`은 정확히 무엇인가

이 프로젝트는 별도 백엔드 서버 없이 Firebase Realtime Database(RTDB)를 설정 저장소처럼 사용합니다.

관리자 페이지에서 `고급 구독 저장` 버튼을 누르면, 브라우저가 아래 경로에 데이터를 직접 저장합니다.

- RTDB 경로: `config/advancedFeatureConfig`

실제 코드 기준:
- [admin.html](/C:/dev/01_career/_assets/tools/skct_tool/admin.html#L1775)
  `collectAdvancedFeatureConfigFromTable()`로 현재 표 내용을 읽습니다.
- [admin.html](/C:/dev/01_career/_assets/tools/skct_tool/admin.html#L1776)
  `set(ref(db, 'config/advancedFeatureConfig'), advancedFeatureConfig)`로 RTDB에 씁니다.
- [main.js](/C:/dev/01_career/_assets/tools/skct_tool/main.js#L2614)
  사용자 화면은 이 원격 설정을 읽어 인증 기준으로 사용합니다.

즉, 여기서 말한 `Firebase 저장`은:
- 단순히 로컬 브라우저에만 저장하는 것이 아니라
- **운영 웹이 실제로 읽는 원격 구독 설정을 RTDB에 기록하는 것**을 뜻합니다.

이번 작업에서는 코드와 화면은 운영에 반영했지만,  
`고급 구독 저장` 버튼은 아직 실제 운영 데이터 대상으로 누르지 않았습니다.

즉:
- 운영 사이트 코드는 이미 새 구조를 읽을 수 있습니다.
- 하지만 실제 구독자 목록을 새로 추가/수정하는 저장은 아직 별도 실행하지 않았습니다.

## 5. 새 유료 구독자 추가 절차

예를 들어 사용자가 비밀번호로 `sb2839`를 쓰고 싶다고 하면, 아래 순서로 넣으면 됩니다.

1. 관리자 페이지의 `고급 구독 관리` 표에서 `+ 구독 추가`를 누릅니다.
2. 각 칸을 아래처럼 채웁니다.
   - 상태: `사용중`
   - 구독 종류: 기간에 맞는 항목 선택
   - 사용자 식별: 이름/이메일 입력
   - ID: 사용자가 로그인할 계정 ID 입력
   - 비밀번호: `sb2839`
   - 만료일: 원하는 만료 날짜 선택
   - 메모: 필요 시 입력
3. `고급 구독 저장` 버튼을 누릅니다.
4. 저장 후에는 사용자에게 아래 두 개를 전달합니다.
   - ID
   - 비밀번호(`sb2839`)

중요:
- 비밀번호만 정해져 있다고 바로 계정이 완성되는 것은 아닙니다.
- **ID도 반드시 같이 정해야** 사용자가 로그인할 수 있습니다.

## 6. 저장 시 자동으로 막는 실수

현재 저장 로직은 아래 경우를 자동으로 막습니다.

- `사용자 식별`이 비어 있음
- `ID`가 비어 있음
- 같은 `ID`가 여러 행에 중복됨
- 새 계정인데 비밀번호가 비어 있음

반대로 아래는 허용합니다.

- 서로 다른 계정이 같은 비밀번호를 쓰는 것
  - 보안상 권장되지는 않지만, 현재 운영 흐름상 사용자가 원하면 막지는 않습니다.

## 7. 인증은 실제로 어떻게 동작하는가

- 관리자 화면은 비밀번호 원문을 그대로 저장하지 않습니다.
- 저장 시 `passwordSalt + passwordHash`를 만들고,
- 동시에 관리자 로그인 비밀번호로만 다시 열 수 있는 `passwordCipher` 계열 암호문도 함께 저장합니다.
- 사용자 로그인 시에는 입력한 `ID + 비밀번호`를 현재 구독 목록과 비교해 일치 여부를 확인합니다.

즉 구조는 아래처럼 나뉩니다.
- 사용자 인증용: `passwordSalt + passwordHash`
- 관리자 확인용: `passwordCipher + passwordCipherIv + passwordCipherSalt`

중요:
- 기존에 이미 저장돼 있던 예전 해시-only 계정은 `확인` 버튼으로 원래 비밀번호를 보여줄 수 없습니다.
- 그런 계정은 `초기화`나 새 비밀번호 입력 후 저장하면, 그 다음부터는 관리자 확인이 가능해집니다.

핵심 코드:
- [main.js](/C:/dev/01_career/_assets/tools/skct_tool/main.js#L737)
  `validateAdvancedCredentialsDetailed(loginId, password)`
- [index.html](/C:/dev/01_career/_assets/tools/skct_tool/index.html#L461)
  사용자용 `고급 ID 입력` 칸
- [advanced-tools.html](/C:/dev/01_career/_assets/tools/skct_tool/advanced-tools.html#L125)
  보조 인증 창의 `ID` 입력 칸

## 8. 운영 코드 반영 후 남은 일

코드 배포는 끝났고, 실제 운영 데이터 기준으로는 아래가 남아 있습니다.

1. 관리자 실제 로그인 후 새 UI가 정상 표시되는지 확인
2. 테스트용 계정 1건을 넣어 저장 버튼 동작 확인
3. RTDB `config/advancedFeatureConfig`에 새 구조가 정상 저장되는지 확인
4. 일반 사용자 화면에서 `ID + 비밀번호` 인증 확인
5. 필요하면 실제 유료 구독자 계정 추가

## 9. 한 줄 결론

- 지금은 **운영 코드 반영 완료 / 운영 구독 데이터 저장은 아직 미실행** 상태입니다.
- `Firebase 저장`은 **운영 웹이 읽는 RTDB 구독 설정을 실제로 쓰는 것**입니다.
- 이제 관리자 화면은 `외부 ID`가 아니라 **사용자 식별 + ID + 비밀번호** 기준으로 관리합니다.
