# SKCT Tool 보안 렌더링 체크리스트
작성일시: 2026-04-09 02:42:28 KST

이 문서는 새 HTML 경로, 새 저장 경로, 새 문구 렌더링 경로를 추가할 때 같은 보안 실수를 반복하지 않기 위한 최소 체크 기준입니다.

## 1. 새 HTML 경로를 추가할 때
- 메인 연습 화면과 꼭 섞어야 하는지 먼저 확인합니다.
- 별도 인증이나 별도 데이터 경계가 필요하면 별도 경로로 분리합니다.
- 콘솔 오류만 보지 말고, 비인증 상태에서 민감 정보가 보이지 않는지도 같이 확인합니다.

## 2. 브라우저 저장소를 쓸 때
- 장기 보관이 꼭 필요한 값인지 먼저 구분합니다.
- 세션이 끝나면 없어져도 되는 값은 `sessionStorage`를 우선 검토합니다.
- 비밀번호, 실패 횟수, 중복 제출 판정 정보처럼 민감하거나 오해를 부를 값은 `localStorage`에 오래 남기지 않습니다.

## 3. RTDB 새 경로를 만들 때
- 공개 경로인지, 사용자별 비공개 경로인지 먼저 확정합니다.
- 사용자별 데이터면 `auth.uid === $uid` 형태로 읽기, 쓰기 경계를 먼저 만듭니다.
- 로컬 코드만 추가하고 rules 배포를 빼먹지 않았는지 마지막에 다시 확인합니다.

## 4. HTML 문구를 렌더링할 때
- `supportConfig`, `siteTextConfig`처럼 운영자가 입력하는 HTML은 sanitize 경로를 거쳐야 합니다.
- 허용 태그와 속성을 넓혀야 하면, 왜 필요한지와 영향 범위를 먼저 문서화합니다.
- 새 렌더링 경로를 추가할 때는 `스크립트 실행 불가`, `이벤트 속성 차단`, `링크 처리 방식`을 같이 확인합니다.

## 5. 운영 기본값과 코드 기본값이 같이 있을 때
- 운영 DB의 값이 정말 사용자 커스터마이징인지, 예전 코드 기본값 잔재인지 먼저 구분합니다.
- 승인 없이 운영 값을 덮어쓰지 말고, 가능하면 코드에서만 레거시 기본값 보정 계층을 둡니다.

## 6. 배포 전 최소 확인
- 비인증 사용자가 민감 경로를 읽지 못하는지
- 정상 사용자가 신청, 조회, 고급 진입, 문구 저장 흐름을 그대로 쓸 수 있는지
- 새 HTML 경로와 새 저장 경로가 기존 기능을 가리지 않는지
