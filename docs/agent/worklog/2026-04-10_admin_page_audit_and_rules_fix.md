# 관리자 페이지 점검 및 저장 권한 복구
작성일시: 2026-04-10 10:38:13 +09:00

## 요청 요약

- 관리자 페이지 기능을 실제 로그인 기준으로 점검하고, 개선할 부분을 손본다.
- 공지 저장 실패 원인을 찾아 해결한다.
- 공개 관리자 경로는 유지 차단하고, 로컬 관리자 도구만 안전하게 개선한다.

## 실제 원인

- 관리자 로그인 자체는 살아 있었지만, 운영 RTDB rules에서 `config` 루트 쓰기 권한이 빠져 있었습니다.
- `config/$key/.write = auth != null`만으로는 `config/notice_sidebar`, `config/notice_help`처럼 명시적으로 선언된 자식 경로 저장이 풀리지 않았습니다.
- 그래서 `공지 저장`, `기본 설정 저장`, `후원/문구 저장` 같은 `config/*` 쓰기 작업이 `permission-denied`로 떨어졌습니다.

## 이번 작업

### 1. 운영 RTDB rules 복구

- `database.rules.json`의 `config` 블록에 `.write = auth != null`를 추가했습니다.
- 운영 Firebase RTDB에 rules를 실제 배포했습니다.
- 이 변경으로 로그인된 관리자 기준의 `config/*` 저장이 다시 가능해졌습니다.

### 2. 관리자 상단 상태 배너 추가

- `admin.html`, `admin.css`에 `세션 상태 + 마지막 저장 결과` 배너를 추가했습니다.
- `auth` 토큰 확인, RTDB 재연결, 저장 성공/실패가 상단에 바로 보이게 했습니다.

### 3. 관리자 쓰기 작업 상태 전파 보강

- 공지 저장, 기본값 저장, 고급 구독 저장, 전체 저장뿐 아니라 아래 동작도 상단 상태줄과 실패 토스트를 함께 쓰도록 맞췄습니다.
- 테스트 기본값 저장
- 테스트 기본값 실제 서버 반영
- 관리자 답글 등록
- 코드 기본값을 관리자 기준값으로 저장
- 백업 복원
- 신청 승인 / 반려 / 차단 / 삭제
- 신청 암호화 키 생성
- 라이선스 서명키 생성

### 4. 신청 처리 실패 흐름 보정

- 신청 승인/반려/차단/삭제 함수는 이전에는 필수값 부족 시 내부에서 메시지만 띄우고 `return`해, 바깥 공통 상태줄이 잘못 `완료`로 찍힐 수 있었습니다.
- 이 분기를 `throw -> 공통 catch` 구조로 바꿔, 실제 실패가 상단 상태와 토스트에 정확히 반영되게 했습니다.

### 5. 로그인 UX 정리

- 관리자 로그인 화면을 `<form>` submit 구조로 바꿨습니다.
- Enter 제출 흐름이 자연스러워졌고, 클릭 전용 구조보다 브라우저 기본 동작과 맞게 정리됐습니다.

## 검증

- 로컬 관리자 페이지 `http://127.0.0.1:8136/admin.html`에서 실제 관리자 로그인 확인
- 로그인 후 상단 상태 배너 렌더링 확인
- `공지 저장` 성공 토스트 `✅ 공지 저장 완료 (사이드바)`와 상단 상태 `공지 저장 완료 · 시각` 확인
- `타이머/레이아웃 저장` 성공 흐름 확인
- 구독 운영 탭 신청 목록 및 `복호화` 버튼 노출 확인
- 운영 RTDB rules 배포 후 `config/notice_sidebar` 쓰기 가능 상태 재확인

## 운영 반영 경계

- 이번 작업에서 **실제 운영 반영된 것**은 RTDB rules(`config/.write`)입니다.
- `admin.html`, `admin.css` 개선은 로컬 관리자 도구용 변경이며, 공개 GitHub Pages `admin.html` 차단 정책은 유지합니다.
- 따라서 공개 사용자 화면 재배포(`public-clean`)는 이번 작업에 포함하지 않았습니다.
