# 2026-04-09 구독 로그인 ID·이메일 정리
작성일시: 2026-04-09 08:41:44 KST

## 사용자 요청
- 구독 신청 시 이메일을 로그인 ID로 써도 되는지 명확히 해달라.
- 로그인은 `로그인 ID 또는 이메일`만 허용하고 `신청번호`는 로그인에 쓰지 않게 해달라.
- 관리자가 수기로 추가한 `ID만 있는 계정`도 그대로 로그인되게 유지해달라.
- 관련 페이지 문구도 실제 동작과 맞게 정리해달라.

## 안전 기준
- 작업 시작 전 안전 기준 커밋은 `4511dfa` (`backup: snapshot before login id-email cleanup`)로 확보되어 있었다.
- 운영 `main` 브랜치 push, 운영 Firebase 기본값 변경은 하지 않았다.

## 수행 내용
- `main.js`
  - 공개 로그인 경로를 `로그인 ID 또는 이메일`로만 정리했다.
  - `REQ-...` 형태 신청번호를 로그인 식별자로 받아들이던 분기를 제거했다.
  - 이메일 입력 시에는 먼저 신청 이메일 조회를 시도하고, 해당 신청이 없으면 같은 문자열을 계정 로그인 ID로 다시 시도하게 유지했다.
  - 이메일이 아닌 값은 계정 로그인 경로로만 처리해 수기 등록 `ID-only` 계정이 그대로 로그인되게 유지했다.
  - 신청 상태 조회는 이메일 형식이 아닐 때 즉시 막고, 안내 문구도 이메일 전용으로 바꿨다.
- `index.html`
  - 고급 안내, 신청 흐름, 신청서 설명, 비밀번호 안내, 상태 조회 설명을 모두 `신청 이메일 또는 로그인 ID` 기준으로 정리했다.
  - 신청번호는 로그인에 쓰지 않는다는 문구를 명시했다.
  - 로그인 ID 입력란 예시에 일반 ID와 이메일 형식을 모두 보여주도록 바꿨다.
- `advanced-tools.html`
  - 팝업 인증 화면 문구를 `신청 이메일 또는 로그인 ID` 기준으로 정리했다.
  - 신청번호는 로그인에 쓰지 않는다고 직접 안내하도록 보강했다.
- `admin.html`
  - 관리자용 설명에서 수기 등록 계정은 이메일이 없어도 `로그인 ID + 비밀번호`로 로그인된다는 점을 명시했다.
  - 표 헤더와 도움말을 `로그인 ID` 기준으로 통일했다.
- `site-text-config.js`
  - 기본 문구와 상태 메시지를 새 로그인 정책에 맞게 갱신했다.
  - 운영 Firebase에 남아 있을 수 있는 예전 기본 문구가 새 기본값으로 자동 교체되도록 legacy 기본값 매핑을 보강했다.

## 검증
- `node --check main.js`
- `node --check site-text-config.js`
- 로컬 정적 서버 `http://127.0.0.1:8124/` 재사용 후 Playwright로 확인
- 메인 고급 안내 모달에서 아래 문구 확인
  - 로그인: `신청 이메일 또는 로그인 ID`
  - 신청 흐름: 신청 상태는 이메일로 확인, 승인 후 이메일 또는 로그인 ID로 로그인
  - 신청서 설명: 로그인 ID는 일반 ID와 이메일 형식 모두 가능
- 신청 상태 조회 입력란에 `REQ-TEST-1234`를 넣었을 때 즉시 `신청 조회는 신청 이메일과 조회 비밀번호로만 할 수 있습니다.` 문구가 나오는 것 확인
- 브라우저 안에서 `window.applyManualSubscriptionConfig(...)`와 fetch/crypto mock으로 로그인 분기 확인
  - `REQ-TEST-1234` 로그인 시도 -> `ok: false`, `reason: "not_found"`
  - `manualOnlyId` 로그인 시도 -> `ok: true`, `mode: "account"`
  - `me@example.com` 로그인 시도 -> `ok: true`, `mode: "account"`
- `advanced-tools.html`에서 신청번호 비사용 문구 반영 확인

## 메모
- 신청번호(`requestId`)는 관리자 승인/복구/백업 추적용 내부 키로만 유지하고, 공개 로그인/조회 경로에서는 사용하지 않게 정리했다.
