# 2026-04-09 실시간 접속자 정리 및 고급 구독 저장 권한 핫픽스
작성일시: 2026-04-09 00:36:52 KST

사용자 요청: 현재 접속자 수가 343명으로 보이는 이유를 점검하고, 관리자 페이지에서 기존 고급 구독 만료일 수정 시 `Permission denied`가 뜨는 원인을 해결한다.

## 1. 접속자 수 점검 결과
- 최초 확인 시 운영 `active_visitors`에는 숫자 세션과 예전 형식 값이 섞여 있었다.
- 중간 시점 스냅샷 기준:
  - 총 키 348개
  - 숫자 세션 345개
  - stale 숫자 세션 333개
  - legacy 비숫자 값 3개
- 이후 최신 재조회(`2026-04-09 00:35~00:36 KST`) 결과:
  - 총 키 330개
  - stale/legacy 4개
- stale/legacy 4건을 실제 운영 DB에서 제거한 뒤 재확인 결과:
  - 총 키 327개
  - 최근 150초 안 하트비트를 보낸 fresh 세션 327개
  - stale/legacy 0개

## 2. 해석
- 즉 `343명`은 전부 유령값만으로 부풀어진 수치는 아니었다.
- 다만 예전 형식 값과 오래된 세션 찌꺼기는 일부 남아 있었고, 관리자 버튼 이름이 이를 더 헷갈리게 만들고 있었다.
- 현재 구조는 **사람 수**가 아니라 **최근 하트비트를 보낸 브라우저 세션/탭 수**를 센다.

## 3. 접속자 버튼 수정
- 기존 버튼명 `실시간 접속자 초기화`는 실제 의미와 달랐다.
- `admin.html`에서 아래처럼 수정했다.
  - 버튼명: `유령 세션 정리`
  - 동작: 최근 150초 안의 fresh 세션은 유지하고, stale/legacy 키만 삭제
  - 결과 문구: 실제 활성 세션은 유지된다는 점을 명시

## 4. `Permission denied` 원인
- 관리자 페이지의 고급 구독 저장은 `advancedAccountLicenses` 루트를 통째로 `set(...)`하는 구조였다.
- 하지만 운영 rules는 `$loginIdKey` 자식 쓰기만 열려 있고, 루트 `.write`는 닫혀 있었다.
- 그래서 아래 같은 수정에서 저장이 막혔다.
  - 기존 사용자 만료일 수정
  - 새 수동 구독 추가

## 5. 권한 핫픽스
- `database.rules.json`
  - `advancedAccountLicenses/.write = "auth != null"` 추가
- 이렇게 해서 관리자 로그인 상태에서는 루트 갱신이 가능해졌고, 기존 `set(ref(db, 'advancedAccountLicenses'), ...)` 흐름이 정상 동작하게 했다.

## 6. 운영 반영
- RTDB rules 재배포 완료
- stale/legacy `active_visitors` 4건 운영 정리 완료
- 이후 운영 `active_visitors`는 stale/legacy 0건 상태 확인

## 7. 사용자 안내 포인트
- 접속자 수는 현재 구조상 “브라우저 세션 수”다. 사람 수와 1:1이 아니다.
- `esun`, `sh` 같은 수동 계정은 삭제됐으므로, 이제 새로 추가 후 저장하면 실제 로그인 레코드까지 같이 생성되어야 한다.
- 기존 관리자 페이지가 열려 있었다면 `Ctrl+F5` 새로고침 후 다시 시도하는 것이 안전하다.
