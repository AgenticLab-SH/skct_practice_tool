# 2026-04-08 구독 관리자 UI 단순화 및 ID/비밀번호 체계 전환 작업 기록
작성일시: 2026-04-08 11:22:05 KST

## 사용자 요청
- 개발자 페이지의 구독 관리 영역이 너무 복잡해서 무엇을 어떻게 관리해야 하는지 이해하기 어렵다.
- `외부 ID` 같은 용어가 어렵고, `ID + 비밀번호`로 관리되게 바꾸고 싶다.
- 관리자 표는 아래 항목 중심으로 다시 정리하고 싶다.
  - 상태
  - 구독 종류
  - 사용자 식별(이름/이메일)
  - ID
  - 비밀번호
  - 만료일
  - 메모
  - 삭제
- 만료일은 해당 날짜의 정오 12:00(KST)까지 유효하게 처리하고 싶다.

## 수행 내용
- `admin.html`
  - 구독 관리 설명 문구를 `ID + 비밀번호` 체계 기준으로 다시 작성했다.
  - `사용자 식별`, `ID`, `비밀번호`, `만료일`이 각각 무슨 뜻인지 한눈에 보이도록 안내 카드 추가했다.
  - 구독 표 헤더를 `상태 / 구독 종류 / 사용자 식별 / ID / 비밀번호 / 만료일 / 메모 / 삭제`로 단순화했다.
  - `외부 ID` 입력 칸을 제거하고 `ID` 입력 칸으로 교체했다.
  - `구독 종류`를 자유입력이 아니라 `3일 이용권 / 7일 이용권 / 2주 이용권 / 1달 이용권 / 1년 이용권` 선택형으로 변경했다.
  - 상태 표시를 `active/paused/expired` 대신 `사용중/일시중지/만료`로 보이게 바꿨다.
  - 저장 시 `ID 중복`만 막고, `비밀번호 중복`은 더 이상 강제하지 않도록 정리했다.
  - 새 행은 `ID`, `사용자 식별`, `비밀번호`가 비어 있으면 저장 단계에서 바로 막도록 검증을 추가했다.
  - 비밀번호는 저장 후 원문이 다시 보이지 않는다는 점을 화면에서 바로 알 수 있게 문구를 바꿨다.
  - 빈 구독 목록도 저장 가능하게 하되, 화면 편집을 위해서는 빈 행 1개를 임시로 렌더링하도록 분리했다.
- `main.js`
  - 구독 데이터 정규화 로직을 `planType`, `userIdentity`, `loginId` 중심으로 읽도록 바꿨다.
  - 예전 데이터의 `planName`, `memberLabel`, `externalId`는 새 필드로 흡수하는 마이그레이션 호환 경로만 남겼다.
  - 고급 인증 판정을 `ID + 비밀번호` 조합 기준으로 바꿨다.
  - 만료 시간 계산을 `23:59:59`가 아니라 `12:00:00+09:00` 기준으로 변경했다.
  - 사용자 안내 문구와 오류 문구를 `비밀번호만`이 아니라 `ID + 비밀번호` 기준으로 다시 작성했다.
  - 브리지 객체에 `validateCredentials`, `validateCredentialsDetailed`를 추가해 보조 인증 창도 같은 규칙을 쓰게 했다.
- `index.html`
  - 고급 기능 안내 모달에 `고급 ID 입력` 칸을 추가했다.
  - 사용자 설명 문구를 `관리자에게 받은 ID와 비밀번호` 기준으로 변경했다.
- `advanced-tools.html`
  - 보조 인증 창에도 `ID` 입력 칸을 추가했다.
  - 상단 설명 및 실패 문구를 `ID + 비밀번호` 체계에 맞게 수정했다.

  - 빈 상태에서 `고급 모드 열기` 클릭 시 `ID와 비밀번호를 모두 입력해주세요.` 메시지 확인
- `admin.html`
  - 로그인 전 DOM 기준으로 구독 표 헤더가 `상태 / 구독 종류 / 사용자 식별 / ID / 비밀번호 / 만료일 / 메모 / 삭제`로 바뀐 것 확인
  - 설명 영역에 `ID + 비밀번호` 체계 안내 문구가 포함된 것 확인
- `advanced-tools.html`
  - `ID` 입력 칸과 `비밀번호` 입력 칸이 모두 보이는 것 확인
  - 상단 안내 문구가 `ID와 비밀번호 확인 후`로 바뀐 것 확인

## 제한 및 메모
- 관리자 실제 로그인 후 Firebase 저장까지는 이번 턴에서 실행하지 않았다. 이유는 실서버/실DB 반영과 연결될 수 있어, 로컬 화면/DOM 기준 검증까지만 수행했다.
- 기존 운영 기본 계정 데이터는 호환을 위해 읽을 수 있게 남겨두었지만, 앞으로는 관리자 화면에서 `ID`를 채운 구조로 정리해 두는 것이 가장 안전하다.
