## 기존 구조 확인 결과
- 기존 `community.js` 게시판은 `닉네임 + 비밀번호` 편집/삭제 구조지만, 게시글 본문은 Firebase에 평문 저장됨
- 따라서 기존 게시판을 그대로 `비밀글`처럼 가리는 방식은 개발자도구/직접 RTDB 조회로 우회 가능하므로 보안상 부적합
- 기존 고급 구독은 이미 `admin.html`의 `advancedFeatureConfig.subscriptions`에서 `ID + 비밀번호 + 만료일`로 관리되고 있음
- 관리자 페이지는 이미 비밀번호 해시와 관리자 복구용 암호문을 함께 저장하는 구조가 있어, 발급 계정 관리는 이 표를 확장 활용하는 편이 가장 자연스러움

## 최종 적용 방향
- 사용자 쪽은 `공개 비밀글 목록` 대신 `신청번호 + 조회 비밀번호` 조회형으로 전환
- 요청 본문은 브라우저에서 암호화 후 Firebase에 저장
- 관리자 쪽은 공개키/개인키 기반으로 요청 본문을 복호화
- 승인 시 `advancedFeatureConfig.subscriptions`에 실제 고급 계정을 자동 추가
- 사용자 조회 화면에서는 승인 결과로 발급된 `ID / 비밀번호 / 만료일`을 다시 확인 가능

## 운영상 중요한 판단
- 사용자가 신청 시점에 최종 고급 비밀번호를 직접 남기게 하지 않았음
- 이유
  - 결제 확인 전 민감정보를 더 많이 저장할 필요가 없음
  - 운영자가 승인 시 즉시 발급하는 편이 중복/오타/분실 대응이 쉬움
  - 나중에 토스 자동 승인으로 전환할 때도 흐름을 그대로 재활용하기 쉬움

## 남은 운영 메모
- 신청 암호화 공개키는 Firebase에 저장되지만, 개인키는 현재 관리자 브라우저에만 남음
- 개인키를 안전한 오프라인 위치에도 별도 보관해야 기존 신청 본문을 계속 복호화할 수 있음
- 수동 승인 전에는 관리자 페이지의 `백업 다운로드` 버튼으로 JSON을 먼저 받아두는 습관이 필요함
- 복원은 관리자 페이지에서 내려받은 `전체 DB 백업 JSON`만 허용되며, 부분 백업 파일은 복원 대상에서 제외함
- 고급 안내에 실제 UI 캡처나 예시 이미지가 들어가면 처음 이용자 입장에서 이해 속도를 더 끌어올릴 수 있음
