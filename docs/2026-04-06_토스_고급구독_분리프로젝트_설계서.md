# 2026-04-06 토스 고급구독 분리프로젝트 설계서
작성일시: 2026-04-06 14:02:00 KST

이 문서는 현재 SKCT 웹과 분리된 별도 프로젝트로 진행할 `토스 결제 기반 고급 구독 시스템`의 초안 설계서입니다. 아직 구현은 하지 않고, 구조와 우선순위만 확정합니다.

## 목표
- 일반 웹과 별도로 고급 구독 결제와 계정 발급을 처리한다.
- 결제 성공 사용자가 직접 원하는 고급 비밀번호를 설정할 수 있게 한다.
- 기존 정적 웹에 평문 비밀번호를 내려보내지 않는다.
- 나중에 토스 결제 성공 내역이 관리자 구독 표와 자동 연동되게 한다.

## 왜 분리 프로젝트로 가야 하는가
- 토스 결제 승인 API와 웹훅 검증에는 시크릿 키가 필요하다.
- 시크릿 키는 정적 HTML/JS에 넣을 수 없다.
- 따라서 결제 승인, 웹훅 수신, 구독 생성, 비밀번호 해시 저장은 반드시 서버 또는 서버리스 함수에서 처리해야 한다.

## 권장 아키텍처
1. `billing-web`
- 사용자용 결제 진입 페이지
- 상품 선택
- 토스 결제위젯 실행

2. `billing-api`
- 결제 승인 API
- 웹훅 수신 API
- 구독 생성/갱신 API
- 비밀번호 설정/변경 API

3. `shared-db`
- 결제 주문
- 구독 레코드
- 비밀번호 해시
- 만료일 / 상태 / 외부 주문 ID 저장

4. `skct_tool`
- 고급 모드 인증 시 API 또는 미러링된 안전 데이터만 읽음
- 평문 비밀번호는 저장하지 않음

## 권장 데이터 모델
### 주문
- `orderId`

## 사용자 흐름
1. 결제 페이지에서 상품 선택
2. 토스 결제 진행
3. 결제 승인 완료
4. 구독 생성
5. `고급 비밀번호 설정` 페이지로 이동
6. 사용자가 원하는 비밀번호 입력
7. 해시 저장
8. SKCT 웹에서 해당 비밀번호로 고급 모드 진입

## 관리자 흐름
1. 관리자 페이지에서 구독 목록 조회
2. 상태 변경
3. 만료일 수정
4. 비밀번호 강제 초기화
5. 수동 구독 추가
6. 토스 결제건과 수동 구독건을 같은 표에서 관리

## 1차 구현 범위
- 토스 결제위젯 기반 1회 결제
- 결제 성공 후 구독 생성
- 사용자 비밀번호 설정
- 관리자 구독 표 조회/수정
- SKCT 웹 고급 인증 연동

## 2차 구현 범위
- 구독 연장 / 재결제
- 자동 만료 처리
- 웹훅 재처리
- 결제 실패/환불 반영
- 관리자 검색/필터/CSV 내보내기

## 보안 원칙
- 토스 시크릿 키는 서버 환경변수로만 보관
- 비밀번호는 평문 저장 금지
- 해시는 `salt + sha-256` 이상으로 저장
- 결제 승인 여부는 반드시 서버에서 검증
- 프론트에서 결제 성공 화면만 보고 구독 생성하지 않음

## 현재 SKCT 웹과의 연결 원칙
- 지금 단계에서는 SKCT 웹 관리자 페이지의 `고급 구독 표`를 수동 관리 UI로 먼저 만든다.
- 나중에 분리 프로젝트가 준비되면, 같은 구독 스키마를 사용해 자동 반영되게 한다.
- 즉 현재 스테이징 구현은 미래 토스 프로젝트와 이어질 수 있는 관리 스키마를 먼저 맞추는 작업이다.
