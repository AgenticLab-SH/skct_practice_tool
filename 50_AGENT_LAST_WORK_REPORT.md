# SKCT Tool 최근 운영 반영 리포트
작성일시: 2026-04-09 01:55:00 KST

이 문서는 최근 운영 반영 기준을 빠르게 이어보기 위한 요약 기록입니다.

## 2026-04-09 운영 반영 예정: 신청번호 사용자 비노출 및 이메일 조회 전환
- 이용권 신청 조회는 이제 사용자 기준으로 `이메일 + 조회 비밀번호` 중심 흐름으로 정리했습니다.
- 신청번호는 Firebase 내부 식별과 관리자 작업용으로만 남기고, 사용자 신청/조회 결과/고급 모드 안내에서는 숨기도록 코드를 조정했습니다.
- 새 신청 저장 시에는 `subscriptionRequests/<requestId>` 외에 `subscriptionRequestLookup/<sha256(email::password)>`도 같이 저장합니다.
- 이 lookup 저장이 실패하면 신청 전체를 되돌리고 실패 처리하므로, 성공한 신청은 항상 이메일 기준 조회가 가능해야 합니다.
- 관리자 페이지에서 신청 삭제 시 lookup 인덱스도 같이 제거되도록 맞췄습니다.
- 원격 `config/siteTextConfig`는 코드 기본값으로 다시 동기화해, 예전 `신청번호` 안내 문구가 운영 화면에 남지 않도록 정리했습니다.

## 2026-04-09 운영 반영 예정: 실시간 접속자 정리 버튼 의미 수정 및 고급 구독 저장 권한 복구
- 관리자 페이지의 `실시간 접속자 초기화`는 실제 활성 세션까지 없애는 기능처럼 보였지만, 운영상 필요한 건 stale/legacy 세션 정리였다.
- 버튼을 `유령 세션 정리`로 바꾸고, 최근 150초 안 하트비트를 보낸 세션은 유지한 채 오래된 값만 삭제하도록 수정했다.
- 운영 `active_visitors`는 정리 후 stale/legacy 0건 상태를 확인했다.
- 고급 구독 저장 시 `Permission denied`가 나던 원인은 `advancedAccountLicenses` 루트 쓰기 권한이 rules에 없었기 때문이다.
- `advancedAccountLicenses/.write = auth != null` 규칙을 추가해, 기존 사용자 만료일 수정과 새 수동 구독 저장이 다시 가능하도록 복구했다.

## 2026-04-09 운영 반영 완료: 수동 고급 계정의 실제 로그인 경로 복구
- 문제의 핵심은 `고급 구독 내역` 저장과 공개 페이지 로그인 경로가 서로 달랐다는 점입니다.
- 관리자 페이지는 `config/advancedFeatureConfig`에 계정을 저장했지만, 공개 페이지는 그 경로를 더 이상 로그인에 쓰지 않아 `esun` 같은 수동 계정이 즉시 로그인되지 않았습니다.
- 이를 해결하기 위해 `advancedAccountLicenses/<loginId>` 공개 로그인 레코드를 새로 도입했습니다.
  - 관리자 저장 시 서명된 라이선스 번들을 만들고
  - 그 번들을 계정 비밀번호로 다시 암호화해 저장하며
  - 공개 페이지는 이 레코드를 읽어 비밀번호로 복호화한 뒤 공개키로 서명을 검증합니다.
- 이 구조 덕분에 비밀번호 해시를 공개하지 않으면서도 수동 추가 ID/비밀번호 로그인이 다시 동작합니다.
- 세션용 평문 보조값 `_plainPasswordLocal`은 관리자 화면 메모리에만 남기고, Firebase 저장본에서는 제거하도록 정리했습니다.
- 운영 RTDB `advancedAccountLicenses`에는 신청 원본에서 비밀번호를 복구할 수 있는 `ods1106`, `mingyu7275` 레코드를 먼저 재생성했습니다.
- `esun`, `sh`는 해시만 남아 있어 평문 비밀번호를 역산할 수 없으므로, 운영자가 비밀번호를 한 번 다시 입력해 저장해야 실제 로그인도 살아납니다.

## 2026-04-08 운영 반영 완료: 관리자 페이지 탭 재구성 및 키 안내 정리
- 관리자 페이지를 긴 단일 스크롤 대신 탭 구조로 재구성했습니다.
  - `🔐 구독`
  - `⚙️ 기본 설정`
  - `📝 문구/공지`
  - `💬 커뮤니티`
  - `🧭 운영 개요`
- 빠른 이동 버튼은 이제 대상 카드가 있는 탭을 먼저 연 뒤 해당 위치로 이동합니다.
- `신청 복호화 키`와 `라이선스 서명 키`의 역할을 실제 운영 흐름 기준으로 다시 설명하고, 표준 저장 경로를 관리자 화면에 직접 표시했습니다.
- 운영 HTTPS 관리자 페이지에서는 로컬 키 브리지를 자동 시도하지 않도록 바꿨습니다. 기본 흐름은 `키 폴더 연결`이며, 로컬 브리지는 개발용 예비 도구로만 남겼습니다.
- `고급 구독 내역`의 비밀번호 `확인` 버튼은 관리자 암호문이 없는 계정이라도, 메모 안에 연결된 신청번호가 있고 신청 복호화 키가 준비되어 있으면 수동 신청 원본에서 현재 비밀번호를 다시 보여줄 수 있습니다.

## 2026-04-08 운영 반영 완료: 수동 구독 중복 방지 및 승인 동기화
- 수동 구독 신청 폼에 제출 중 잠금과 최근 동일 신청 차단을 넣어, 같은 브라우저에서 짧은 시간 안에 같은 신청이 여러 번 들어가는 문제를 줄였습니다.
- 관리자 승인 버튼은 이제 라이선스만 발급하지 않고 `config/advancedFeatureConfig`의 `고급 구독 내역`까지 같이 갱신합니다.
- 승인 카드의 기본 만료일 계산 기준을 `현재 시각`이 아니라 `신청자가 고른 시작일 + 이용권 일수`로 보정했습니다.
- 운영 RTDB의 실제 중복 신청 2묶음을 정리했습니다.
  - `ods1945@naver.com` 최신 신청 `REQ-MNPYGTBA-B5Y4` 승인, 이전 5건 반려
  - `mingyu7275@naver.com` 최신 신청 `REQ-MNQ2I11P-IIXB` 승인, 이전 2건 반려
- 운영 RTDB `config/advancedFeatureConfig`에는 `ods1106`, `mingyu7275` 두 계정이 실제 저장됐고, 두 승인 신청 모두 사용자 비밀번호 경로 복호화 및 라이선스 서명 검증을 통과했습니다.
- 자세한 데이터 처리 기록은 `docs/agent/worklog/2026-04-08_manual_subscription_duplicate_audit_and_approval.md`를 참고합니다.

## 2026-04-08 운영 반영 완료: 보안 강화 및 인증 구조 정리
- 관리자 로그인은 공개 RTDB 해시 비교 대신 Firebase Auth 기준으로 정리했습니다.
- 고급 기능 해금은 로컬 저장소 플래그가 아니라 서명된 라이선스 번들 검증 구조로 바꿨습니다.
- 수동 신청 승인 흐름은 고급 계정을 DB에 직접 추가하는 대신, 신청별 라이선스를 발급해 사용자 브라우저에서 검증하는 방식으로 바뀌었습니다.
- 운영/스테이징 페이지의 공지, 후원 문구, 링크 렌더링 경로에 sanitize 를 적용해 저장형 XSS 재현 경로를 막았습니다.
- 공개 페이지는 더 이상 Firebase `config` 루트를 통째로 읽지 않고, 공개가 필요한 키만 개별 조회합니다.
- `database.rules.json` 기준 RTDB rules 배포까지 완료했습니다.
- 운영 RTDB `config/manualSubscriptionConfig`에는 라이선스 검증 공개키를 추가했고, `config/siteTextConfig/advancedGuide`, `config/siteTextConfig/messages`도 최신 흐름 기준으로 동기화했습니다.
- 운영 RTDB의 레거시 민감 경로 `config/adminHash`, `config/advancedFeatureConfig`는 실제 DB에서도 제거했습니다.
