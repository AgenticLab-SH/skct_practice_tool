# SKCT Tool 학습 노트
작성일시: 2026-04-09 01:33:00 KST

이 문서는 이번 작업에서 다시 배운 점을 초보자도 따라갈 수 있게 적는 메모입니다.

## 관리자 장부와 실제 로그인 경로는 다를 수 있다
- 화면에 `저장 완료`가 떠도, 그 데이터가 실제 로그인 검증 코드에서 읽히지 않으면 사용자는 로그인할 수 없습니다.
- 이번 문제는 `config/advancedFeatureConfig`에 수동 계정이 잘 저장됐는데도, 공개 페이지가 그 경로를 로그인에 쓰지 않아 생겼습니다.
- 그래서 “Firebase에 저장됐는가?”만 보면 안 되고, 꼭 아래 3가지를 같이 확인해야 합니다.
- 관리자 저장 경로
- 공개 페이지가 실제로 읽는 경로
- 그 둘을 이어 주는 검증 로직

## 해시만 있으면 로그인 검증은 되지만 평문 복구는 안 된다
- `passwordSalt + passwordHash`만 있으면 사용자가 입력한 비밀번호가 맞는지 비교할 수는 있습니다.
- 하지만 그 상태에서는 관리자도 원래 비밀번호를 다시 볼 수 없습니다.
- 평문을 다시 보여주려면 최소한 아래 둘 중 하나가 더 필요합니다.
- 관리자만 풀 수 있는 복구용 암호문
- 원본 신청 본문처럼 비밀번호가 들어 있는 다른 안전한 원천

## 공개 로그인용 데이터는 공개해도 되는 형태로 따로 만들어야 한다
- 민감한 관리자 장부를 그대로 공개하면 안 됩니다.
- 이번에는 공개 페이지 전용으로 `advancedAccountLicenses/<loginId>`를 만들고, 여기에 아래만 저장했습니다.
- 서명된 라이선스 번들
- 그 번들을 비밀번호로 다시 암호화한 결과
- 공개 페이지는 비밀번호로 복호화하고, 공개키로 서명을 검증합니다.
- 이렇게 하면 비밀번호 해시나 관리자 내부 장부를 공개하지 않고도 ID/비밀번호 로그인을 만들 수 있습니다.

## 세션 편의값과 영구 저장값은 반드시 분리해야 한다
- 관리자 화면에서 방금 입력한 비밀번호를 다시 보여주기 위해 메모리에 잠깐 들고 있는 값은 편의용입니다.
- 이런 값이 그대로 Firebase에 저장되면 보안 문제가 됩니다.
- 따라서 이번처럼 `_plainPasswordLocal` 같은 값은 아래처럼 경계를 분명하게 나눠야 합니다.
- 화면 세션에서는 허용
- Firebase 저장 직전에는 제거

## TODO는 해결 상태별로 나누면 다음 작업자가 훨씬 빠르게 들어온다
한 폴더 안에 `완료`, `부분 해결`, `미해결`이 뒤섞여 있으면 다음 작업자는 먼저 “지금 뭘 해야 하지?”부터 다시 판별해야 합니다.

그래서 TODO를 다시 정리할 때는 아래처럼 나누는 편이 좋습니다.

- `부분 해결`: 이미 코드가 일부 들어간 항목
- `미해결`: 구현 흔적이 거의 없는 항목

이렇게 나누면 다음 작업자는 처음부터 전체를 읽지 않고도, 바로 자기 작업 범위를 잡을 수 있습니다.

## TODO 문서는 요청문 복사본이 아니라 실행 문서여야 한다
좋은 TODO 문서에는 단순 요구사항만 있으면 부족합니다.

다음 정보가 함께 있어야 실제로 바로 작업할 수 있습니다.

- 현재 상태
- 이미 반영된 근거
- 아직 미완료인 이유
- 바로 시작할 첫 단계
- 완료 판단 기준

이 다섯 가지가 있으면, 다음 에이전트가 같은 분석을 다시 반복하지 않아도 됩니다.

## 이메일 조회는 평문 이메일 인덱스보다 `이메일 + 비밀번호` 조합 키가 안전하다
- 사용자가 신청 상태를 조회할 때 이메일로 찾게 하면 편하지만, 단순히 `email -> requestId` 공개 인덱스를 만들면 누가 어떤 이메일로 신청했는지 추측하기 쉬워집니다.
- 그래서 이번에는 평문 이메일 대신 `정규화한 이메일 + 조회 비밀번호`를 합쳐 SHA-256 키를 만들고, 그 키로만 조회 인덱스를 찾게 했습니다.
- 이렇게 하면 사용자는 여전히 “내 이메일로 조회한다”고 느끼지만, 실제 공개 경로에는 이메일 원문이 남지 않습니다.
- 즉 편의성과 공개 노출 억제를 둘 다 챙길 때는 `사용자 입력 2개를 합친 조회 키`가 실용적인 절충안이 됩니다.

## 실행 TODO와 추가 개선 백로그는 분리해야 한다
- 어떤 항목은 지금 바로 구현해야 하는 작업이고, 어떤 항목은 제품 품질을 높이는 개선 과제입니다.
- 이 둘을 한 폴더에 섞어 두면 다음 작업자가 “지금 당장 해야 할 일”과 “나중에 더 좋게 만들 일”을 구분하느라 시간을 씁니다.
- 그래서 이번처럼 `10_부분해결`, `20_미해결`은 실행 TODO로 두고, `30_추가개선`은 구조/문구/UX/보안 품질 향상용 백로그로 따로 빼는 편이 더 실용적입니다.

## 개선 문서에도 기능 보존 체크가 먼저 와야 한다
- “더 좋게 바꾸기” 작업은 겉으로는 리스크가 낮아 보여도, 실제로는 기존 사용 흐름을 가장 쉽게 망가뜨립니다.
- 그래서 개선 TODO를 쓸 때도 아래 항목을 먼저 적는 편이 안전합니다.
- 반드시 보존할 현재 기능
- 위험 구간
- 코드 기준 보존 포인트
- 검증 체크리스트
- 이렇게 해 두면 다음 작업자가 구조나 문구를 손보면서도 핵심 기능을 덜 잃어버립니다.
