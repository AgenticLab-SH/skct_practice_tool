# SKCT Tool 학습 노트
작성일시: 2026-04-09 17:12:00 KST

이 문서는 이번 작업에서 다시 배운 점을 초보자도 따라갈 수 있게 적는 메모입니다.

## 운영 공지는 저장 속도와 표시 속도를 같이 맞춰야 한다
- 관리자 화면에서 저장이 성공해도, 공개 페이지가 처음 한 번만 읽는 구조면 이미 열려 있던 화면은 바뀌지 않습니다.
- 이번 문제의 핵심은 Firebase 저장 실패가 아니라, 메인 페이지가 `config/notice_sidebar`를 다시 읽지 않는 구조였다는 점입니다.
- 그래서 공개 설정 전체는 1회 조회를 유지하되, 공지처럼 자주 바뀌고 저장 직후 확인해야 하는 값만 `onValue` 실시간 구독으로 따로 빼는 편이 안전합니다.
- 또 공개 키 하나가 잠깐 읽기 실패해도 나머지 설정까지 같이 멈추지 않게, 키별 오류를 분리해 처리하는 편이 운영 확인에 유리합니다.

## 관리자 저장 실패는 클릭 지점에서 바로 알려줘야 한다
- 이번 확인에서 운영 DB `config/notice_sidebar` 값이 그대로였다는 것은, 화면 반영 문제가 아니라 저장 자체가 실패했다는 뜻이었습니다.
- 그런데 기존 관리자 화면은 `공지 저장` 같은 개별 버튼 클릭 시 `await set(...)` 실패를 사용자에게 거의 알려주지 못했습니다.
- 특히 Firebase RTDB 쓰기는 `auth != null` 조건이라 로그인 세션이 만료되면 `Permission denied`가 나는데, 이걸 바로 설명해 주지 않으면 사용자는 “저장은 된 줄 알았는데 왜 안 바뀌지?”로 느끼게 됩니다.
- 그래서 저장 버튼에서는 성공 메시지뿐 아니라 실패 메시지도 같은 강도로 보여주는 편이 운영 안정성에 중요합니다.

## 상시 노출 패널은 정보 가치보다 공간 비용이 더 클 수 있다
- 어떤 정보가 유용하더라도, 항상 화면을 차지하면 핵심 작업 영역을 더 많이 해칠 수 있습니다.
- 이번처럼 계산기와 메모장 높이가 줄어들면 사용자는 정보보다 불편을 먼저 느낍니다.
- 그래서 상태성 정보는 아래처럼 나누어 생각하는 편이 안전합니다.
- 지금 바로 봐야 하는 핵심 정보인가
- 버튼을 눌렀을 때만 봐도 충분한 정보인가
- 핵심 작업 공간을 줄일 만큼 가치가 있는가

## 고급 기능도 기본 화면 밀도를 먼저 지켜야 한다
- 기능이 많다고 해서 한 화면에 계속 보여주는 것은 좋은 고급화가 아닙니다.
- 실사용자가 자주 쓰는 영역은 타이머, 메모장, 계산기, OMR입니다.
- 고급 정보는 좌측 버튼이나 모달로 빼고, 기본 작업 면적을 유지하는 쪽이 더 실전적일 수 있습니다.

## 운영 공지는 노출 위치별로 키를 분리하는 편이 덜 꼬인다
- Help 모달 공지와 사이드바 공지를 같은 키로 묶으면, 한쪽을 바꿀 때 다른 쪽 의도까지 같이 흔들리기 쉽습니다.
- 이번에는 `config/notice_sidebar`를 따로 만들고, 관리자 페이지에서도 `사이드바 공지`를 별도 대상으로 두었습니다.
- 이렇게 하면 운영자는 아래를 더 분명하게 관리할 수 있습니다.
- 어디에 뜨는 공지인지
- 어떤 문구가 현재 화면 기준인지
- 어느 공지만 잠깐 끄고 싶은지

## 자주 바뀌는 운영 문구는 버튼 라벨과 본문 저장 경로를 나눠도 된다
- 버튼 라벨은 UI 일부라서 `siteTextConfig`처럼 화면 문구 체계에 두는 편이 자연스럽습니다.
- 반면 실제 공지 본문은 `show`, `type`, `updated` 같은 운영 상태를 함께 가지므로 독립 공지 저장 경로가 더 잘 맞습니다.
- 즉 “무조건 한 군데에 몰기”보다, 데이터 성격에 따라 경계를 나누는 것이 오히려 관리가 쉬울 수 있습니다.

## textarea 드래그 선택은 브라우저마다 native 동작 차이가 크다
- 같은 `textarea`라도 브라우저마다 하단 경계 근처 drag auto-scroll과 selection 확장 동작이 다르게 느껴질 수 있습니다.
- 이번에는 Chromium은 스크롤과 selection이 비교적 자연스럽게 같이 따라왔지만, Firefox와 WebKit은 스크롤만 먼저 움직이거나 선택 끝점이 더 보수적으로 늘었습니다.
- 그래서 메모장처럼 긴 텍스트를 자주 드래그하는 UI는 native 동작만 믿지 말고 아래를 같이 준비하는 편이 안전합니다.
- 경계 근처 auto-scroll 보조
- capture 단계 이벤트 추적
- selection 확장이 멈췄을 때의 fallback

## 드래그 종료 직후 다른 패널 포커스가 끼어들면 끊김처럼 느껴진다
- 사용자는 내부적으로 무슨 포커스 이동이 일어났는지 보지 못합니다.
- 대신 “드래그하다가 뭔가 걸린다”는 체감으로 기억합니다.
- 그래서 텍스트 드래그 직후 다른 패널이 바로 focus를 가져가는 구조는 UX를 쉽게 망칩니다.
- 이번처럼 아주 짧은 suppression 구간을 두면 체감 끊김을 줄이는 데 효과가 있습니다.

## 관리자 장부와 실제 로그인 경로는 다를 수 있다
- 화면에 `저장 완료`가 떠도, 그 데이터가 실제 로그인 검증 코드에서 읽히지 않으면 사용자는 로그인할 수 없습니다.
- 이번 문제는 `config/advancedFeatureConfig`에 수동 계정이 잘 저장됐는데도, 공개 페이지가 그 경로를 로그인에 쓰지 않아 생겼습니다.
- 그래서 “Firebase에 저장됐는가?”만 보면 안 되고, 꼭 아래 3가지를 같이 확인해야 합니다.
- 관리자 저장 경로
- 공개 페이지가 실제로 읽는 경로
- 그 둘을 이어 주는 검증 로직

## 해시만 있으면 로그인 검증은 되지만 평문 복구는 안 된다
- `passwordSalt + passwordHash`만 있으면 사용자가 입력한 비밀번호가 맞는지 비교할 수는 있습니다.
- 하지만 그 상태에서는 관리자도 원래 비밀번호를 다시 볼 수 없습니다.
- 평문을 다시 보여주려면 최소한 아래 둘 중 하나가 더 필요합니다.
- 관리자만 풀 수 있는 복구용 암호문
- 원본 신청 본문처럼 비밀번호가 들어 있는 다른 안전한 원천

## 공개 로그인용 데이터는 공개해도 되는 형태로 따로 만들어야 한다
- 민감한 관리자 장부를 그대로 공개하면 안 됩니다.
- 이번에는 공개 페이지 전용으로 `advancedAccountLicenses/<loginId>`를 만들고, 여기에 아래만 저장했습니다.
- 서명된 라이선스 번들
- 그 번들을 비밀번호로 다시 암호화한 결과
- 공개 페이지는 비밀번호로 복호화하고, 공개키로 서명을 검증합니다.
- 이렇게 하면 비밀번호 해시나 관리자 내부 장부를 공개하지 않고도 ID/비밀번호 로그인을 만들 수 있습니다.

## 세션 편의값과 영구 저장값은 반드시 분리해야 한다
- 관리자 화면에서 방금 입력한 비밀번호를 다시 보여주기 위해 메모리에 잠깐 들고 있는 값은 편의용입니다.
- 이런 값이 그대로 Firebase에 저장되면 보안 문제가 됩니다.
- 따라서 이번처럼 `_plainPasswordLocal` 같은 값은 아래처럼 경계를 분명하게 나눠야 합니다.
- 화면 세션에서는 허용
- Firebase 저장 직전에는 제거

## TODO는 해결 상태별로 나누면 다음 작업자가 훨씬 빠르게 들어온다
한 폴더 안에 `완료`, `부분 해결`, `미해결`이 뒤섞여 있으면 다음 작업자는 먼저 “지금 뭘 해야 하지?”부터 다시 판별해야 합니다.

그래서 TODO를 다시 정리할 때는 아래처럼 나누는 편이 좋습니다.

- `부분 해결`: 이미 코드가 일부 들어간 항목
- `미해결`: 구현 흔적이 거의 없는 항목

이렇게 나누면 다음 작업자는 처음부터 전체를 읽지 않고도, 바로 자기 작업 범위를 잡을 수 있습니다.

## TODO 문서는 요청문 복사본이 아니라 실행 문서여야 한다
좋은 TODO 문서에는 단순 요구사항만 있으면 부족합니다.

다음 정보가 함께 있어야 실제로 바로 작업할 수 있습니다.

- 현재 상태
- 이미 반영된 근거
- 아직 미완료인 이유
- 바로 시작할 첫 단계
- 완료 판단 기준

이 다섯 가지가 있으면, 다음 에이전트가 같은 분석을 다시 반복하지 않아도 됩니다.

## 이메일 조회는 평문 이메일 인덱스보다 `이메일 + 비밀번호` 조합 키가 안전하다
- 사용자가 신청 상태를 조회할 때 이메일로 찾게 하면 편하지만, 단순히 `email -> requestId` 공개 인덱스를 만들면 누가 어떤 이메일로 신청했는지 추측하기 쉬워집니다.
- 그래서 이번에는 평문 이메일 대신 `정규화한 이메일 + 조회 비밀번호`를 합쳐 SHA-256 키를 만들고, 그 키로만 조회 인덱스를 찾게 했습니다.
- 이렇게 하면 사용자는 여전히 “내 이메일로 조회한다”고 느끼지만, 실제 공개 경로에는 이메일 원문이 남지 않습니다.
- 즉 편의성과 공개 노출 억제를 둘 다 챙길 때는 `사용자 입력 2개를 합친 조회 키`가 실용적인 절충안이 됩니다.

## 실행 TODO와 추가 개선 백로그는 분리해야 한다
- 어떤 항목은 지금 바로 구현해야 하는 작업이고, 어떤 항목은 제품 품질을 높이는 개선 과제입니다.
- 이 둘을 한 폴더에 섞어 두면 다음 작업자가 “지금 당장 해야 할 일”과 “나중에 더 좋게 만들 일”을 구분하느라 시간을 씁니다.
- 그래서 이번처럼 `10_부분해결`, `20_미해결`은 실행 TODO로 두고, `30_추가개선`은 구조/문구/UX/보안 품질 향상용 백로그로 따로 빼는 편이 더 실용적입니다.

## 개선 문서에도 기능 보존 체크가 먼저 와야 한다
- “더 좋게 바꾸기” 작업은 겉으로는 리스크가 낮아 보여도, 실제로는 기존 사용 흐름을 가장 쉽게 망가뜨립니다.
- 그래서 개선 TODO를 쓸 때도 아래 항목을 먼저 적는 편이 안전합니다.
- 반드시 보존할 현재 기능
- 위험 구간
- 코드 기준 보존 포인트
- 검증 체크리스트
- 이렇게 해 두면 다음 작업자가 구조나 문구를 손보면서도 핵심 기능을 덜 잃어버립니다.

## 다음 에이전트가 바로 움직이게 하려면 작업 큐 문서가 따로 있어야 한다
- 인덱스와 런북만 있으면 “원칙”은 알 수 있어도, 지금 무엇을 집어야 하는지 헷갈릴 수 있습니다.
- 그래서 실행용 TODO에는 `순차 작업 큐`를 따로 두는 편이 좋습니다.
- 이 큐에는 최소한 아래 네 가지가 있어야 합니다.
- 현재 순서
- 각 항목의 시작 조건
- 완료 조건
- 작업 종료 시 남길 기록
- 이렇게 하면 다음 에이전트에게 “TODO 폴더 보고 순서대로 해”라고만 말해도, 다시 계획을 새로 짜지 않고 바로 움직일 수 있습니다.

## 복붙용 전달 프롬프트를 문서로 고정하면 지시 품질이 덜 흔들린다
- 실행 규칙이 문서에만 있고, 실제로 사용자나 상위 agent가 매번 새로 문장을 만들면 지시가 조금씩 달라질 수 있습니다.
- 그래서 `복사해서 그대로 보내는 프롬프트 문서`를 따로 두는 편이 좋습니다.
- 이렇게 하면 다음과 같은 장점이 있습니다.
- 누가 보내도 같은 기준으로 시작할 수 있습니다.
- 중요한 조건이 빠질 가능성이 줄어듭니다.
- 부분 작업용, 전체 작업용 프롬프트를 나눠 둘 수 있습니다.

## 운영 기본값과 코드 기본값이 둘 다 있으면 오래된 운영값이 새 화면 문구를 덮을 수 있다
- 이번에는 코드에서 분명히 새 버튼 이름을 넣었는데도, 화면에는 예전 문구가 남아 있었습니다.
- 원인은 브라우저나 코드 버그가 아니라, 운영 Firebase `siteTextConfig`에 저장된 예전 기본값이 새 코드 기본값 위에 다시 덮여쓰기 때문이었습니다.
- 이런 상황에서는 무조건 운영값을 새 값으로 덮어쓰는 것보다, `이 값이 진짜 운영자가 고른 값인지`, `예전 코드 기본값의 잔재인지`를 먼저 구분해야 합니다.
- 그래서 이번처럼 `레거시 기본값이면 새 코드 기본값으로 자동 보정`하는 계층을 두면, 운영 커스터마이징은 보존하면서 오래된 기본값만 자연스럽게 교체할 수 있습니다.

## 캔버스 커서 문제는 CSS 커서 하나만 믿지 말고 오버레이로 제어할 수 있다
- 브라우저 기본 `crosshair`는 운영체제, 브라우저, 배경색에 따라 대비가 들쭉날쭉할 수 있습니다.
- 그래서 커서가 중요 기능인 캔버스에서는 실제 입력 좌표를 바꾸지 않는 `시각용 오버레이`를 따로 두는 방식이 실용적입니다.
- 이때 핵심은 두 가지입니다.
- 오버레이는 `pointer-events: none`으로 두어 입력을 막지 않아야 합니다.
- 터치 입력이나 탭 전환 때는 숨겨서 두 개의 커서처럼 보이지 않게 해야 합니다.

## 개인 저장 기능은 메인 도구에 섞기보다 별도 경로와 소유자 규칙으로 분리하는 편이 안전하다
- 개인 학습자료 보관함을 메인 연습 화면에 바로 넣으면, 기존 연습 흐름의 집중도가 떨어지고 인증 경계도 더 복잡해집니다.
- 그래서 이번에는 `study-archive.html`처럼 별도 경로로 분리하고, RTDB도 `userStudyLibrary/<uid>` 경계로 끊었습니다.
- 이렇게 하면 사용자는 자기 자료만 보게 만들기 쉽고, 메인 도구 화면도 그대로 유지할 수 있습니다.
- 즉 `기능 분리`와 `권한 분리`를 같이 잡아야 새 기능을 붙여도 기존 앱이 덜 흔들립니다.

## 고급 전용 기능은 버튼만 숨기지 말고 URL·문구·로그인까지 한 묶음으로 맞춰야 한다
- 처음에는 `일반 모드에서 버튼만 숨기면 끝 아닌가?`라고 생각하기 쉽습니다.
- 그런데 실제 운영에서는 아래 셋이 따로 놀면 바로 문제가 생깁니다.
- 버튼은 숨겼는데 직접 URL로는 들어가는 경우
- URL은 막았는데 도움말이나 고급 안내 문구에는 아직 일반 기능처럼 적혀 있는 경우
- 고급 라이선스만 확인하고, 그 안의 개인 자료는 로그인 없이 보이게 남겨 두는 경우
- 그래서 이번처럼 고급 전용 기능을 손볼 때는 항상 아래 4가지를 같이 확인해야 합니다.
- 메인 화면 노출 조건
- 직접 URL 접근 제한
- 추가 로그인 또는 소유자 검증
- `siteTextConfig`와 문서 동기화
- 초보자가 기억할 한 줄은 이것입니다.
- `숨김 기능`은 CSS 한 줄이 아니라 `권한 규칙 한 묶음`입니다.

## 에이전트 관리 문서는 루트가 아니라 `docs/agent/` 하위 표준 폴더로 몰아야 덜 망가진다
- 전역 `AGENTS.md`가 루트 직하 `00_`, `35_`, `50_`, `70_`, `90_` 문서를 자연스럽게 만들도록 유도하면, 실제 프로젝트 루트가 코드 파일과 운영 문서로 뒤섞입니다.
- 이 문제는 문서 이름이 많은 게 핵심이 아니라, 문서가 놓이는 기본 경로를 코드와 규칙이 함께 강제하지 않았다는 점이 핵심입니다.
- 그래서 일반 프로젝트에서는 아래처럼 폴더 역할을 먼저 고정하는 편이 안전합니다.
- `docs/agent/runtime/`: 현재 라운드 기준으로 계속 갱신되는 활성 문서
- `docs/agent/output/`: notify/결과 출력 문서
- `docs/agent/worklog/`: 턴 단위 작업 기록
- `docs/agent/archive/`: 예전 handoff, 중복본, 퇴역 문서
- 규칙 문서만 바꾸면 다음 세션에 또 루트에 생길 수 있으니, startup prompt 생성기와 notify 저장 기본 경로도 함께 바꿔야 재발을 막을 수 있습니다.

## 기능 소실을 막으려면 긴 로그보다 얇은 기준서를 먼저 읽게 해야 한다
- 기능이 많은 프로젝트에서 에이전트가 매번 긴 작업 로그부터 읽으면 토큰은 많이 쓰고, 정작 지금 보존해야 할 기능은 놓치기 쉽습니다.
- 그래서 이번에는 문서를 두 층으로 나누는 편이 더 낫다고 정리했습니다.
- `docs/SKCT_TOOL_기능_카탈로그.md`: 사용자와 운영자가 보는 현재 기능 기준
- `docs/agent/runtime/30_MASTER_DOC.md`: 에이전트가 작업 전에 먼저 읽는 얇은 기준서
- 이렇게 나누면 “현재 기능이 무엇인지”와 “이번 작업에서 무엇을 읽고 무엇을 갱신해야 하는지”가 분리되어, 컨텍스트 낭비가 줄고 회귀 위험도 줄어듭니다.
- 핵심은 문서를 많이 만드는 것이 아니라, **짧은 기준 문서를 하나 만들어 작업 시작 순서를 고정하는 것**입니다.

## 런타임 홈 정리는 "전부 최신화"보다 "활성 표면 분리"가 먼저다
- `.codex`처럼 문서와 실행 흔적이 같이 쌓이는 런타임 홈에서는, 모든 문서를 같은 비중으로 읽거나 고치기 시작하면 오히려 컨텍스트 낭비가 커집니다.
- 이번 정리에서 더 효과적이었던 기준은 아래 세 가지였습니다.
- 지금도 먼저 읽는 활성 문서
- 실행 중 다시 만들어지는 생성물과 상태 파일
- 당시 판단 근거만 남기는 보관 문서
- 이렇게 나누면 오래된 handoff나 전달용 정리 문서를 지우지 않아도, 시작 허브에서 내려 보내는 것만으로 작업 흐름이 훨씬 자연스러워집니다.
- 특히 코드가 아직 쓰는 예외 경로는 바로 삭제하지 말고, 문서상으로 `기본값 아님`으로 강등하는 편이 안전합니다. 이번 `workspace_homes`가 그 사례였습니다.

## 고급 기능은 "있다"보다 "지금 열려 있다"를 보여줘야 사용자가 덜 헤맨다
- 기능이 많아질수록 사용자는 `이 버튼이 지금 내 계정에서 되는 상태인지`부터 헷갈립니다.
- 그래서 이번에는 고급 모드 진입 후 아래 값을 바로 보여주는 상태 바를 따로 두는 편이 더 낫다고 정리했습니다.
- 로그인 식별자
- 만료 시각
- 자료 보관함 가능 여부
- 우측 실제환경 여백 복원 여부
- 초보자 기준으로는 “기능 설명 문장”보다 “내가 지금 어떤 권한 상태인지”가 먼저 보이는 편이 훨씬 덜 헷갈립니다.

## 도움말 이미지는 스크린샷만 넣지 말고, 어디를 보라는지 같이 강조해야 한다
- 그냥 화면 전체를 넣으면 사용자는 오히려 어디를 봐야 할지 모릅니다.
- 그래서 이번에는 `crop + 강조 박스 + 짧은 라벨`을 같이 쓰는 방식이 더 읽기 쉬웠습니다.
- 핵심은 많이 표시하는 것이 아니라, 한 장당 한두 개만 강조하는 것입니다.
- 라벨이 가장자리에서 잘리거나, 강조 박스가 실제 요소보다 더 눈에 띄면 그 이미지는 과감히 다시 다듬어야 합니다.

## 버전 문자열은 여러 파일에 흩어 놓지 말고 공용 빌드 정보로 모아야 덜 틀어진다
- 이번에는 사용자 화면과 관리자 화면에 같은 버전 정보를 보여주는데, HTML과 JS에 각각 하드코딩돼 있었습니다.
- 이런 구조에서는 한쪽만 바뀌고 다른 쪽이 예전 값을 유지하기 쉽습니다.
- 그래서 `build-info.js`처럼 공용 정보를 한 곳에 두고, 화면은 그 값을 읽어 뿌리게 하면 수정 지점이 줄고 회귀도 줄어듭니다.

## 일반 가이드와 권한형 기능 설명은 같은 상자에 넣지 않는 편이 덜 헷갈린다
- 처음 쓰는 사용자는 `지금 바로 쓸 수 있는 기능`과 `승인 후에 열리는 기능`을 한 화면에서 동시에 보면 오히려 경계를 헷갈립니다.
- 그래서 이번에는 아래처럼 역할을 나누는 편이 더 낫다고 정리했습니다.
- `가이드`: 일반 모드 기준 사용 순서와 기본 기능만 설명
- `고급 기능`: 승인 전 사용자가 "무엇이 추가되는지"와 "어떻게 여는지"를 확인
- `고급 활용`: 승인 후 사용자가 실제로 어떤 버튼을 어떤 순서로 쓰는지 확인
- 핵심은 설명을 많이 넣는 것이 아니라, **사용자 상태별로 문서와 버튼의 책임을 나누는 것**입니다.
