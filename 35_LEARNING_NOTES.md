# SKCT Tool 학습 노트
작성일시: 2026-04-09 00:07:00 KST

이 문서는 이번 작업에서 다시 배운 점을 초보자도 따라갈 수 있게 적는 메모입니다.

## 관리자 장부와 실제 로그인 경로는 다를 수 있다
- 화면에 `저장 완료`가 떠도, 그 데이터가 실제 로그인 검증 코드에서 읽히지 않으면 사용자는 로그인할 수 없습니다.
- 이번 문제는 `config/advancedFeatureConfig`에 수동 계정이 잘 저장됐는데도, 공개 페이지가 그 경로를 로그인에 쓰지 않아 생겼습니다.
- 그래서 “Firebase에 저장됐는가?”만 보면 안 되고, 꼭 아래 3가지를 같이 확인해야 합니다.
  - 관리자 저장 경로
  - 공개 페이지가 실제로 읽는 경로
  - 그 둘을 이어 주는 검증 로직

## 해시만 있으면 로그인 검증은 되지만 평문 복구는 안 된다
- `passwordSalt + passwordHash`만 있으면 사용자가 입력한 비밀번호가 맞는지 비교할 수는 있습니다.
- 하지만 그 상태에서는 관리자도 원래 비밀번호를 다시 볼 수 없습니다.
- 평문을 다시 보여주려면 최소한 아래 둘 중 하나가 더 필요합니다.
  - 관리자만 풀 수 있는 복구용 암호문
  - 원본 신청 본문처럼 비밀번호가 들어 있는 다른 안전한 원천

## 공개 로그인용 데이터는 공개해도 되는 형태로 따로 만들어야 한다
- 민감한 관리자 장부를 그대로 공개하면 안 됩니다.
- 이번에는 공개 페이지 전용으로 `advancedAccountLicenses/<loginId>`를 만들고, 여기에 아래만 저장했습니다.
  - 서명된 라이선스 번들
  - 그 번들을 비밀번호로 다시 암호화한 결과
- 공개 페이지는 비밀번호로 복호화하고, 공개키로 서명을 검증합니다.
- 이렇게 하면 비밀번호 해시나 관리자 내부 장부를 공개하지 않고도 ID/비밀번호 로그인을 만들 수 있습니다.

## 세션 편의값과 영구 저장값은 반드시 분리해야 한다
- 관리자 화면에서 방금 입력한 비밀번호를 다시 보여주기 위해 메모리에 잠깐 들고 있는 값은 편의용입니다.
- 이런 값이 그대로 Firebase에 저장되면 보안 문제가 됩니다.
- 따라서 이번처럼 `_plainPasswordLocal` 같은 값은
  - 화면 세션에서는 허용
  - Firebase 저장 직전에는 제거
  처럼 경계를 분명하게 나눠야 합니다.
