# SKCT Tool 2차 수정 지시서 v3

## 목적

이번 라운드는 새로운 기능 추가가 아니라, 현재 코드에서 남아 있는 명확한 문제만 수정한다.

수정 범위는 아래 4개로 제한한다.

1. 차트 렌더링 버그 수정
2. 공지 데이터 소스 단일화
3. 관리자 게시글 목록 실시간 구독 제거
4. README 최신화

이번 라운드에서는 아래 작업은 하지 않는다.

1. Firebase 방문 통계 구조 전면 개편
2. 런타임 설정 UI 추가
3. 게시판 데이터 모델 추가 개편
4. 도메인 관련 작업
5. 광고 관련 작업

---

## 1. 차트 렌더링 버그 수정

### 현재 문제

최신 `index.html`의 최근 7일 차트 옵션에 아래 코드가 있다.

```js
suggestedMax: Math.max(.chartData, 5) + 2
```

이 코드는 잘못된 문법이다. 실제 브라우저에서 차트 렌더링 실패나 콘솔 에러를 만들 수 있다.

### 수정 지시

아래 기준으로 수정한다.

1. `Math.max(.chartData, 5)`를 올바른 코드로 교체한다.
2. 의도는 `chartData` 배열의 최대값을 구한 뒤 최소 기준 5와 비교하는 것이다.
3. 따라서 아래 형태와 동일한 의미가 되도록 수정한다.

```js
suggestedMax: Math.max(...chartData, 5) + 2
```

4. `chartData`가 빈 배열이거나 숫자가 아닌 값이 섞여 있어도 깨지지 않도록 방어 로직을 넣는다.
5. 차트 렌더링 전체를 한 번 더 점검해서 콘솔 에러가 없는지 확인한다.

### 완료 기준

1. stats 모달을 열었을 때 차트가 정상 표시된다.
2. 콘솔에 차트 관련 syntax error 또는 runtime error가 없다.
3. 최근 7일 데이터가 0이어도 차트가 깨지지 않는다.

## 2. 공지 데이터 소스 단일화

### 현재 문제

현재 공지 데이터 소스가 섞여 있다.

1. 관리자 페이지는 `config/notice`를 저장한다.
2. 공개 페이지는 `config/notice`를 1회 조회한다.
3. 동시에 `script.js`에는 `notice.json` 폴백 로직이 남아 있다.
4. README도 아직 `notice.json` 수정 중심으로 설명한다.

이 상태는 운영 기준이 모호하다.

### 이번 라운드의 운영 기준

운영 기준은 `Firebase Realtime Database의 config/notice` 하나로 고정한다.

즉, 이번 라운드 이후 공지의 단일 진실 소스는 아래다.

```text
config/notice
```

### 수정 지시

아래를 수행한다.

1. 공개 페이지의 공지 표시 기준을 `config/notice` 하나로 통일한다.
2. `script.js`에 남아 있는 `notice.json` 폴백 로직을 제거한다.
3. `notice.json`을 더 이상 런타임 공지 소스로 사용하지 않는다.
4. 관리자 페이지의 저장 로직은 계속 `config/notice`를 사용한다.
5. 주석과 설명에서 `notice.json`이 현재 운영 공지 소스처럼 보이는 문구를 제거한다.

### 참고

`notice.json` 파일 자체를 즉시 삭제할 필요는 없다. 다만 이번 라운드에서는 런타임에서 사용하지 않게 만들어야 한다.

### 완료 기준

1. 공지를 수정하면 관리자 저장 후 공개 페이지에서 Firebase 공지만 반영된다.
2. `script.js`에 `fetch('notice.json')` 기반 공지 폴백이 남아 있지 않다.
3. README와 코드 주석에서 공지 운영 기준이 `config/notice`로 일치한다.

## 3. 관리자 게시글 목록 실시간 구독 제거

### 현재 문제

공개 커뮤니티는 이미 1회 조회 기반으로 바뀌었지만, 관리자 페이지는 아직 게시글 목록을 실시간 구독하고 있다.

이 구조는 관리자가 페이지를 열어두는 동안 불필요한 Firebase 다운로드를 계속 발생시킨다.

관리자 게시판은 실시간일 필요가 없다. 관리자는 직접 새로고침해도 충분하다.

### 수정 지시

아래 기준으로 바꾼다.

1. 관리자 페이지에서 `posts` 전체를 `onValue`로 실시간 구독하는 로직을 제거한다.
2. 관리자 페이지 진입 시 게시글 목록은 1회만 로드한다.
3. 상단 또는 게시글 관리 섹션에 `새로고침` 버튼을 추가한다.
4. 새로고침 버튼 클릭 시에만 게시글 목록을 다시 불러온다.
5. 필터 버튼은 현재 로드된 메모리 데이터에서만 동작하게 유지한다.
6. 관리자 답글 작성, 삭제, 핀 고정, FAQ 이동 같은 작업 후에는 자동으로 전체 실시간 구독하지 말고, 필요한 시점에만 목록을 다시 로드한다.
7. 가능하면 공용 함수 이름을 명확히 한다. 예시:
   - `loadPostsOnce()`
   - `refreshPosts()`
   - `renderPostTable()`

### 추가 지시

관리자 댓글 모달도 같은 원칙을 따른다.

1. 댓글 목록은 모달을 열 때 1회 조회
2. 실시간 구독은 하지 않음
3. 댓글 작성/수정/삭제 후에는 해당 게시글 댓글만 다시 읽거나 캐시를 무효화하고 다시 1회 로드

### 완료 기준

1. 관리자 페이지에서 게시글 목록이 자동 실시간 갱신되지 않는다.
2. 새로고침 버튼으로만 목록이 갱신된다.
3. 관리자 기능은 기존과 동일하게 동작한다.
4. `onValue(ref(db, 'posts'))` 또는 동등한 실시간 전체 구독이 관리자 페이지에서 제거된다.

## 4. README 최신화

### 현재 문제

README 설명이 현재 코드와 맞지 않는 부분이 남아 있다.

대표적으로 아래가 문제다.

1. 공지를 `notice.json` 직접 수정으로 설명
2. 방문자 카운터 설명이 현재 구조와 불일치
3. 현재 관리자 페이지 및 Firebase 기반 운영 구조 반영 부족

### 수정 지시

README를 현재 실제 운영 구조 기준으로 갱신한다.

아래 내용을 반영한다.

#### 4-1. 공지 설명 수정

기존처럼 `notice.json` 수정으로 공지가 반영된다고 쓰지 않는다.

대신 아래 취지로 바꾼다.

1. 공지는 관리자 페이지에서 수정한다.
2. 저장 위치는 Firebase `config/notice`다.
3. 공개 페이지와 커뮤니티는 이 공지를 읽어 표시한다.

#### 4-2. 방문 통계 설명 수정

현재 실제 사용 중인 구조만 적는다.

1. 현재 접속자 관련 정보는 Firebase 기반
2. 공개 페이지와 관리자 페이지는 최적화된 조회 방식 사용
3. 필요 이상으로 오래된 방문자 카운터 설명은 제거

#### 4-3. 프로젝트 구조 설명 수정

현재 실제 파일 기준으로 맞춘다.

1. `admin.html`
2. `admin.css`
3. `community.js`
4. `script.js`
5. `index.html`
6. `notice.json`은 남아 있더라도 운영 공지 소스가 아니라는 점을 명시하거나, 설명에서 비중을 낮춘다.

#### 4-4. 운영 방식 설명 수정

README에 아래를 반영한다.

1. 공개 게시판은 실시간 전체 구독이 아니라 최적화된 1회 조회 기반
2. 관리자 공지 수정은 Firebase 기반
3. 현재 구조가 GitHub Pages + Firebase + GA4 조합이라는 점

### 완료 기준

1. README가 현재 코드 구조와 모순되지 않는다.
2. 공지 운영 기준이 `config/notice`로 명확히 적혀 있다.
3. 오래된 `notice.json 직접 수정` 설명이 제거되거나 보조 파일 수준으로 정리된다.

## 구현 시 주의사항

1. 이번 라운드에서는 기능 UI를 크게 바꾸지 않는다.
2. 기존 사용자 경험은 유지한다.
3. 코드 스타일은 현재 프로젝트 스타일을 따른다.
4. 수정 후 콘솔 에러를 직접 확인한다.
5. 불필요한 리팩터링은 하지 않는다.
6. 이번 라운드는 명확한 문제 수정만 수행한다.

## 최종 체크리스트

아래 4개가 모두 충족되면 이번 라운드는 완료다.

1. stats 모달 차트가 정상 렌더링된다.
2. 공지 소스가 `config/notice` 하나로 정리된다.
3. 관리자 게시글 목록 실시간 구독이 제거되고 수동 새로고침 방식이 적용된다.
4. README가 최신 운영 구조와 일치한다.
