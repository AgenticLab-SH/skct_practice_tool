# 차트 수정 전용 작업 지시서

## 목적

이번 작업은 `stats` 모달의 최근 7일 차트가 왜 깨질 수 있는지, 왜 단순 치환이 아니라 **의도를 이해한 수정**이 필요한지 명확히 설명하고, agent가 스스로 검토 후 안정적으로 수정하도록 안내하는 문서다.

이 문서는 **차트 부분만** 다룬다.

---

## 문제 요약

현재 최근 7일 차트의 y축 `suggestedMax` 계산식이 잘못 작성되어 있다.

핵심 문제는 아래 두 가지다.

1. `Math.max` 사용 방식이 잘못되었다.
2. 데이터 정제 없이 배열을 바로 최대값 계산에 넣으려는 흔적이 있다.

즉, 이 문제는 단순히 오타 하나가 아니라, 아래 로직 전체를 다시 확인해야 하는 문제다.

- 차트 데이터는 어떤 타입으로 들어오는가
- `chartData`가 배열인지 보장되는가
- 값이 문자열일 수 있는가
- 값이 비어 있거나 `null`, `undefined`, `NaN`일 수 있는가
- y축 상한을 왜 자동 계산하려는가

---

## 왜 이 수정이 필요한가

### 1. `Math.max`는 배열 자체를 받는 함수가 아니다

`Math.max`는 아래처럼 **숫자 목록**을 인자로 받아야 한다.

```js
Math.max(1, 2, 3)
```

배열을 그대로 넣으면 의도대로 동작하지 않는다.

```js
Math.max([1, 2, 3])
```

즉, `chartData`가 배열이라면 `Math.max(...chartData)`처럼 **spread** 하거나, 먼저 안전한 숫자 배열로 변환한 뒤 넣어야 한다.

### 2. 현재 데이터는 “무조건 숫자 배열”이라고 가정하면 안 된다

최근 7일 데이터는 외부 소스에서 로드되므로, 아래 상황을 반드시 고려해야 한다.

1. 문자열 `"3"` 형태
2. `null`
3. `undefined`
4. 비어 있는 배열
5. 숫자가 아닌 값

이 상태에서 바로 `Math.max(...chartData)`를 쓰면 `NaN`이 전파되거나 예외 상황이 생길 수 있다.

### 3. `suggestedMax`는 차트를 보기 좋게 만들기 위한 값이다

이 값의 목적은 “실제 최대값보다 약간 여유 있는 y축 상한”을 만드는 것이다.

예를 들어 데이터가 `[2, 5, 3]`이면, 최대값 `5`에 여유치 `+2`를 더해 `7` 정도를 상한으로 주면 시각적으로 보기 좋다.

즉, 이 로직은 아래 의도를 만족해야 한다.

1. 데이터 최대값을 찾는다
2. 너무 작으면 최소 기준값을 유지한다
3. 마지막에 적당한 여유를 더한다

이 의도가 깨지면 차트는
- 너무 눌려 보이거나
- 상한이 이상해지거나
- 렌더링이 깨질 수 있다

---

## agent가 이해해야 하는 핵심 의도

이번 수정의 본질은 아래 한 문장으로 요약된다.

**“최근 7일 차트의 y축 상한은, 안전하게 숫자로 정제된 데이터 배열의 최대값과 최소 기준값을 비교해서 계산해야 한다.”**

즉, 수정 포인트는 “문법 고치기”가 아니라 아래 순서다.

1. `chartData`를 안전한 숫자 배열로 정제
2. 정제된 배열 기준으로 최대값 계산
3. 최소 기준값과 비교
4. 여유값 추가
5. 빈 배열이어도 깨지지 않게 처리

---

## 수정 원칙

agent는 아래 원칙에 따라 수정한다.

### 원칙 1. 먼저 데이터 정제

`chartData`를 그대로 믿지 말고, 먼저 숫자 배열로 정제한다.

예시 의도:
- `Number(v) || 0` 또는 동등한 안전 처리
- `null`, `undefined`, 비숫자 값이 있어도 차트가 깨지지 않게 할 것

### 원칙 2. 최대값 계산은 정제 후 수행

최대값은 반드시 **정제된 배열** 기준으로 계산한다.

### 원칙 3. 최소 기준값 유지

데이터가 모두 0이거나 너무 작더라도 y축이 지나치게 납작해지지 않도록 최소 기준값을 유지한다.

현재 요구 기준은 최소값 `5`를 유지하는 방향이 적절하다.

### 원칙 4. 여유값 추가

최대값 또는 최소 기준값을 결정한 뒤, 최종 y축 상한에는 약간의 여유값을 더한다.

현재 요구 기준은 `+2`가 적절하다.

### 원칙 5. 빈 배열에서도 정상 동작

데이터가 비어 있어도 차트가 렌더링 자체는 되어야 한다.

즉,
- syntax error 없어야 함
- runtime error 없어야 함
- `suggestedMax`가 `NaN`이 되면 안 됨

---

## 권장 구현 방향

구현 방식은 agent가 스스로 선택해도 된다. 다만 아래 의도를 만족해야 한다.

### 권장 흐름

```js
const safeChartData = (chartData || []).map(v => Number(v) || 0);
const computedMax = Math.max(...safeChartData, 5);
const suggestedMax = computedMax + 2;
```

이 흐름이 좋은 이유는 다음과 같다.

1. `chartData`가 없을 때도 안전하다
2. 문자열 숫자를 정수/실수로 바꿀 수 있다
3. 비정상 값은 0으로 흡수 가능하다
4. 최소 기준값 5를 유지할 수 있다
5. 최종 y축 여유값까지 명확하다

---

## 절대 하지 말 것

1. `Math.max`에 배열을 그대로 넣지 말 것
2. `chartData`가 무조건 숫자 배열이라고 가정하지 말 것
3. 수정 없이 단순 문자열 치환만 하고 끝내지 말 것
4. `suggestedMax`가 `NaN`이 될 가능성을 남기지 말 것
5. 차트가 “데이터가 없을 때는 숨겨지면 된다”라고 처리하지 말 것  
   - 이번 요구는 “빈 데이터에서도 깨지지 않는 차트”다

---

## 수정 후 검증 항목

agent는 수정 후 아래를 반드시 직접 검토한다.

### 1. 콘솔 검증
- syntax error 없음
- runtime error 없음
- Chart.js 관련 경고/오류 없음

### 2. 정상 데이터 검증
예시:
- `[1, 2, 3, 4, 5]`
- 정상적으로 차트 표시
- y축 상한이 데이터 최대값보다 약간 큰 값으로 보이는지 확인

### 3. 0 데이터 검증
예시:
- `[0, 0, 0, 0, 0, 0, 0]`
- 차트가 깨지지 않는지
- y축 상한이 너무 낮게 눌리지 않는지 확인

### 4. 비정상 데이터 검증
예시:
- `["3", null, undefined, "x", 5]`
- 차트가 깨지지 않는지
- `suggestedMax`가 `NaN`이 되지 않는지 확인

### 5. 빈 배열 검증
예시:
- `[]`
- 차트 인스턴스 생성 및 모달 오픈/클로즈가 정상인지 확인

---

## 완료 기준

아래 조건을 모두 만족하면 이 작업은 완료다.

1. `suggestedMax` 계산식이 문법적으로 올바르다
2. 최대값 계산이 정제된 숫자 배열 기준으로 수행된다
3. 최소 기준값 5가 유지된다
4. 최종 상한에 여유값 `+2`가 반영된다
5. 빈 배열, 0 배열, 문자열 섞인 배열에서도 차트가 깨지지 않는다
6. stats 모달을 실제로 열어봤을 때 콘솔 에러 없이 정상 렌더링된다

---

## 최종 한 줄 요약

이번 수정은 단순 오타 수정이 아니라,  
**“차트 y축 상한 계산을 안전한 숫자 데이터 기반으로 다시 세우는 작업”**으로 이해하고 처리할 것.
